# Agent brief — postgres demo

This file is the human-authored input to the lazytui loop. Everything
else in this directory (`tui.yml`, `scripts/`, `plugins/`,
`docker-compose.yml`) is produced by an AI coding agent reading this
prompt plus the output of `node js/app/tui.js --spec`. Re-running that
combination should regenerate a functionally equivalent demo.

## What I want

A TUI that exercises the full build / test / run lifecycle of
PostgreSQL from source in Docker.

As an operator I want to:

- Kick off a clean build of postgres from a release tag and watch the
  output stream live.
- Run the regression suite and have failures surfaced automatically.
- Drop into an interactive `psql` session against the running server.
- Follow server logs in a side tab while I poke at the database.
- See container CPU/MEM live while a workload is hitting it.
- Browse `PGDATA` from inside the TUI — `postgresql.conf`,
  `pg_hba.conf`, `PG_VERSION`, the WAL directory, etc. The data dir
  lives in a docker named volume (not a bind mount) so a host-side
  file panel can't see it; use lazytui's `source: docker` files panel
  to list and read paths inside the running `pg` container.

## Constraints

- **Build target**: postgres 16 release tag (`REL_16_STABLE`).
- **Test target**: `make check` from the source tree.
- **Host prerequisites**: stock Linux host with Docker installed.
  Nothing else — no `kubectl`, no homebrew, no host-side postgres
  install.
- **Host pollution**: all side effects must live under
  `demo/postgres/`. No writes to `/var`, `/etc`, or `$HOME` on the
  host.
- **Network**: outbound network during build is fine (apt, source
  tarball or git clone).

## Action surface I expect

- `build` — `./configure && make` inside a builder image. Streams
  output to the detail panel. Slow (~5 min cold) — mark `tab: true`
  so the user can flip away and back.
- `test` — `make check`. On non-zero exit, automatically surface
  `regression.diffs` (or equivalent) in the detail panel.
- `start` / `stop` / `restart` — postgres container lifecycle.
- `psql` — `type: spawn`, interactive session against the running
  container.
- `logs` — `type: run` + `tab: true`, `docker logs -f` follow.
- Built-in `containers` panel for status; built-in `stats` panel for
  CPU/MEM.

## Out of scope (v1)

- Replication, streaming, logical decoding.
- pgbench or any benchmarking harness.
- Multi-version testing (16 only).
- Backup / restore tooling.
- TLS / SSL setup.
- Connection pooling (pgbouncer, etc.).

## Layout preference

Two-column standard. Left: `containers` + `groups` + a `files` panel
with `source: docker, container: pg, root: /var/lib/postgresql/data`
so PGDATA is one tab away. Right: `actions` + `stats` + `detail`.
Dracula theme. Detail panel ~60% height.

## Notes for the agent

- Lean on the built-in docker plugin for container ops. Don't
  reinvent.
- The "surface `regression.diffs` on failure" hook is the one place a
  small wrapper script likely earns its keep. Prefer a shell wrapper
  in `scripts/` over a custom JS plugin if it gets the job done.
- The build is slow. Use `type: run` so output streams visibly, mark
  it `tab: true` so the user can navigate away without losing it.
- Mark `psql` `type: spawn` — it needs the full TTY.
- If you find yourself wanting state across runs (incremental builds,
  cached objects), use a named volume in `docker-compose.yml`. The
  demo cycle collapses if every build starts from scratch.
- **Bake action scripts into the image, do not bind-mount them.** Bind
  mounts of host paths break under docker-in-docker setups: the host
  docker daemon and the calling container see different filesystems
  at the same path, so the bind mount target is empty inside the
  container. `COPY scripts /scripts` in the Dockerfile works in both
  DinD and on metal. (Discovered the hard way on the first loop run.)
- Read `node js/app/tui.js --spec` before producing any code. The YAML
  schema, plugin contract, markup rules, and hub/decorator slots are
  all there.
