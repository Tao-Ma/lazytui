# Agent brief — cloudberrydb demo

Same loop as the postgres demo. This file is the human-authored input;
everything else in this directory (`tui.yml`, `scripts/`, optional
`plugins/`) is produced by an AI coding agent reading this prompt plus
`node js/tui.js --spec`.

## Context (read first)

Apache Cloudberry (incubating) is an open-source MPP analytical
database, a Greenplum / PostgreSQL fork now hosted under the ASF at
**`github.com/apache/cloudberry`**. It runs as a cluster: one
**coordinator** process + *N* **segment** processes, each a
postgres-derived backend. The source build is large (~30 min cold) and
cluster bring-up uses `gpinitsystem`.

**This demo wraps upstream's existing docker sandbox rather than
rolling its own.** Upstream ships a complete sandbox under
`devops/sandbox/` (Dockerfile, compose, init script, orchestration
`run.sh`). The lazytui demo's value is the *YAML/CLI action surface
on top* — not reinventing the docker plumbing. (Earlier rewrite ran
the loop without consulting upstream and produced a debian-based
Dockerfile + ad-hoc compose that diverged from upstream conventions
in five separate ways; that was thrown away and the demo restarted
from this prompt. See POSTMORTEM_v1.md.)

## What I want

A TUI that lets me drive the upstream cloudberry sandbox interactively:
build the image, bring the container up, run installcheck, drop into
psql, follow per-process logs, and see live container stats — using
upstream's `run.sh` and the cluster bring-up that upstream's
`init_system.sh` already handles.

## Constraints

- **Source location**: a sibling `upstream/` directory inside this demo
  dir, cloned from `https://github.com/apache/cloudberry` with
  `--recurse-submodules`. Document the clone command in README.md; do
  not vendor as a git submodule (keeps the demo lightweight and lets
  the user pin whichever tag they like).
- **No custom Dockerfile or docker-compose.yml in this demo.** Delegate
  to upstream's `devops/sandbox/run.sh` and
  `docker-compose-rockylinux9.yml`. If a build step is needed, call
  `./run.sh -c local -b` (build-only) or `-c main`.
- **Default cluster shape**: single-node (one `cbdb-cdw` container
  running coordinator + 2 primary segments internally, port 5432
  exposed on host 15432). The multi-container mode (`run.sh -m`) is
  available but out-of-scope for v1 — it adds an extra coordinator
  standby (`cbdb-scdw`) plus two separate segment hosts, and the
  v1 demo's value is established with one container.
- **Host pollution**: all side effects under `demo/cloudberrydb/`
  (including `demo/cloudberrydb/upstream/` after the clone).
- **Container name**: `cbdb-cdw` (upstream's convention). The
  containers panel will show one row in single-node mode.
- **Cluster data path**: `/data0/database/coordinator/gpseg-1` for
  coordinator, `/data0/database/primary/gpseg{0,1}` for segments
  (upstream's gpinitsystem layout). Log files live in each data
  dir's `log/` subdir as CSV.
- **Bake action scripts into the image — do not bind-mount them.**
  Propagated from the postgres demo's first live run (see
  `demo/postgres/POSTMORTEM.md`). Bind mounts break under
  docker-in-docker setups. *In this demo*, however, scripts run on
  the **host** via `docker exec` rather than from inside the
  container — so the demo's `scripts/` directory stays on the host
  and the rule effectively doesn't apply here. Still worth knowing
  because any future custom container image in this demo would hit
  the same trap.

## Action surface I expect

**Group: sandbox**
- `up` — `./upstream/devops/sandbox/run.sh -c main`. Builds image if
  needed, runs container, init_system.sh inside the container
  initializes and starts the cluster. Slow on first run (~30 min
  cold for the build).
- `up-local` — same but `-c local`, builds from `./upstream/` local
  source (faster on subsequent runs if user is hacking on upstream).
- `build-only` — `./run.sh -c main -b` (image build, no container).
- `down` — `docker rm -f cbdb-cdw`.

**Group: cluster** (actions against the running cbdb-cdw)
- `status` — `gpstate -s`, `tab: true`.
- `start` — `gpstart -a` (idempotent).
- `stop` — `gpstop -af`, `confirm:`.
- `restart` — `gpstop -afr`.
- `psql` — `type: spawn`, drop into psql as `gpadmin` against
  template1 on port 5432 inside the container.
- `logs-coordinator`, `logs-segment-0`, `logs-segment-1` — three
  separate `type: run` + `tab: true` actions, each tailing the
  most recent CSV log from that process's data dir.

**Group: test**
- `installcheck` — `make installcheck-world` from inside the
  container (against the cloned source under `/home/gpadmin/cloudberry`).
- `installcheck-good` — fast subset (`make installcheck-good` from
  `src/test/regress`).
- On failure, surface `regression.diffs` in the detail panel — same
  hook as the postgres demo.

**Panels**
- `containers` panel — one row (`cbdb-cdw`) in single-node mode.
- `stats` panel for CPU/MEM via the built-in `docker.stats` topic.
- No custom panels in v1.

## Out of scope (v1)

- Multi-container mode (`run.sh -m` with cbdb-scdw + segment hosts).
  Deferred until v2; the v1 action surface is established cleanly.
- Mirror segments, resource queues, gpbackup/gprestore, pgbouncer.
- Streaming replication, multi-host clusters.
- Custom upstream patches (the demo runs vanilla upstream).
- Segment-health plugin (revisit in v2 if visitors ask for it).

## Layout preference

Two-column. Left: `containers` + `groups`. Right: `actions` + `stats`
+ `detail`. Dracula theme. Detail ~60% height.

## Notes for the agent

- **Do not write a Dockerfile or docker-compose.yml.** Upstream has
  them. Wrap, don't replace.
- **All action `script:`s execute on the host.** They invoke either
  `./upstream/devops/sandbox/run.sh ...` for container lifecycle, or
  `docker exec cbdb-cdw ...` for in-container operations. The
  framework's `project_dir: .` setting gives them the right cwd.
- **psql access**: from inside the container, `psql -d template1`
  works (gpadmin is the default user, default port 5432). From the
  host, port 15432 is mapped to the container's 5432.
- **Log discovery is non-trivial**: gpinitsystem writes CSV logs to
  `<datadir>/log/gpdb-YYYY-MM-DD_HHMMSS.csv` and rotates daily. The
  log-tail scripts must `ls -t` and pick the latest, not assume a
  fixed filename.
- **First-run UX**: `up` action will be slow (~30 min). The README
  should set this expectation. Subsequent `down` + `up` is fast
  because the image is cached. The build-cache concern is moot here
  because upstream's Dockerfile handles it.
- **Installcheck path**: inside the container, the source tree is at
  `/home/gpadmin/cloudberry/` per upstream's Dockerfile. Run
  `installcheck-world` from there; `installcheck-good` from
  `src/test/regress`.
- Read `node js/tui.js --spec` before producing any code. Same
  contract as the postgres demo.
