# v0.6.1 → v0.6.2 migration guide

> v0.6.2 lifts the layout from "left column + right column" to an
> ordered list of N columns. The YAML schema changed shape; v0.6.1
> configs do not parse.

## TL;DR — one change to your YAML

Wrap the existing `layout.left` / `layout.right` blocks in a
`layout.columns:` list. Drop the keys `left:` and `right:`; each
former block becomes a list entry.

```yaml
# v0.6.1 — REJECTED with SchemaError under v0.6.2
layout:
  left:
    width: 30
    panels: [groups, files]
  right:
    panels:
      - actions
      - { tabs: [detail], height: 60% }

# v0.6.2
layout:
  columns:
    - width: 30
      panels: [groups, files]
    - panels:
        - actions
        - { tabs: [detail], height: 60% }
```

That's the entire mechanical change. Per-cell shape (bare pool-id
strings, `tabs: [...]` mappings, `height: 60%`, `heightPct`,
`collapsed`, explicit `hotkey:`) is unchanged. The top-level
`panels:` pool block is unchanged. Group/action/file declarations are
unchanged.

## Why the change

The hardcoded two-column shape ruled out workflows that wanted three
or more columns (the postgres demo needed a third column for SQL
output). The layout is now an ordered list, and v0.6.2 also ships
the runtime UX (drag-edge spawn + `:add-column` / `:remove-column`
cmdline verbs) to grow / shrink the list at runtime — see "New
runtime features" below.

## Schema details

```yaml
layout:
  columns:            # required, non-empty list
    - width: 30       # optional integer — cells, last column's is implicit
      panels: [...]   # required list (may be empty for spacer columns)
    - width: 24
      panels: [...]
    - panels: [...]   # last column, takes the remaining width
```

- `columns:` is required; must be a non-empty list.
- Each column is a mapping with `panels:` (required list) and an
  optional `width:` (positive integer).
- The LAST column's `width:` is ignored and produces a warning at
  parse — the last column always takes the remainder of the
  terminal's width.
- Empty `panels: []` is allowed (creates a spacer column you can
  drag panes into during free-config).

## Layout invariants (carry over from v0.6.1)

These didn't change; the error messages did:

- Exactly one tab of kind `detail` anywhere in the layout. (Was:
  "must be in the right column"; now: "must be in the last column".)
- At most one tab of kind `actions` anywhere.
- Detail must be in the LAST PANE of the LAST COLUMN.
- Detail must be the ONLY tab in its pane (multi-tab + detail
  deferred to v0.7).
- Detail and actions must live in the LAST column (not earlier).

## Hotkey assignment

The auto-pool per column is unchanged in shape, just generalized:

- First column → `1`-`6` (LEFT_HOTKEY_POOL)
- Last column → `7`-`9` (RIGHT_HOTKEY_POOL)
- Middle columns (only in 3+ column layouts) → no auto-pool. Each
  pane in a middle column must specify its hotkey explicitly via
  `hotkey:` on the cell mapping, OR accept an empty hotkey (the pane
  is still reachable via mouse / `:focus <title>`).

## Column caps soft-warn

The soft cap warnings still fire (over-cap doesn't refuse, just
warns):

- First column cap: 6 panes
- Last column cap: 3 panes
- Middle columns: same as the first column

Above the soft cap the renderer compresses; below MIN_PANEL_H (3
rows) panels start visibly clipping. The runtime + parser don't
enforce hard caps.

## New runtime features

### Drag-edge spawn

In free-config mode, dragging a pane (or a pool entry from the `w`
overlay) to one of these zones spawns a fresh column:

- **Left edge** (cursor within 2 cells of the terminal's left edge) —
  new column at position 0.
- **Column gap** (cursor within 2 cells of any internal column
  boundary) — new column inserted between the two adjacent columns.
- **Right edge** is HIT-TESTED but always REFUSED in v0.6.2 (would
  push detail off the last column). `:add-column` doesn't accept
  position == N either. v0.7 may relax this.

Detail and actions sources are refused for new_column drops in
v0.6.2 — they live in the last column by invariant.

A successful drop emits a green status notice ("added new column at
position N+1") in the footer.

### Cmdline verbs

- `:add-column [N]` — insert an empty column at 1-based position `N`.
  `N` must be 1..N_cols (where N_cols is the current column count);
  the new column lands at slot `N`, shifting later columns rightward.
  Default (no arg): the just-before-last slot — equivalent to
  passing `N_cols` (e.g. with 2 columns today, `:add-column` ≡
  `:add-column 2` and inserts between the existing columns). The
  right-edge slot (`N_cols + 1`) is refused — would push the
  detail-bearing last column off "last". Drag panes in afterwards.
- `:remove-column <N>` — remove the column at 1-based index `N`.
  Refused if `N` is the last column or if the column has panes
  (hide them first with `:hide <id>` or by picking them in the `w`
  overlay). Auto-cleanup from drag-edge spawn already removes empty
  source columns; this verb is for programmatic removal.

### Auto-cleanup on drag

When a drag-edge spawn moves the only pane out of a non-last column,
the source column is removed automatically (no orphaned blank
strip).

## What's NOT in v0.6.2

- Moving detail / actions across columns. They stay anchored to the
  last column.
- Right-edge spawn (position == N). Hit-tests show the zone for
  diagnostic feedback in the footer, but the validator refuses.
- Per-column collapse / expand. (Per-pane collapse is unchanged.)
- Row-splits inside a column. A column is still a vertical stack.

## Round-trip with `:save-layout`

`:save-layout` writes the runtime layout back to YAML in the
`columns:` form. The last column's implicit width is preserved
(no `width:` key emitted); columns spawned via drag-edge or
`:add-column` get their explicit width persisted.

## Error reference

| Error | Cause | Fix |
| --- | --- | --- |
| `v0.6.1 layout shape (\`left:\` / \`right:\`) is not supported` | YAML still uses `left:`/`right:` | Wrap in `columns:` list (see TL;DR) |
| `'layout' must declare a \`columns:\` list` | `layout:` block missing `columns:` | Add `columns:` and at least one entry |
| `'layout.columns' must be a non-empty list` | `columns: []` | Add at least one column |
| `column must be a mapping` | An entry isn't `{panels: [...], width?: int}` | Use a mapping, not a string/list |
| `'panels' must be a list` | A column has `panels:` set to a non-list | Use a YAML list (or `[]` for empty) |
| `'width' on the last column is ignored` (warning) | Last column has a `width:` key | Drop the key; it's a no-op |
| `tab of kind 'detail' must be in the last column` | Detail isn't in the last column | Move it to the last list entry |
| `tab of kind 'detail' must be in the last pane of the last column` | Detail isn't the last pane | Make it the last list entry of the last column |
