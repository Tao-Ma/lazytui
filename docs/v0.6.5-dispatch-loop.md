# Domain detangle — Stage 2-B: relocate the dispatch loop (the deep inversion)

Status: 📋 SPEC / proposed. Companion to `docs/v0.6.5-render-exit.md` ("Domain
detangle"), which shipped Stage 1 (feature-extract, SCC 4→3) and Stage 2-A
(panel-host seam, SCC 3→0). This doc specs **Stage 2-B**, the *deeper* inversion
that 2-A deliberately deferred.

> **2-A vs 2-B — read this first.** Stage 2-A already dissolved the static
> `{dispatch, overlay, panel}` import SCC via the `panel-host` seam (commit
> `ac92f3a`): pure pointer indirection, the module graph is a clean DAG **today**.
> 2-B does NOT fix a remaining cycle — there is none. 2-B fixes a *placement*
> wrong that 2-A left standing: **the TEA update loop physically lives in the
> `panel` layer**, and 2-A inverted its imports rather than moving it home. 2-B is
> the architecture the reference TEA implementations actually use (the runtime
> owns the loop and sits above components; components return effect descriptors
> and never reach the runtime). It is correctness/legibility work, not cycle work.
> If the static DAG is "good enough", 2-B is optional. If the goal is for the
> codebase to *structurally mirror* TEA, 2-B is the target.

## The problem 2-B fixes

The runtime is **split across two layers**:

| Runtime piece | Lives in | Layer | Correct? |
|---|---|---|---|
| Root reducer driver — `applyMsg` (`update`→`setModel`→`runEffects`) | `dispatch/dispatch.js` + `dispatch/reducer.js` | dispatch | ✅ |
| Effect executor + registry — `runEffects` / `registerEffect` / `_handlers` | `dispatch/effects.js` | dispatch | ✅ |
| **Component fan-out** — `dispatchMsg` / `_dispatchMsgInner` / `_runInstance` / `dispatchKeyToFocused` | **`panel/api.js`** | **panel** | ❌ runtime in the component layer |
| **Post-dispatch finalizer** — `_finalizeDispatch` / `_finalizeLayout` / depth counter (scroll clamp, viewer `innerH` = blessed-exception B, instance reconcile) | **`panel/api.js`** | **panel** | ❌ same |

Components themselves are already TEA-correct: `update(msg, slice) → [slice,
effects]` returns effects as **data**, and the codebase already redispatches via
the `msg` effect (`effects.js:139` — `{type:'msg', msg}` routes to `dispatchMsg`
or `applyMsg`). So the contract is right; only the **runtime's home** is wrong.

2-A inverted the *imports* (panel calls dispatch's `runEffects`/`applyMsg`/… via
the `panel-host` port). 2-B **moves the runtime up** so the call direction is
natural: dispatch drives panel components downward; panel never calls up at all.

## Target architecture

```
dispatch/   ← owns the WHOLE runtime: root reducer driver + component fan-out +
              effect executor + post-dispatch finalizer. Reads the component
              registry from panel/api (dispatch→panel, DOWN, legal).
panel/api   ← a pure component-framework SURFACE: the registry (`components`,
              `registerComponent`), read-helpers (getModel-backed getters,
              filter/scroll/sel reads), the view-contribution system, the
              `(msg,slice)→[slice,effects]` contract + `augmentMsg`. NO code that
              drives a dispatch. Imported BY dispatch, imports nothing upward.
panel/*     ← Components: return `[slice, effects]`. Never call the runtime.
```

After 2-B the `panel-host` seam is **largely retired** — its slots existed to let
panel call up; once panel no longer calls up, they have no callers. (A residual
may remain for genuinely-injected services; see "What panel-host becomes".)

## The real blast radius: panel-internal `dispatchMsg` callers

Components are easy (they already return effects). The hard part is that
`dispatchMsg` — the fan-out we want to move *up* into `dispatch` — is **called
imperatively from inside `panel/`**. Move the callee up and every one of these
becomes a fresh `panel→dispatch` edge unless its caller is also handled. Census
(verify exhaustively as Stage 0 — `grep -rn dispatchMsg js/`):

| Caller | Sites | Nature | Fate under 2-B |
|---|---|---|---|
| `dispatch/dispatch.js`, `dispatch/input.js` | many | already in dispatch | import fan-out from `dispatch/`, no direction change |
| `app/state.js`, `app/tui.js` | 4 | boot/top | import from `dispatch/`, legal (app→dispatch) |
| **`panel/nav-state.js`** writers (`appendViewerLines`, `setViewerContent`, group toggles, multisel, scroll) | ~9 | **write-side facade mis-filed in panel** | **re-home nav-state's WRITE half to `dispatch/`** (it's dispatch-ish, not a Component). Read half (`getSel`/`allPanels`/`multiSelCount`) stays a panel/leaf read. |
| **`panel/commands.js`** run-closures | ~10 | **command runner mis-filed in panel** | command specs become **declarative** (return a Msg/effect descriptor) OR the runner moves to `dispatch/`; the cmdline (dispatch) executes. |
| `panel/free-config-view.js` | 2 | view-side gesture writer | fold the dispatch into the layout Component's update, or re-home the writer. |
| `panel/viewer/tabs.js` | 2 | content-tab writer (also the feature-host wirer) | the `addContentTab`/`updateContentTabLines` writers emit through the fan-out; since they're already injected via `feature-host`, route their dispatch through dispatch-owned code. |
| `panel/api.js` internal (`refreshAll`, `setActiveTab`, `hub.setDispatch`) | 3 | api's own convenience dispatches | move with the fan-out (they become dispatch-owned). |

**Key reframing:** `nav-state` (write side) and the command-runner are *not
Components* — they are dispatch translation code that happens to be filed under
`panel/`. 2-B is partly recognizing that and re-homing them, exactly as the
render-exit arc recognized `render/panel.js` was a leaf. This is where the work
is, and why 2-A chose the seam instead.

## The 5 imperative up-calls → returned descriptors

2-A routed these through the seam; 2-B converts them to effects-as-data (the
canonical TEA form). Most have a clean path because the substrate exists:

| Up-call (2-A seam slot) | Call site today | 2-B conversion |
|---|---|---|
| `applyMsg({register_push})` ×2 | `select.commit()` (viewer keyboard `y` → in `update()`); `select.settle()` (called from **`dispatch/input.js:628`**) | `settle()`: already returns text → `input.js` (dispatch) does the `applyMsg`. `commit()`: emit a `{type:'msg', msg:{register_push}}` effect from viewer `update()`. |
| `applyMsg({terminal_exit})` (`leaveTerminalMode`) | docker `dockerExec`/`dockerShell` **effect handlers** (run inside dispatch's `runEffects`) | the handler already executes in dispatch — once docker's effect handlers are registered/run by dispatch (see registerEffect inversion), it calls `applyMsg` directly, no seam. |
| `streamCommand` | same docker effect handlers | emit a `stream`/`run_action`-style effect descriptor (the streaming effect handlers already live in `effects.js`); drop docker's direct `streamCommand` import. |
| `cleanup` (`:quit`) | `commands.js` quit closure | `:quit` becomes a Msg/effect (`{type:'quit'}`); a dispatch effect handler runs `cleanup()` + `process.exit` (mirrors Bubble Tea's `tea.Quit`). |
| `showHelp` (`:help`) | `commands.js` help closure | `:help` emits a `show_help` effect; the handler (already in `actions.js`/`effects.js`) shows the overlay. |

## registerEffect inversion (effect-manager registration)

Today `api.registerComponent` calls `comp.installEffects(registerEffect)` and
`registerEffect` proxies up to `dispatch/effects` (a panel→dispatch call, 2-A
seamed it). Invert by **pull, not push**: `dispatch` (in `installBuiltins`, or a
new boot step) iterates `api.getComponents()` and calls each
`comp.installEffects(effects.registerEffect)` with dispatch's own registry
(`dispatch→panel` read of the spec, DOWN, legal). `panel/api` stops touching the
effect registry entirely.

## What `panel-host` becomes

After 2-B, panel makes no upward calls, so the `panel-host` slots
(`applyMsg`/`runEffects`/`registerEffect`/`streamCommand`/`cleanup`/`showHelp`)
have **no callers** and the seam + `dispatch/host-wiring.js` are deleted. If any
genuinely-injected capability survives (none expected), it keeps a minimal port.
**2-A's seam was interim scaffolding that pinpointed exactly the call sites 2-B
converts — nothing wasted.**

## The finalizer + blessed-exception B

`_finalizeDispatch` (scroll clamp + viewer `innerH` write + per-pane reconcile)
moves *with* the fan-out into `dispatch`. This is its **correct home** — it is
once-per-dispatch derived-geometry runtime work, not component code. Blessed-
exception B (the `innerH` finalizer write, re-confirmed load-bearing 2026-06-17)
is unaffected in behavior; it simply stops being a "panel writes derived state"
exception and becomes "the dispatch runtime writes derived state", which is
exactly what a runtime is allowed to do. **2-B may let B's `blessed` note be
downgraded** (re-evaluate after the move; do NOT assume).

One subtlety: `applyMsg` (root path) does **not** run `_finalizeDispatch` today;
only `dispatchMsg` (component path) does. Preserve that asymmetry on the move (or
unify deliberately, with the finalizer's `_inScrollFinalize`/depth guards intact)
— do not accidentally start finalizing every root Msg (perf + the bench-visible
`calcLayout` cost the layout memo exists for).

## Staging (each stage independently green: suite + 10 smokes + bench parity)

0. **Measure first (MANDATORY).** Reuse `js/scripts/dep-walker.js`'s parser in a
   sim to confirm the *target* import graph is acyclic AND that the move doesn't
   manufacture a new cycle (the render-exit naive-move regressed 5→6). Produce the
   exact caller census for every `dispatchMsg`/`dispatchKeyToFocused` site. No
   code until the sim is green. (This is the render-exit method lesson.)
1. **registerEffect inversion** — dispatch pulls `installEffects` from the
   registry; drop `api.registerEffect` + the seam slot. (Smallest, isolated.)
2. **Re-home `nav-state` write half** → `dispatch/` (read half stays). Repoint
   the ~9 writer callers.
3. **Commands → declarative / runner-in-dispatch.** Convert `commands.js` run
   closures to return descriptors (or move execution to dispatch). Includes the
   `:quit`/`:help` descriptor conversions.
4. **Convert the remaining imperative up-calls** (`select` register_push, docker
   `terminal_exit`/`streamCommand`, `free-config-view`, `viewer/tabs` writers) to
   returned effects.
5. **Move the fan-out + finalizer** (`dispatchMsg` family + `_finalizeDispatch`)
   `panel/api.js → dispatch/fanout.js`. `panel/api` keeps registry + reads +
   contributions only. Repoint the now-external importers (all in dispatch/app —
   legal down-edges).
6. **Delete `panel-host` + `host-wiring`** (no remaining callers). Re-evaluate
   blessed-exception B's note.
7. **Verify** — `dep-walker` SCC `[]` AND `panel` strictly below `dispatch` with
   no `panel→dispatch` edges at all (stronger than 2-A); suite/smoke/bench green;
   update `PRINCIPLES.md`, `blessed-exceptions.md`, this doc + the re-audit memory.

## Risks

- **Every-Msg blast radius.** The fan-out is THE component dispatch path; the
  finalizer runs after every component dispatch. A subtle move-bug (depth counter,
  `_inScrollFinalize` guard, freeConfig freeze gate, finalize asymmetry) corrupts
  scroll/clamp/repaint for the whole app. The smokes (post-input frame asserts)
  and `test-select`/`test-jobs-activate` (clamp seeds) are the teeth.
- **`nav-state` is dual-consumed** (panel reads + dispatch/app/render/overlay).
  Splitting read/write must not break the read importers (`render/footer`,
  `overlay/copy`, etc. read `getSel`/`multiSelCount`). Keep reads where they are.
- **Require cycles on the move.** `dispatch/fanout.js` will import `panel/api`
  (registry/reads), `model/store`, `leaves/*`, `dispatch/effects`. Confirm no
  eager cycle (api must not eager-require fanout — it currently hoists
  `dispatchMsg`; that hoist disappears).
- **Boot order.** The fan-out + finalizer relocation changes what loads when;
  re-run the real-PTY boot smoke (it caught the F3 lazy-require fallout).
- **Scope creep into `commands`/`free-config`.** These are sizeable; if Stage 3/4
  balloon, they can ship as their own sub-arcs — but the fan-out move (Stage 5)
  is blocked until all panel-internal callers are handled (all-or-nothing for the
  `panel→dispatch` = 0 goal).

## Honest cost/benefit

- **Benefit:** the codebase structurally matches canonical TEA — the runtime owns
  the loop and sits above components; `panel` becomes a true leaf layer with
  **zero** `panel→dispatch` edges (2-A left the runtime in panel and inverted
  imports; 2-B removes the need for the inversion). Cleaner story for replay
  (a single runtime entry), for new Component authors (you return data, full
  stop), and it retires the `panel-host` seam.
- **Cost:** large, every-Msg blast radius, touches `nav-state`/`commands`
  re-homing that are tangential to "the loop". Days, not hours, with a full
  pre-release review pass.
- **Not urgent:** 2-A already bought the acyclic DAG. 2-B buys *correctness of
  placement*, not a missing invariant. Recommend scheduling it as a dedicated arc
  only when there's appetite for a runtime-touching refactor — not bolted onto
  another release's tail.
