# v0.6.3 deep architecture review

> **Status: in progress.** 4-track adversarial arch review completed
> 2026-06-07 (tip `60d9c8b` on `v0.6.3`). This doc captures findings +
> sequencing plan. Each track lands as a labelled phase on the same
> branch; v0.7 is unblocked once the four phases close.

## Guiding principle — Elm/TEA discipline

Every refactor in this arc respects the original Elm Architecture
contract:

- `Component.update(msg, slice) → (next, cmds)` — fixed signature, two
  args. **Never** `(msg, slice, model)`. Confirmed via Bubble Tea +
  Bubbles + Elm research (2026-06-07) — see `[[elm-tea-discipline]]`.
- When a reducer needs a root fact (currentGroup, config, modes), the
  **dispatcher** reads `getModel()` at dispatch time and threads the
  needed facts via the Msg payload (the `modelBundle` pattern from
  v0.6.3 Phase 3c).
- Reducer arms are pure functions of `(slice, msg)`. No `getModel()`,
  no `getInstanceSlice()`, no module-globals.
- Tempting shortcuts (model-as-3rd-arg, ctx.getModel inside leaves,
  ambient access via `_xxxRef`) are rejected even when piecemeal
  threading feels boilerplatey — the boilerplate IS the feature: it
  surfaces "does this reducer really need this fact?" and keeps
  Components plug-and-play.

## Frame

lazytui is a TUI built as a **tiling window manager + tab system**:

- **Pane = window**: identity (`paneId`), bounds (x,y,w,h), chrome
  (`[≡] [_] [+] [X]`, hotkey, title), focus state, holds 1+ tabs
- **Tab inside pane = document**: stable key, content, per-tab view
  state (scroll, search, cursor)
- **Layout = window manager**: tiling rules, view modes (normal /
  half / full), drag/resize/swap in free-config mode

The four-track audit asks whether the codebase REFLECTS this mental
model — or whether vocabulary, file layout, dispatch wiring, and state
ownership have drifted into singleton-detail assumptions that block
multi-instance / multi-tab non-detail panes.

## TL;DR

**Half-migrated.** v0.5/v0.6 established the right shapes (TEA,
panes-as-cells in v0.6.1, dual-write `panelBounds[type|paneId]`,
single-writer per slice). The completion work was deferred and is
now visible as drift:

- **Identity-by-paneId** was data-modeled in v0.6.1 but not propagated
  to consumers (focus, slice keying, chrome)
- **TEA reducer purity** was completed for `panel/layout.js` in
  v0.6.3 but not extended to `viewer/`, `navigator/groups`,
  `navigator/config-status`
- **Mouse routing** was never refactored to match keyboard's
  `CHAIN_MODES` pattern
- **Window-manager logic** was split across two `layout.js` files when
  free-config landed and never recombined

## Cross-cutting theme — identity confusion

All four tracks land on the same root cause expressed at different
levels: the runtime indexes by `pane.type` / Component kind in places
where it should index by `pane.paneId`. Today every Component has
exactly one instance (singleton-per-kind), so `tab-id === pane.type
=== pane.id === Component kind` everywhere — and a `===` test against
`pane.type` happens to match the right pane. The second instance of
any kind breaks every such comparison silently.

Concrete sites (from Track 2):

- `panel/layout.js:483, 517, 925` — focus comparison/write by type
- `render/layout.js:653, 728, 734, 816` — focus highlight + half-view
  resolution by type
- `panel/layout.js:756` — `slice.focus === pane.id || slice.focus ===
  pane.paneId` (the one site that already handles dual identity)
- `dispatch/input.js:211` writes focus as `ownerPaneId`,
  `leaves/pane-tabs.js:708, 722, 810` writes as `paneId`,
  `panel/layout.js:925` writes as `entry.type`,
  `dispatch/actions.js:239` writes as the literal `'actions'` —
  five producers disagreeing on what `focus` IS

## Track 1 — code layout

**Two `layout.js` files** owning window-manager logic across layers:

| File | LOC | Owns |
|---|---|---|
| `panel/layout.js` | 1014 | Component reducer (viewMode, arrange, free-config, drag/resize, tabs, columns) |
| `render/layout.js` | 1334 | View-mode dispatch (renderNormal/Half/Full), calcLayout, bounds, footer rendering, overlay dispatch |

`render/layout.js` is the single biggest layering inversion in the
tree — it transitively depends on all 12 overlays. Both files own
"window-manager" concerns; the split is by layer, not by concern.

**Window-manager subsystem is scattered.** The "gesture engine"
(free-config drag/resize/pool-drag/tab-drag) is 1623 LOC in 4 files
in `leaves/`. `render/free-config-view.js` (148 LOC) is misplaced in
`render/` next to general-purpose renderers. `leaves/pane.js`,
`leaves/pool.js`, `leaves/arrange.js` are window-manager primitives.

**Vocabulary drift.** "Panel" (Component, pre-v0.6.1) vs "pane"
(window cell, v0.6.1+) — directory still called `panel/`, field names
still `panelBounds`/`panelType`/`panelTypes`. v0.6.1 introduced the
new concept; the rename never followed.

### Track 1 — recommended phase

**Phase A1: rename `panelBounds` field → `paneBounds`** (or accept
`panelBounds` as historical and document; pick one and stick). One
file, mechanical, low risk. Sharpens vocabulary before the bigger
moves.

**Phase A2: carve `js/wm/`** — merge `panel/layout.js` +
`render/layout.js` + `render/free-config-view.js` + 4 ×
`leaves/free-config*.js` + `leaves/tab-drag.js` + `leaves/pane.js` +
`leaves/pool.js` + `leaves/arrange.js`. The window-manager subsystem
becomes one directory the reader can audit start-to-finish. Suggested
shape:

```
js/wm/
  layout.js              ← merged Component
  layout-render.js       ← calcLayout + renderNormal/Half/Full
  view.js                ← from render/free-config-view.js
  pane.js, pool.js, arrange.js
  gesture/
    free-config.js       ← keyboard
    free-config-mouse.js ← mouse
    free-config-core.js  ← shared
    pool-drag.js
    tab-drag.js
```

## Track 2 — pane-as-window

The pane abstraction is **60-70% built**. Data model is there
(`paneId`, `tabs[]`, dual `panelBounds` keys, pane-tabs leaf is
`paneId`-parameterized). The runtime treats panes as singleton-per-kind
everywhere it matters.

### Blockers

**B2.1 Component slices keyed by Component name, not paneId.**
`panel/api.js:162` `route.setInstance(comp.name, comp.name,
comp.init())`. Every Component gets exactly ONE slice at `id=kind`. A
second pane of kind `docker` cannot exist — its `wrapAsPane(entry,
'pane-docker#2')` placement would point at the same slice, sharing
scroll/filter/multiSel with the first.

**Single biggest blocker.** Every other smell below is downstream of
this one.

**B2.2 `slice.focus` semantics are ambiguous.** Five producers write
it with paneId / type / kind. Eight consumers compare it against
`pane.type`. Works by accident; breaks on second instance.

**B2.3 Detail-specific call sites.** `getInstanceSlice('detail')`
hardcoded in `panel/viewer/tabs.js:37,80,84`, `panel/viewer/select.js:45`,
`app/state.js:33`, `panel/viewer/viewer.js:407`, `dispatch/input.js:24`.
Data shape supports multi-tab; call sites pin singleton-detail.

**B2.4 `chromeFor` branches detail vs non-detail asymmetrically.**
`render/decor.js:91-119`. The `[≡]` glyph slot is overloaded by kind:
detail → tab-list trigger; non-detail → pane-select trigger.
Multi-tab non-detail panes need BOTH; can't have both.

**B2.5 `currentGroup` pinned globally.** Every viewer read passes
`getModel().currentGroup`. A second viewer pane cannot show a
different group's tabs.

### Track 2 — recommended phase

**Phase B1: slice keying by paneId.** The atomic refactor:

1. `panel/api.js:162` — at register time, mint NO instance.
2. `app/state.js` initState — walk arrange, call `route.setInstance(pane.paneId, kind, comp.init())` per placed pane.
3. `route._primaryByKind` becomes derived (first paneId of kind).
4. Every `getInstanceSlice('detail')` site becomes
   `getInstanceSlice(paneIdForKind('detail'))` OR — better — the
   caller threads the paneId from its own context (mouse hit-test
   already knows it; tab dispatchers already use `ownerPaneId`).
5. `slice.focus` is **defined** to be a paneId. All comparators
   `p.type === slice.focus` rewrite to `p.paneId === slice.focus`.
   Producer disagreement collapses.

Unlocks (in order): multi-tab non-detail panes (already works in the
leaf), multi-instance non-viewer kinds (e.g., two docker views),
multi-viewer.

## Track 3 — input dispatch

**Keyboard pipeline: clean.** filter → mode chain → focused Component
→ handleNormalKey → render. Textbook event loop. Adding a new mode
is a one-line edit to `modes.js` + entry in `_modeHandlers`.

**Mouse pipeline: drifting.** `handleMouse` in `dispatch/input.js` is
~440 lines of straight-line block-by-block routing:

1. Chrome glyph hit-tests (collapse, close, tab-list trigger,
   pane-select trigger)
2. `tabListMode` block
3. `paneSelectMode` block
4. `freeConfigMode` block (5 sub-blocks: pool-drag, tab-drag,
   tab-bar press, panelList press, free-config drag)
5. `isChainActive` guard
6. Wheel
7. Selection drag
8. Click-loop (focus + select)

Each new modal overlay = another `if (model.modes.fooMode) {...}`
block inserted at the right spot. No registry. `jobsMode` already
exists in MODES with NO mouse block.

### Other issues

**T3.1** Modes are NOT mutually exclusive — `mode_set` doesn't clear
siblings. `dispatch.js:712-716` chain "first active wins" assumes
mutual exclusion.

**T3.2** Wedge guard is keyboard-only. Cmd handlers throwing leave
flag/slice state inconsistent (acknowledged for one path in
`set_arrange`; general resilience absent).

**T3.3** `_suppressesChromeClicks` (8 modes hand-rolled) vs
`isChainActive` (14 chain modes from registry) — two non-derived
lists for the same concept. Should be a `chromeClickThrough` column
on the `modes.js` MODES registry.

**T3.4** Click routing: `focus_set` dispatches BEFORE `navSelect` →
`show_info` fires against stale cursor. Two cascades per click; first
half-correct.

**T3.5** Three call sites encode "exit pane-select" (Enter via
dispatch.js, mouse press via input.js, Esc via dispatch.js) — no
shared exit primitive.

**T3.6** Direct calls bypass dispatch: `actions.js:284 showHelp()`,
`actions.js:270 refreshAll()` — not in the event log, can't be
filter-intercepted, can't be Msg-tested.

### Track 3 — recommended phase

**Phase C1: `_modeMouseHandlers` registry.** Mirror
`_modeHandlers` keyboard pattern. `handleMouse` walks `CHAIN_MODES`
in precedence order, finds the first claiming mouse handler, falls
through to normal-mode click/wheel path. Wedge-guard lifts cleanly
(same try/catch + `mode_clear` on throw). `_suppressesChromeClicks`
collapses into a `chromeClickThrough` column on the registry.

**Phase C2: chrome-widgets registry.** Replace the four inline
hit-test calls in `dispatch/input.js:131-184` with a registry walked
in order. New pane-chrome widget = one entry; ordering is explicit.

After this `handleMouse` shrinks from ~440 lines to ~80, and each
overlay's mouse handling moves next to its key handler in
`dispatch.js`, restoring keyboard/mouse symmetry.

## Track 4 — state ownership

**Genuinely clean at the macro level.** Persistent / ephemeral split
correctly drawn (`:save-layout` serializes only `arrange`). Out-of-TEA
stores (`feature/jobs`, `feature/history`) are minimal and fenced.
Init shapes complete post-v0.6.3 (the MODES-derive in `runtime.js`
closes the `paneSelectMode` drift class). Single-writer-per-slice
holds for WRITES.

**The leaks live at the READ boundary.** v0.6.3 Phase 3 completed
TEA discipline for `panel/layout.js` (zero `getModel()` reads in
reducer arms). Same arc was not extended to other Components:

- `panel/viewer/viewer.js` — **16** `getModel()` reads in reducer arms
  (305, 444, 472, 551, 587, 606, 639, 657, 696, 747, 794, 812, 846,
  927, 1014, 1039)
- `panel/navigator/groups.js` — 5 reads (214, 221, 243-244, 249, 358)
- `panel/navigator/config-status.js` — 2 reads (60-67 helper called
  from arm at 375)

**Cross-slice reads inside arms:**

- `panel/layout.js:672` — `tab_drag_motion` reducer reads
  `getInstanceSlice('detail').tabBounds` (cross-slice, introduced by
  v0.6.3 P4.1)
- `panel/navigator/config-status.js:60-67` called from arm at 375 —
  reads layout slice for the Component's own pool-entry config

**Direct root-model writes outside the reducer:**

- `app/state.js:86` — `m.config = JSON.parse(...)`
- `app/state.js:96-97` — `m.projectDir = ...`, `m.configPath = ...`
- `app/state.js:138` — `getModel().register = ...` (commented BLESSED)

Boot-time only, but breaks "reducer is sole writer to root model".

**Render-time direct-mutation:**

- `panel/viewer/viewer.js:1034` — `slice.tabBounds = ...` in-place
  during render. Other render-side writes go through
  `route.setInstanceSlice` with immutable spread.

**Real perf finding:** `viewer_append` (line 607) and
`viewer_append_lines` (line 658) — the streaming hot path — compute
`pt.flatTabInfo(slice, m, m.currentGroup)` and **never use the
result**. Dead work in the per-line path. Per
`bench-tea-overhead.js`, `flatTabInfo` is the 71µs/op
`getMergedActions` call. Trimming this could lift `viewer_append`
above its current 11.7k ops/sec.

### Track 4 — recommended phase (Elm-disciplined)

Per the guiding principle, all Phase D work threads facts via Msg
payload from the dispatcher; **no** `model` arg added to
`Component.update`.

**Phase D2 ✅ SHIPPED (`ba623ff`).** Killed dead `flatTabInfo` calls
in `viewer_append` / `viewer_append_lines`. `bench-hotpaths.js`
`append from empty` 11.7k → 14.5k ops/sec (+23%).

**Phase D1: viewer + groups + config-status `getModel()` retirement.**
Piecemeal per Msg type. For each arm that reads `getModel()`:

  1. Audit what root fact the arm actually needs.
  2. Dispatcher (mouse handler / chain key handler / Cmd emitter /
     panel/viewer/tabs.js wrapper) reads at dispatch time.
  3. Threads via `msg.X` (e.g. `msg.currentGroup`, `msg.actionCount`)
     or via the existing `pt.modelBundle(model, groupName)` helper.
  4. Reducer arm reads `msg.X` only.

The `_withDerivedFields` finalizer in `viewer.js` is the trickiest:
runs on every Msg, reads `m.currentGroup`. Two routes evaluated:

  - **(a) thread currentGroup into every viewer Msg** — uniform but
    heavy; every dispatcher pays an extra field.
  - **(b) ⭐ keep `_withDerivedFields` outside the reducer-purity
    wall**, document it as the "post-reducer derivation hook" —
    matches Bubbles' "setter between updates" idiom (per
    `[[elm-tea-discipline]]`). The finalizer becomes a hook
    explicitly NOT-a-reducer; reducer arms stay pure; finalizer
    derives view fields freely.

Plan picks **(b)**. The finalizer's seam gets documented in
PRINCIPLES.md as part of Phase D1.

**Phase D3: route boot-time direct root-writes through Msgs.**
`set_config` / `set_project_dir` / `set_register` arms in root
reducer. Today these are direct `m.config = ...` writes in
`app/state.js`. Becomes Msg-driven end-to-end.

**Phase D4: fix cross-slice read at `panel/layout.js:672`.**
`tab_drag_motion` reducer reads `getInstanceSlice('detail').tabBounds`.
Thread `tabBounds` via Msg from the input.js tab-drag dispatcher
(already has the slice in scope).

**Phase D5: kill in-place render-time mutation at
`panel/viewer/viewer.js:1034`** (`slice.tabBounds = ...`). Match the
disciplined render-side write pattern via
`route.setInstanceSlice(id, { ...slice, tabBounds })`. Other
render-side writes in the codebase already use immutable spread.

## Sequencing plan — all four phases land on v0.6.3

Per dependency structure:

1. **Phase D** first (TEA completion under Elm discipline) —
   piecemeal Msg-threading across viewer/groups/config-status
   reducer arms. D2 already shipped (dead-work removal). D1
   includes the `_withDerivedFields` seam decision.
2. **Phase B** next (slice keying by paneId) — Elm-faithful
   change: route by INSTANCE identity (paneId) not by Component
   kind. Unblocks multi-instance/multi-tab. Foundational for v0.7.
3. **Phase A** reorg (`js/wm/` directory) — easier AFTER Phase B
   because the pane-vs-panel vocabulary settles. `panelBounds` →
   `paneBounds` happens here.
4. **Phase C** last (mouse registry) — orthogonal; mirrors the
   keyboard's `_modeHandlers` chain registry under the same
   Elm/TEA discipline (dispatcher reads model, threads facts via
   Msg payload, mouse handlers stay pure).

After all four phases:

- Multi-instance + multi-tab non-detail panes are structurally
  unblocked
- TEA discipline is universal (no `getModel()` reads in reducer arms
  anywhere)
- `js/wm/` is one auditable window-manager subsystem
- Keyboard and mouse pipelines are symmetric
