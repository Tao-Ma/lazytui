# v0.5 — Layout Component (the frame becomes a Component too)

> **Status (2026-05-28): shipped on `v0.5-tea` (15 commits, tip `a9dc456`).**
> A successor initiative to the Phase A → B → C arc that landed earlier this
> branch ([docs/v0.5-layering.md](v0.5-layering.md)). Where that arc made
> every PANEL a Component, this one makes the FRAME a Component too —
> closing the last gap where v0.5's correctness properties hold for
> everything except the container itself.
>
> Awaits user-declared release closure (merge `v0.5-tea` → `main` + tag).
> The Migration plan, End-state shape, and Open implementation questions
> sections below have been updated in place with per-phase ✓ markers and
> resolutions so the doc describes what actually shipped.

## Why this doc exists

Phase A → B → C made every in-tree panel a Component (9/9), with
single-writer per layer and slice-private state. The "frame" surrounding
those panels — the arrangement, focus, view mode, design mode — stayed at
the root model. That worked, but it left one asymmetry:

- **Panels** own a slice + an `update`, written by exactly one writer.
- **Frame** is scattered across root-model fields (`model.layout`,
  `model.focus`, `model.viewMode`, `model.modal.design`, `model.layoutDirty`)
  and across `layout.js` / `design.js` / `model-design.js` / `yaml-layout.js`,
  written by ~15 root-reducer branches plus a "blessed exception" for the
  render-pass geometry writes in `layout.js`.

This doc proposes the next move: a **chrome Component** named `layout` that
owns the frame the same way each panel owns itself. The panels themselves
nest under it as a composition tree, and dispatch becomes Elm-style with
mapped child Msgs. As part of the move, the Plugin API retires — every
in-tree thing was already a Component, so the Plugin path was load-bearing
only for external authors, and we're dropping that compat target.

## Scale callout

This is **larger than the Phase A → B → C arc**. Phase A → B → C migrated
9 panels one-by-one into a stable Component shape. This arc redefines the
Component shape itself (true-Elm dispatch) AND folds in the frame AND
retires the Plugin API AND moves nav-chrome into per-panel slices. The
phase count is bigger, the audit surface is bigger, and Phase 2
(pass-through dispatch) is a one-way door — every dispatch site changes.

## End-state shape

### Slice tree

```
root-model = {
  ui:     { ... non-nav-chrome bits },         # nav chrome moved into panels
  modes:  { menuOpen, confirmMode, promptMode, cmdMode, copyMode,
            terminalMode, filterMode, prefixMode, listSelectMode, ... },
  modal:  { menu, confirm, prompt, cmd, copy, prefixSeq, ... },  # no design
  config:        { ... },
  currentGroup:  '...',
  register:      { history: [] },

  layout: {                                # ← THE layout Component slice
    arrange:       { left:[...], right:[...], leftWidth, detailHeightPct },
    focus:         'groups',
    viewMode:      'normal',
    dirty:         false,
    design:        { drag, selectedIdx, titleEdit, undo, redo, enabled },
    panelHeights:  {},                     # view-output (own-slice write)
    panelBounds:   {},                     # view-output (own-slice write)
    panels: {
      groups:         { list, expanded, tab, cursor, scroll, multiSel },
      detail:         { contentTabs, terminalTabs, activeTab, search,
                        select, lines, scroll, ... },
      docker:         { ..., cursor, scroll, multiSel },
      files:          { ..., cursor, scroll, multiSel },
      actions:        { ..., cursor, scroll, multiSel },
      history:        { ..., cursor, scroll, multiSel },
      'file-manager': { ..., cursor, scroll, multiSel },
      'config-status':{ ..., cursor, scroll, multiSel },
      stats:          { ... },                            # Monitor — no cursor
    },
  },
}
```

Notes:
- **Per-panel cursor/scroll/multiSel** — Navigator chrome (`ui.sel/scroll/multiSel`)
  moves out of root into each Navigator panel's nested slice. Monitors
  (stats) don't get them. The Viewer (detail) has its own selection state
  already.
- **Design state** lives at `layout.design`, not `modal.design`. Other
  modals (menu, confirm, prompt, cmdline, copy) stay at root — they overlay
  everything, including layout, and are not layout-owned.
- **panelHeights / panelBounds** live on layout's slice. The blessed-
  exception comment in `layout.js` retires — these are now a Component
  writing its own slice during its own render pass.

### Dispatch — true Elm with mapped Msgs

Each Component declares its own Msg space. Top-level Msgs are either:
- **Layout-owned** — handled by layout's update directly (`focus_set`,
  `set_view_mode`, `design_*`, `save_layout`, …).
- **Wrapped child Msgs** — `{ kind: <ComponentName>, msg: <child Msg> }`.
  Layout unwraps, runs the child Component's update on its nested slice,
  stitches the new slice back.

```js
// Dispatch site
const wrap = (kind, msg) => ({ kind, msg });
dispatch(wrap('Groups', { type: 'toggle_group', name: 'a' }));

// Layout's update (sketch)
function layoutUpdate(slice, msg, model) {
  const child = childRegistry[msg.kind];
  if (child) {
    const prev = slice.panels[child.sliceKey];
    const out  = child.update(prev, msg.msg, model);   // child sees own slice
    return {
      slice: { ...slice, panels: { ...slice.panels, [child.sliceKey]: out.slice } },
      effects: (out.effects || []).map(remapChildCmd(msg.kind)),
    };
  }
  // ... layout-owned msg branches
}
```

Cmds returned by a child's update have their continuation Msgs remapped
through the same wrapper (Elm's `Cmd.map`) so callbacks land on the right
child on completion.

Today's `apply_msg` (Component → root) and `dispatch_msg` (root →
Component, Component → Component) collapse into ONE channel: any update
can emit a wrapped Msg targeting any other update. Layout's update is the
routing hub.

### View contributions (decorators retire)

Each Component exposes an optional set of view contributions:

```js
{
  render(panel, w, h, slice) { ... },           // panel grid (panels)
  viewContributions: {
    footerLeft(slice, ctx)  { return '...' },   // optional
    footerRight(slice, ctx) { return '...' },   // optional
  },
}
```

`decorators.js` retires. Footer's left/right segments concatenate
contributions from registered Components in registration order. Same effect
as today's decorator registration; one mechanism instead of two.

## What this absorbs

| Current mess | Where it lives in the new shape |
|---|---|
| `model.layout` written by `runtime.update` + `layout.js` + `model-design.js` | `layout.slice.arrange`, written only by layout's update |
| `model.focus` written by `focus_set` in root | `layout.slice.focus`, written by layout's update |
| `model.viewMode` written by root reducer | `layout.slice.viewMode` |
| `model.modal.design` written by ~10 root branches | `layout.slice.design`, written by layout's update |
| `model.layoutDirty` write-scattered | `layout.slice.dirty` |
| `model.panelHeights` / `panelBounds` (blessed-exception render writes) | `layout.slice.panelHeights/Bounds` (own-slice writes) |
| `ui.sel[panel]` / `ui.scroll[panel]` / `ui.multiSel[panel]` written by root `nav_select` etc. | `layout.slice.panels[panel].cursor/scroll/multiSel`, written by that panel's update |
| `apply_msg` Cmd + `dispatch_msg` Cmd (two parallel channels) | One wrapped-Msg dispatch through layout |
| `decorators.js` registration (Plugin extensibility) | Component `viewContributions` |
| Plugin API surface in `plugins/api.js` (the Plugin half) | Deleted |
| `getPanelDef` Plugin fallback in `layout.js` `rendererFor` | Deleted |

## Architectural decisions (locked)

| Decision | Choice |
|---|---|
| Layout becomes a Component | Yes — chrome Component (no `panelTypes`) |
| Scope | Frame + nested panel registry (Maximal) |
| Routing model | True Elm with mapped child Msgs |
| `ui.sel/scroll/multiSel` location | Per-panel nested slice |
| `panelHeights` / `panelBounds` | Move into layout slice |
| Modals migrating | Only `design`; others stay at root |
| Decorators | Fold into Component `viewContributions` |
| Plugin support | Dropped |

## Migration plan

Each phase ships as its own commit; the suite stays green between them.

### Phase 0 — API extensions (no behavior change) — ✓ shipped

- Extended Component definition to accept `panelTypes: undefined` (chrome-only Component).
- Extended Component definition to accept `viewContributions: { footerLeft, footerRight }`.
- Introduced the wrapped-Msg dispatch shape `{ kind, msg }` with a
  back-compat path that routes unwrapped Msgs through today's flat
  dispatch.
- Tests + audit; no production behavior changes.

### Phase 1 — Layout Component skeleton — ✓ shipped (1 a-g + a post-Phase-1 shim sweep)

- Created `js/components/layout.js` (filename followed the Phase 1h
  rename — see resolved Q-1; the doc originally proposed
  `js/plugins/core/layout.js`).
- Migrated into `layout.slice`: `arrange`, `focus`, `viewMode`, `dirty`,
  `design`, `panelHeights`, `panelBounds`, plus a placeholder
  `panels: {}` map (used from Phase 3).
- Migrated ~15 root-reducer branches into layout's `update`: `focus_set`,
  `set_view_mode*`, `design_*` (incl. mouse drag/resize state machine),
  `save_layout`, `restore_layout`, layout edits, …
- `js/layout.js` view code became layout's render contribution.
- `js/design.js` overlay and `js/model-design.js` leaf became layout's
  view + leaf modules.
- `js/yaml-layout.js` became layout's save/restore leaf.
- Panel slices stayed flat at root until Phase 3 nested them.
- Readers updated to `getComponentSlice('layout').<field>`.
- Decorator system still present in this phase (Phase 5 folded it in).
- **Post-Phase-1 shim sweep** removed every back-compat property on the
  root model (`model.focus`, `model.layoutDirty`, `model.layout`, etc.);
  caught 2 real bugs (`--design` CLI flag boot-order; `design_enter`
  wiping `design.enabled`).

### Phase 2 — Pass-through dispatch (the one-way door) — ✓ shipped (2 a-f)

- Implemented wrapped-Msg routing in `dispatchMsg`. Every Component-
  specific Msg arrives as `{ kind, msg }` and routes to exactly one
  Component's `update`.
- Migrated `apply_msg` / `dispatch_msg` effects' inner msgs to be
  pre-wrapped at the emit site.
- Each Component's `update` sees only its own Msg shapes; cross-
  Component messaging uses wrapped dispatch.
- Back-compat path for unwrapped Msgs (added in Phase 0) removed in 2f.
  `dispatchMsg` now LOGS + DROPS unwrapped Component-specific Msgs.
- Audit: every dispatch site was rewritten or routed via `api.wrap()`.

This phase was the **point of no return** — every dispatch site changed.

### Phase 3 — Nest panel slices — ✓ shipped

- Migrated each Component's slice from the flat `componentSlices` map
  into `layout.slice.panels[<name>]`.
- `getComponentSlice(name)` walks the nesting transparently.
- Each Component's `update` body unchanged.
- Registration order is now load-bearing: `layout` first; a `_flatFallback`
  catches misordered registers with a console warning.

### Phase 4 — Per-panel nav chrome — ✓ shipped (4 a-b)

- Moved `ui.sel[panel]`, `ui.scroll[panel]`, `ui.multiSel[panel]` from
  the root model into each Navigator's own slice as
  `slice.nav[panelType] = { cursor, scroll, multiSel }`. A shared
  `js/model-nav.js` leaf handles the 5 Msg shapes uniformly across
  every Navigator (`set_cursor`, `set_scroll`, `multisel_toggle`,
  `multisel_select_all`, `multisel_clear`).
- The uniform reducer Msg names (`nav_select`, `multisel_toggle`,
  `set_panel_cursor`, `panel_reset`, `multisel_select_all`) retired in
  Phase 4b; call sites wrap directly via `api.wrap(<component>, …)`.
- The cross-cutting `dispatch.navSelect(model, panelType, index)`
  helper centralises the `set_cursor` + `show_selected_info` + groups
  cascade combo.
- Helpers (`getSel`, `getScroll`, `isMultiSel`, `toggleMultiSel`,
  `clearMultiSel`, `multiSelCount`) re-pointed to walk per-slice via
  `getComponentOwningPanel(panelType)`.
- One straggler: `model.ui.filters[panel]` remains at root (per-panel
  filter text). Migration to `slice.nav[panel].filter` was deferred —
  see Phase 4c follow-up issue.

### Phase 5 — Decorators fold into Component view contributions — ✓ shipped

- The 2 real row decorators (docker status dot, groups running/total
  badge) inlined into their Components' `render()`.
- `viewContributions: { footerLeft, footerRight }` now stands alone,
  with `api.collectViewContributions(slot, ctx)` composing contributors
  (stable sort by weight, `footerRight` reversed, ` │ ` separator).
- `js/decorators.js` + `test-decorators.js` deleted.
- DECORATORS.md became a one-page retirement note.
- The legacy `decorators: { ... }` map on a Component/Plugin export
  logs + ignores at registration.

### Phase 6 — Plugin API retires — ✓ shipped

- Deleted: `registerPlugin`, `loadPlugins`, `getPlugin`,
  `isPluginPanel`, `pluginNames`, `startRefreshLoops`,
  `stopRefreshLoops`, and the Plugin-fallback branches in
  `getPanelDef` / `getItems` / `getCommands` / `getGroupActions` /
  `statusFor`.
- `cleanupPlugins` → `cleanupComponents`.
- `tui.js` no longer loads YAML `plugins:` JS-plugin entries; logs a
  one-time warning if the block is non-empty. The parser still
  preserves the field for round-trip + YAML config splits (parser-
  level merge, not a runtime hook).
- `layout.js#rendererFor` Plugin fallback deleted; every panel
  resolves to its owning Component's `render(panel, w, h, slice)`.
- Tests retired: `test-plugin-lifecycle.js`,
  `test-plugin-registration.js`.
- Docs reorganised: `PLUGINS.md` rewritten Component-only,
  `PRINCIPLES.md §12` collapsed to a single TEA-discipline section,
  `SPEC.md` relabelled "Component Authoring Spec".

### Phase 1h — directory rename — ✓ shipped (trailing cosmetic)

- `js/plugins/` → `js/components/`, flattening the historical `core/`
  subdir (Plugin-era "built-in vs user" split).
- 15 files git-mv'd; require paths rewritten across 57 files.
- `js/cli.js` BUILT_IN_PLUGINS string list updated.
- Doc cross-references repointed at `components/api.js`.

## Open implementation questions — resolutions

All seven questions resolved by the shipped arc:

1. **Directory rename** — `js/plugins/` → `js/components/`? *Resolved
   (Phase 1h, 2026-05-28).* Renamed + flattened `core/` subdir. ~57 files
   touched for require-path rewrites; suite stayed green.
2. **`getModel()` vs slice access from inside a Component** — *Resolved:
   `getModel()` is the escape hatch.* Components freely call it for
   app-global reads (focus, currentGroup, mode flags). A context-arg
   variant would have rippled through every render/update signature for
   modest discipline gain; deferred indefinitely.
3. **`getComponentSlice` rename → `getPanelSlice`?** — *Resolved: kept.*
   Chrome-only Components (layout, future status-bar) aren't panels, so
   `Panel` would have been a misnomer. The current name reflects the
   underlying registry; the slice-is-panel-data framing lives in docs.
4. **Cmd-result-Msg auto-wrapping** — *Resolved: explicit wrap.* The
   `tick` Cmd and other async-result Cmds carry an explicit
   `msg: api.wrap('owner', innerMsg)` field at the emit site (see
   `docker.js`'s self-re-arming tick pattern). Auto-wrap was a tempting
   convenience but obscured which Component "owns" the continuation;
   explicit beats magic here.
5. **Test surface** — *Resolved: direct-slice-poke kept for unit
   ergonomics.* Many tests still build a synthetic slice and call
   `Component._update(msg, slice)` directly — this is the test-only
   escape hatch the `_update` / `_init` exports exist for. End-to-end
   dispatch happens in `test-component`, `test-live-render`, and the
   panel-specific integration tests.
6. **Boot-time slice initialization** — *Resolved: `registerComponent`
   inits the slice in place.* Each Component's `init()` runs at
   `registerComponent` time; the framework calls it once and stores the
   result via `_writeSlice`. The layout slice's `panels: {}` is seeded
   empty and populated as each panel Component registers. No wrapped
   `init` Msg was needed.
7. **External doc updates** — *Resolved.* `PLUGINS.md`, `SPEC.md`,
   `PRINCIPLES.md`, `CMDMODE.md`, `LAYOUT.md`, `PROJECT.md`, `HUB.md`,
   `STATS.md`, `DECORATORS.md` all updated through Phases 5/6. (Some
   stale cross-references still flagged in the pre-release doc parity
   audit — see the follow-up backlog rather than the spec.)

## Risk and reversibility

- Phase 0 + Phase 1 are reversible (layout Component skeleton; rest of
  system unchanged behind a back-compat shim).
- Phase 2 (pass-through dispatch) is the **point of no return** — every
  Component's dispatch shape changes. Rolling back would require an
  audit-grade reverse migration.
- Phases 3–6 are mechanical and each independently revert-able.

## Out of scope

- **Pure immutability across all updates** — skipped by design in the
  prior arc; still skipped here.
- **Multi-viewer / split-pane** — a future feature this Component would
  enable but is not part of this initiative.
- **New panel kinds (4th taxonomy slot)** — Navigator / Viewer / Monitor
  stays. The layout Component is a 4th *role* (chrome) but not a panel-
  kind; it does not render in the panel grid.
