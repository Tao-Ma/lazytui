# v0.5 ‚Äî Layout Component (the frame becomes a Component too)

> **Status (2026-05-28): proposed, not started.** A successor initiative to
> the Phase A ‚Üí B ‚Üí C arc that landed earlier this branch
> ([docs/v0.5-layering.md](v0.5-layering.md)). Where that arc made every
> PANEL a Component, this one makes the FRAME a Component too ‚Äî closing the
> last gap where v0.5's correctness properties hold for everything except
> the container itself.

## Why this doc exists

Phase A ‚Üí B ‚Üí C made every in-tree panel a Component (9/9), with
single-writer per layer and slice-private state. The "frame" surrounding
those panels ‚Äî the arrangement, focus, view mode, design mode ‚Äî stayed at
the root model. That worked, but it left one asymmetry:

- **Panels** own a slice + an `update`, written by exactly one writer.
- **Frame** is scattered across root-model fields (`model.layout`,
  `model.focus`, `model.viewMode`, `model.modal.design`, `model.layoutDirty`)
  and across `layout.js` / `design.js` / `model-design.js` / `yaml-layout.js`,
  written by ~15 root-reducer branches plus a "blessed exception" for the
  render-pass geometry writes in `layout.js`.

This doc proposes the next move: a **chrome Component** named `layout` that
owns the frame the same way each panel owns itself. The panels themselves
nest under it as a composition tree, and dispatch becomes Elm-style with
mapped child Msgs. As part of the move, the Plugin API retires ‚Äî every
in-tree thing was already a Component, so the Plugin path was load-bearing
only for external authors, and we're dropping that compat target.

## Scale callout

This is **larger than the Phase A ‚Üí B ‚Üí C arc**. Phase A ‚Üí B ‚Üí C migrated
9 panels one-by-one into a stable Component shape. This arc redefines the
Component shape itself (true-Elm dispatch) AND folds in the frame AND
retires the Plugin API AND moves nav-chrome into per-panel slices. The
phase count is bigger, the audit surface is bigger, and Phase 2
(pass-through dispatch) is a one-way door ‚Äî every dispatch site changes.

## End-state shape

### Slice tree

```
root-model = {
  ui:     { ... non-nav-chrome bits },         # nav chrome moved into panels
  modes:  { menuOpen, confirmMode, promptMode, cmdMode, copyMode,
            terminalMode, filterMode, prefixMode, listSelectMode, ... },
  modal:  { menu, confirm, prompt, cmd, copy, prefixSeq, ... },  # no design
  config:        { ... },
  currentGroup:  '...',
  register:      { history: [] },

  layout: {                                # ‚Üê THE layout Component slice
    arrange:       { left:[...], right:[...], leftWidth, detailHeightPct },
    focus:         'groups',
    viewMode:      'normal',
    dirty:         false,
    design:        { drag, selectedIdx, titleEdit, undo, redo, enabled },
    panelHeights:  {},                     # view-output (own-slice write)
    panelBounds:   {},                     # view-output (own-slice write)
    panels: {
      groups:         { list, expanded, tab, cursor, scroll, multiSel },
      detail:         { contentTabs, terminalTabs, activeTab, search,
                        select, lines, scroll, ... },
      docker:         { ..., cursor, scroll, multiSel },
      files:          { ..., cursor, scroll, multiSel },
      actions:        { ..., cursor, scroll, multiSel },
      history:        { ..., cursor, scroll, multiSel },
      'file-manager': { ..., cursor, scroll, multiSel },
      'config-status':{ ..., cursor, scroll, multiSel },
      stats:          { ... },                            # Monitor ‚Äî no cursor
    },
  },
}
```

Notes:
- **Per-panel cursor/scroll/multiSel** ‚Äî Navigator chrome (`ui.sel/scroll/multiSel`)
  moves out of root into each Navigator panel's nested slice. Monitors
  (stats) don't get them. The Viewer (detail) has its own selection state
  already.
- **Design state** lives at `layout.design`, not `modal.design`. Other
  modals (menu, confirm, prompt, cmdline, copy) stay at root ‚Äî they overlay
  everything, including layout, and are not layout-owned.
- **panelHeights / panelBounds** live on layout's slice. The blessed-
  exception comment in `layout.js` retires ‚Äî these are now a Component
  writing its own slice during its own render pass.

### Dispatch ‚Äî true Elm with mapped Msgs

Each Component declares its own Msg space. Top-level Msgs are either:
- **Layout-owned** ‚Äî handled by layout's update directly (`focus_set`,
  `set_view_mode`, `design_*`, `save_layout`, ‚Ä¶).
- **Wrapped child Msgs** ‚Äî `{ kind: <ComponentName>, msg: <child Msg> }`.
  Layout unwraps, runs the child Component's update on its nested slice,
  stitches the new slice back.

```js
// Dispatch site
const wrap = (kind, msg) => ({ kind, msg });
dispatch(wrap('Groups', { type: 'toggle_group', name: 'a' }));

// Layout's update (sketch)
function layoutUpdate(slice, msg, model) {
  const child = childRegistry[msg.kind];
  if (child) {
    const prev = slice.panels[child.sliceKey];
    const out  = child.update(prev, msg.msg, model);   // child sees own slice
    return {
      slice: { ...slice, panels: { ...slice.panels, [child.sliceKey]: out.slice } },
      effects: (out.effects || []).map(remapChildCmd(msg.kind)),
    };
  }
  // ... layout-owned msg branches
}
```

Cmds returned by a child's update have their continuation Msgs remapped
through the same wrapper (Elm's `Cmd.map`) so callbacks land on the right
child on completion.

Today's `apply_msg` (Component ‚Üí root) and `dispatch_msg` (root ‚Üí
Component, Component ‚Üí Component) collapse into ONE channel: any update
can emit a wrapped Msg targeting any other update. Layout's update is the
routing hub.

### View contributions (decorators retire)

Each Component exposes an optional set of view contributions:

```js
{
  render(panel, w, h, slice) { ... },           // panel grid (panels)
  viewContributions: {
    footerLeft(slice, ctx)  { return '...' },   // optional
    footerRight(slice, ctx) { return '...' },   // optional
  },
}
```

`decorators.js` retires. Footer's left/right segments concatenate
contributions from registered Components in registration order. Same effect
as today's decorator registration; one mechanism instead of two.

## What this absorbs

| Current mess | Where it lives in the new shape |
|---|---|
| `model.layout` written by `runtime.update` + `layout.js` + `model-design.js` | `layout.slice.arrange`, written only by layout's update |
| `model.focus` written by `focus_set` in root | `layout.slice.focus`, written by layout's update |
| `model.viewMode` written by root reducer | `layout.slice.viewMode` |
| `model.modal.design` written by ~10 root branches | `layout.slice.design`, written by layout's update |
| `model.layoutDirty` write-scattered | `layout.slice.dirty` |
| `model.panelHeights` / `panelBounds` (blessed-exception render writes) | `layout.slice.panelHeights/Bounds` (own-slice writes) |
| `ui.sel[panel]` / `ui.scroll[panel]` / `ui.multiSel[panel]` written by root `nav_select` etc. | `layout.slice.panels[panel].cursor/scroll/multiSel`, written by that panel's update |
| `apply_msg` Cmd + `dispatch_msg` Cmd (two parallel channels) | One wrapped-Msg dispatch through layout |
| `decorators.js` registration (Plugin extensibility) | Component `viewContributions` |
| Plugin API surface in `plugins/api.js` (the Plugin half) | Deleted |
| `getPanelDef` Plugin fallback in `layout.js` `rendererFor` | Deleted |

## Architectural decisions (locked)

| Decision | Choice |
|---|---|
| Layout becomes a Component | Yes ‚Äî chrome Component (no `panelTypes`) |
| Scope | Frame + nested panel registry (Maximal) |
| Routing model | True Elm with mapped child Msgs |
| `ui.sel/scroll/multiSel` location | Per-panel nested slice |
| `panelHeights` / `panelBounds` | Move into layout slice |
| Modals migrating | Only `design`; others stay at root |
| Decorators | Fold into Component `viewContributions` |
| Plugin support | Dropped |

## Migration plan

Each phase ships as its own commit; the suite stays green between them.

### Phase 0 ‚Äî API extensions (no behavior change)

- Extend Component definition to accept `panelTypes: undefined` (chrome-only Component).
- Extend Component definition to accept `viewContributions: { footerLeft, footerRight }`.
- Introduce the wrapped-Msg dispatch shape `{ kind, msg }` with a
  back-compat path that routes unwrapped Msgs through today's flat
  dispatch.
- Tests + audit; no production behavior changes.

### Phase 1 ‚Äî Layout Component skeleton

- Create `js/plugins/core/layout.js` (or `js/components/layout/` if the
  directory renames ‚Äî see open Q-1).
- Migrate into `layout.slice`: `arrange`, `focus`, `viewMode`, `dirty`,
  `design`, `panelHeights`, `panelBounds`, plus a placeholder
  `panels: {}` map (not used yet).
- Migrate ~15 root-reducer branches into layout's `update`: `focus_set`,
  `set_view_mode*`, `design_*`, `design_mouse_*`, `save_layout`,
  `restore_layout`, layout edits, ‚Ä¶
- `js/layout.js` view code becomes layout's render contribution.
- `js/design.js` overlay and `js/model-design.js` leaf become layout's
  view + leaf modules.
- `js/yaml-layout.js` becomes layout's save/restore leaf.
- Panel slices STAY flat at root for now (Phase 3 nests them).
- Readers updated to `getModel().layout.<field>`.
- Decorator system still present in this phase (Phase 5 folds it in).

### Phase 2 ‚Äî Pass-through dispatch (the one-way door)

- Implement wrapped-Msg routing in layout's update. Every child
  Component's update gets called THROUGH layout from this phase onward.
- Migrate `apply_msg` / `dispatch_msg` effects into wrapped-Msg dispatch.
- Each Component's update sees only its own Msg shapes; cross-Component
  messaging uses wrapped dispatch.
- Back-compat path for unwrapped Msgs (added in Phase 0) is removed.
- Audit: every dispatch site rewrites or routes via a helper.

This phase is the **point of no return** ‚Äî every dispatch site changes.
Validation gate: Phase 1 stable + layout's own dispatch path
dogfooded before flipping the rest of the codebase.

### Phase 3 ‚Äî Nest panel slices

- Migrate each panel's slice from its current home (e.g.
  `_components[name]`) to `layout.slice.panels[name]`.
- `getComponentSlice(name)` walks into layout.
- Each Component's `update` body unchanged.
- Pure relocation; mechanical audit.

### Phase 4 ‚Äî Per-panel nav chrome

- Move `ui.sel[panel]`, `ui.scroll[panel]`, `ui.multiSel[panel]` from
  root into each Navigator's nested slice (`layout.panels.<X>.cursor/
  scroll/multiSel`).
- The uniform root `nav_select` Msg retires; each Navigator handles its
  own selection-changed Msg in its own update.
- Helpers (`getSel`, `getScroll`, `isMultiSel`) re-point to per-slice
  access.

### Phase 5 ‚Äî Decorators fold into Component view contributions

- Each existing decorator handler migrates to a
  `viewContributions.footerLeft|Right` on its owning Component.
- `js/decorators.js` deletes.

### Phase 6 ‚Äî Plugin API retires

- Plugin-side exports (`registerPlugin`, etc.) deleted.
- `getPanelDef` Plugin fallback in `layout.js`'s `rendererFor` deleted.
- Plugin docs section deleted from `docs/` and `CLAUDE.md`.
- `plugins/` directory possibly renames to `components/` (see open Q-1).

## Open implementation questions

To be resolved as each phase is detailed, but worth flagging up front:

1. **Directory rename** ‚Äî `js/plugins/` ‚Üí `js/components/`? *Resolved
   (Phase 1h, 2026-05-28).* Renamed + flattened `core/` subdir. ~57 files
   touched for require-path rewrites; suite stayed green.
2. **`getModel()` vs slice access from inside a Component** ‚Äî Components
   today freely call `getModel()` for app-global reads. Under nesting,
   keep `getModel()` as the escape hatch, or push more reads through a
   layered context arg into `update` / `render`?
3. **`getComponentSlice` rename ‚Üí `getPanelSlice`?** ‚Äî naming follows the
   new shape; or keep `getComponentSlice` to avoid churn.
4. **Cmd-result-Msg auto-wrapping** ‚Äî when a child's Cmd resolves, its
   continuation Msg lands back in layout's dispatch. Auto-wrap with the
   originating child's `kind`, or require the child to wrap explicitly?
5. **Test surface** ‚Äî `js/test/` has many tests that poke `model.layout.*`,
   `model.focus`, `model.modal.design.*` directly. Rewritten to dispatch
   wrapped Msgs end-to-end, or keep direct-slice-poke for unit ergonomics?
6. **Boot-time slice initialization** ‚Äî panel slice init currently runs
   during `registerComponent`. Under nesting + wrapped dispatch, does each
   panel still init via `registerComponent` (now itself a wrapped Msg into
   layout), or does layout's slice define a default `panels` map populated
   at root-model construction?
7. **External doc updates** ‚Äî `CLAUDE.md`, decorator docs, plugin docs in
   `docs/` all need edits to reflect retired surfaces. Audit at Phase 6.

## Risk and reversibility

- Phase 0 + Phase 1 are reversible (layout Component skeleton; rest of
  system unchanged behind a back-compat shim).
- Phase 2 (pass-through dispatch) is the **point of no return** ‚Äî every
  Component's dispatch shape changes. Rolling back would require an
  audit-grade reverse migration.
- Phases 3‚Äì6 are mechanical and each independently revert-able.

## Out of scope

- **Pure immutability across all updates** ‚Äî skipped by design in the
  prior arc; still skipped here.
- **Multi-viewer / split-pane** ‚Äî a future feature this Component would
  enable but is not part of this initiative.
- **New panel kinds (4th taxonomy slot)** ‚Äî Navigator / Viewer / Monitor
  stays. The layout Component is a 4th *role* (chrome) but not a panel-
  kind; it does not render in the panel grid.
