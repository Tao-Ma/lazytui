# v0.6.6 — toward more-pure TEA (pre-fix plan)

**Status:** PLANNING → IN PROGRESS. This is the *pre-fixing* spec: the list of
every impurity that v0.6.6 aims to retire, the pure-TEA technique for each, the
cost/risk, and the per-item decisions. Build work tracks in §6.

**Theme.** Prior releases *accepted* a named set of impurities (the blessed
exceptions B/C and the #D5/#D14 boundaries). v0.6.6's mandate is to revisit the
*movable* ones and drive them toward pure TEA — `frame = f(model)` as far as a
TUI that embeds a PTY can honestly reach.

This is grounded in a full source sweep (2026-06-24), not the ledger alone. See
also `docs/msg-routes.md` (the route + purity catalog), `docs/blessed-exceptions.md`
(the ledger), and `docs/DATAFLOW.md`.

## 1. Where purity stands today (the four categories)

A recap of the mental model, so "fix" vs "accept" is unambiguous:

1. **Intended impure shell** — effect interpreter (`effects.js`), input/handler
   layer (`dispatch/control/*`), subscriptions. TEA *wants* impurity here. Not a
   bug. (Formal exception **C** = the read-and-stamp shell.)
2. **Named deviations** — blessed exceptions **B** (`innerH` finalizer write)
   and **C** (impure-shell reads).
3. **Documented boundaries** — **#D5** (render reads off-model live stores) and
   **#D14** (terminal/xterm non-TEA island).
4. **Retired** — paneBounds/tabBounds/innerH render-writes, set_scroll clamp,
   wall-clock-in-render, etc. (the trajectory toward empty).

**Verified-clean today (do NOT touch — already pure):** root reducer has zero
`getModel()` reads; `setModel` has exactly one caller (`loop.js:70`); the only
non-commit slice write is exception B (`finalize.js:113`); the wall clock enters
only via `clock_tick → model.now`; component `update()` bodies are pure (reads
relocated to impure accessors). The decision layer is genuinely pure.

## 2. The fix list (priority order)

### FIX-1 — Sample the live stores into the model (the #D5 headline)

**Impurity.** Render/overlays read off-model live stores at paint time, so
`frame = f(model + live stores)`:
- `feature/jobs.list()` — `overlay/jobs.js:97,110` (Running overlay)
- `io/diag-log.snapshot()` — diagnostics overlay
- `feature/history.all()` — `panel/navigator/history.js:22,153`

**Pure-TEA fix.** A subscription samples each into the model on change (e.g.
`model.jobs` / `model.diagLog` / `model.history`) via Msgs; render reads
`model.*` purely. Same technique that already made the wall clock replay-safe
(`Date.now()` → `clock_tick` → `model.now`).

**Sub-items, easiest → hardest:**
- **history** — nearly free; already threaded via `augmentMsg`
  (`history.js:153` stamps `entries` onto the Msg). Promote to a model field.
- **diag-log** — bounded ring buffer → bounded model field; append-Msg per write.
- **jobs** — hardest. The model can hold only a *projection* (status/lines),
  never the live `ChildProcess` handles. Event-driven Msg on add/update/exit.

**Cost/risk.** Medium–Large. jobs/diag are append-heavy → **MUST bench**
model-churn before choosing whole-snapshot vs event-delta Msgs. Replay-positive
(Msgs record the changes). Terminal stays out (#D14).

**Decision — ✅ BENCHED (2026-06-24): whole-snapshot.** Two facts from the
measurement settled it:
- **Frequency (from code):** all three stores mutate on DISCRETE lifecycle
  events — jobs `register/update/close`, diag `push` on warn/error, history
  `register/end`. Notably `history.append(line)` is **silent** (no
  `scheduleRender`), so there is **no per-output-line hot path**; sample on
  lifecycle, not per line.
- **Cost (`js/test/bench-livestore-sampling.js`):** whole-snapshot `{...model, store: list.slice()}`
  runs **15M–45M ops/sec** at realistic caps (jobs 64, diag 200, history 100),
  and 1.3M even at 5000 entries. It's actually *faster* than event-delta at
  these sizes (one `slice()` beats `slice(1)+push()`). The real per-change cost
  is the dispatch+finalizer (~160k ops/sec floor, bench-hotpaths), not the copy.

So: **one `*_synced` Msg per store carrying the full bounded list; reducer is
`{...model, store: list}`.** Simpler than per-entry deltas (no delta Msg types,
no reducer-side cap-trim — the store already caps), faster, and replay-clean
(each Msg is a self-contained snapshot, no delta-ordering dependence). The
entry objects are shared refs, so the snapshot copies the ref array, not content.

### FIX-2 — Eliminate exception B (the `innerH` finalizer write)

**Impurity.** `finalize.js:113` writes derived `innerH` (viewport height)
straight onto the viewer slice via `setInstanceSlice`, bypassing the viewer's
`update`. The one structural "module X writes layer Y's state outside Y."

**Why kept until now.** The viewer reducer *reads* `slice.innerH` to clamp
scroll/cursor, so it had to live in-slice.

**Pure-TEA fix — two options:**
- **(a) Thread-the-fact.** Make `innerH` a pure derived selector (like
  `paneBounds` A.2 / `tabBounds` A.3); stamp it onto the clamp arms' Msgs
  (modelBundle pattern) instead of reading `slice.innerH`.
- **(b) Clamp-in-finalizer.** Move the clamp *entirely* into the finalizer
  (half already there via resize-as-Msg), so the reducer never needs `innerH`
  and it becomes a pure on-read selector.

**Cost/risk.** Medium. Trace every `slice.innerH` reader; scroll/cursor
regressions are the risk (covered by `test-scroll-clamp`, resize tests).

**Decision — ✅ SHIPPED via (a).** Enumerating the readers killed (b): the viewer
reducer reads `innerH` *semantically* in many arms (scroll-to-bottom, page
distance, scroll-to-search-match), not just as a safety clamp — so it genuinely
needs the height at reduce-time. Took (a) the clean way: `viewer.augmentMsg`
(which already receives `(msg, model, slice)` and the slice carries its paneId)
stamps `msg.innerH` from the pane's committed geometry; `update()` projects it
onto the working slice at ONE entry point, so every arm + delegated leaf reducer
reads it unchanged via `_innerH(slice)`. The viewer's own reducer is now the
single writer of `slice.innerH`; the finalizer's exception-B write is deleted.

Outcome: **zero test migration** (`slice.innerH` stays a seed/fallback; tests
that don't go through `augmentMsg` keep their seeds), **fixes a latent
multi-viewer bug** (the finalizer only refreshed the *primary* viewer's innerH;
per-Msg stamping covers every pane), and **bench-parity / marginally faster**
(the finalizer dropped a per-dispatch geometry compute + `setInstanceSlice`).
This reopens TEA review #3 **D16** (which KEPT B): D16's "must stay in-slice"
premise assumed the value wasn't threaded — the `augmentMsg` seam (which D16
didn't weigh) makes threading cheap, so the cost that drove the KEEP no longer
applies. Gate: 97/97 + smoke 11/11 + acyclic + bench parity.

**⚠️ Why this isn't the v0.6.5 reverted attempt.** `docs/v0.6.5-render-exit.md`
§"Blessed-exception B — attempted, REVERTED" records the *same idea* (thread
`innerH` via `augmentMsg`, persist in `update`) being tried and reverted for two
reasons. Both are empirically refuted here, by a concrete mechanism difference:
- **Perf (their ~16× append regression):** their `api.viewportHForPane`
  recomputed layout *per Msg*. Ours calls `getPanelViewportH(…, layout=undefined, …)`
  → the **memoized `boundsFor` selector** (A.2), which returns a cached result
  while the layoutSlice is unchanged. Full bench A/B (append from empty / 10k /
  50k): **165k/21.4k/5774 vs baseline 171k/20.6k/5653 ops/sec — parity, marginally
  faster on the large buffers** (the finalizer also dropped its per-dispatch
  compute + `setInstanceSlice`).
- **Determinism (their broken seed tests):** ours short-circuits — `_paneInnerH`
  returns **0** when no viewer-pane geometry resolves (the minimal unit-test
  setup), so `augmentMsg` leaves `msg.innerH` unset and the `update()` projection
  (guarded on `msg.innerH > 0`) preserves the seeded `slice.innerH`. Probe-
  confirmed: the compute path is never reached in `test-select`; `test-select` +
  `test-jobs-activate` stay green. Their attempt returned nonzero defaults that
  clobbered seeds.

### FIX-3 — Unify external event sources under declarative `Model → Sub`

**Impurity.** The #D13 reconciler (`app/state.reconcileSubscriptions`) is
`Model → Sub` for **hub topics only**. The actual OS/external event sources are
still hand-rolled imperative timers/listeners, outside the Sub system:
- 250ms `setInterval` overlay poll — `app/tui.js:316` (D15: kept, smoke-guarded)
- docker-events stream — `docker.js` `setTimeout` reconnect loops
- clock tick — `effects.js` self-arming
- stdout `'resize'` listener — `tui.js`

**Pure-TEA fix.** Extend the subscription mechanism so these are declared as
Subs (`{kind:'interval'|'resize'|'clock'|'docker-events', …}`) and the
reconciler starts/stops them with teardown — the canonical Elm-Architecture
subscriptions model (timers/resize/foreign sources are all Subs).

**Cost/risk.** Medium–Large (framework extension). Most *principled* item;
absorbs four ad-hoc mechanisms into one declarative seam. The D15 poll's
`smoke/pty-overlay` scenario guards the conversion.

**Relationship to FIX-1:** the mechanism that samples jobs/diag/history into the
model *is* a subscription → FIX-3 is the natural home for FIX-1. But FIX-1
(esp. history) does NOT hard-depend on the full framework; it can land as plain
Msgs first and migrate onto the Sub seam.

### FIX-4 — Converge the dims provider off the live `_term` singleton — ✅ ALREADY DONE (no change)

**Investigated 2026-06-24 — resolved by prior work; no code change.**
`panel/api.js:41` wires the dims provider consumed by the pure leaf
`leaves/render/draw.js` (`setDimsProvider` → `viewportDims`). It ALREADY prefers
`layoutSlice.dims` (model state) and reads live `_term.cols()/rows()` only as the
**pre-first-`term_resized` boot fallback**. That fallback is load-bearing and
covered by `test-overlay-dims §2` (overlays rendered before a layout slice
exists). `layout.init()` seeds `dims {80,24}`, so once the instance is registered
the fallback never fires; it exists solely for the pre-instance boot window.

There is no `getTermSize` symbol (mis-named in the plan). The dims source is
already unified onto the model; the residual io read is a necessary, tested boot
seed → **accept** (moves to §3). Deleting it would only remove a tested safety
net for zero steady-state purity gain.

## 7. FIX-3 design pass — declarative `Model → Sub` for external sources

**Goal.** One declarative mechanism for *every* ongoing event source, so a new
Component (or the app) gets start/stop/teardown for free and `update` never
reaches out. Today there are **two** mechanisms: the #D13 hub reconciler
(declarative, but hub-topics only) and a scatter of **hand-rolled imperative
sources**. Unify the second into the first.

### 7.1 Ground truth — what exists today

| Source | Where | Mechanism | Scope |
|---|---|---|---|
| hub topics | `app/state.reconcileSubscriptions` | declarative reconciler (#D13) | per-pane |
| clock tick | `effects.js#arm_clock` | self-re-arming `setTimeout` Cmd, re-emitted by the `clock_tick` arm while an age overlay is open | app (model-conditional) |
| generic tick | `effects.js#tick` | self-re-arming `setTimeout` Cmd (docker poll uses it) | per-pane |
| stdout resize | `tui.js:324` | `process.stdout.on('resize')` → dispatch `term_resized` | app |
| terminal-overlay poll (#D15) | `tui.js:316` | always-on `setInterval(…, 250)` → `renderTerminalOverlay` | app |
| docker-events | `docker.js` | spawned `docker events` + `setTimeout` reconnect/backoff (incl. the T17 spawn-fail edge) | per-pane |

The #D13 reconciler is **hard-coded to the hub kind**: descriptor `{topic, window}`,
`start = hub.subscribe`, `stop = hub.unsubscribe`. To unify, generalize it.

### 7.2 The core abstraction — a kind-dispatched reconciler

Keep the existing reconcile *loop* (diff desired-set vs `_liveSubs`, stop the
gone, start the new — runs each dispatch via the finalizer hook). Make it
**kind-agnostic** via a small handler registry:

```
SubKind = {
  key(desc)            → stable string id   // dedup + "already live?" test
  start(desc, ctx)     → token              // ctx = { dispatch, applyMsg, scheduleRender }
  stop(token)          → void                // tears down the resource
}
```

The reconciler becomes: for each desired descriptor, `kind = registry[desc.kind]`;
diff by `kind.key(desc)`; `kind.start` the new, `kind.stop` the gone. The **hub
kind is just the first registered handler** (`key = "hub:topic:window"`,
`start = hub.subscribe(onUpdate: scheduleRender)`, `stop = hub.unsubscribe`) — a
pure refactor of today's code, no behavior change. New kinds plug in beside it.

**Stable keys are load-bearing.** The reconciler runs every dispatch, so a
descriptor's key MUST be stable across dispatches for an unchanged logical source
(`"interval:clock"`, `"resize"`, `"process:docker-events"`) — otherwise the
handler is torn down + restarted every dispatch (a timer that never fires, a
subprocess respawned constantly). Key on the *logical source*, never on a
changing field.

### 7.3 Two scopes — per-pane and app-global

- **Per-pane** (today's path): `comp.subscriptions(paneDef, model) → [desc]`,
  walked over placed panes. Gains the new kinds for free (docker-events, docker
  poll move here).
- **App-global** (new): a framework-level `appSubscriptions(model) → [desc]` for
  sources not owned by a pane (clock, resize, overlay-poll). Evaluated in the
  same pass, merged into the one desired-set. *(Fork F2 below: this vs. a
  pseudo-component — `appSubscriptions` is simpler and keeps the framework
  sources out of the component registry.)*

### 7.4 The new sub-kinds

| kind | `start` | `stop` | who declares it |
|---|---|---|---|
| `hub` (existing) | `hub.subscribe` | `hub.unsubscribe` | per-pane |
| `interval` | `setTimeout`-loop (unref) dispatching `desc.msg` every `ms` | clear timer | app: clock, overlay-poll; per-pane: docker poll |
| `resize` | attach `stdout.on('resize')` → dispatch `desc.toMsg(dims)` | remove listener | app |
| `process-stream` | `spawn` + wire stdout lines → dispatch; on-exit → backoff reconnect (owns its reconnect timer, unref) | kill proc + clear reconnect timer | per-pane (docker-events) |

All timers `.unref()` (preserve the clean-teardown-on-quit property the current
code relies on).

### 7.5 Per-source migration + risk

| Source | New form | Risk | Notes |
|---|---|---|---|
| resize | app `{kind:'resize', toMsg}` | **Low** | one listener; idempotent dispatch; verify boot-seed ordering (initState's `term_resized` seed still fires). |

### 7.6 Purity / replay

Unchanged from #D13 and the §12 convention: a Sub is **pure to declare**
(`Model → [desc]`) and **impure to run** (the handlers, in the shell). The Msgs
the handlers produce are in the log → replay reconstructs the model. The model
stays pure; the reconciler + handlers are the named impure shell.

## 8. FIX-1 design pass — sample the live stores into the Model

**Goal (the #D5 headline).** Render reads three off-model live stores at paint
time — `feature/jobs.list()` (`overlay/jobs.js`), `io/diag-log.snapshot()`
(`overlay/diag-log.js`), `feature/history.all()` (`history.js`). Sample each into
the Model so `frame = f(model)` (except the #D14 terminal island), and render
reads `model.{jobs,diagLog,history}`. The bench (FIX-1 bench, §2) decided
**whole-snapshot** Msgs; FIX-3 gives us the Sub framework to ride.

### 8.1 The bridge mechanism — a new `store-mirror` Sub kind

The question FIX-1 turns on: how does a module-local store's mutation become a
`*_synced` Msg, WITHOUT the store (a low `feature/`/`io/` layer) importing the
dispatch loop? Answer — a new **`store-mirror`** sub-kind, consistent with FIX-3.

#### The mirrorable-store contract

Every store FIX-1 mirrors (and every future component that wants a live external
source under the model) implements ONE uniform shape:

```js
// the "mirrorable store" contract
{
  snapshot()          // pure whole-snapshot read (newest-first / display order)
  setOnChange(cb)     // single change-notify seam; the store calls cb() on
                      // every mutation. cb === null unregisters.
}
```

This unifies the two axes that are inconsistent today — the **reader name**
(`all()` / `snapshot()` / `list()` → all become `snapshot()`) AND the
**change-notify mechanism** (hub topic / nothing / direct `scheduleRender()` →
all become the one `setOnChange` seam). `setOnChange` is the load-bearing half:
it is exactly what the Sub's `start`/`stop` wire `cb` into / tear down. The
store's own `scheduleRender()` calls become `cb()` (render still repaints, now
via the `*_synced` dispatch); a store that had no notification at all (diag-log)
gains one.

#### Wiring

- Descriptor: `{kind:'store-mirror', id, store, msgType, field}` — `store` is the
  `{snapshot, setOnChange}` contract object.
- `start(d, ctx)` registers `cb = () => ctx.applyMsg({ type: d.msgType, [d.field]: d.store.snapshot() })` via `d.store.setOnChange(cb)`; `stop` unregisters (`setOnChange(null)`).
- The reducer's `*_synced` arm sets `model[field] = msg[field]` (whole-snapshot).
- Render reads `model[field]`.

Declared **app-global, always-active** in `_appSubscriptions` (the Model must
always mirror the store, so render is pure whenever it reads `model[field]`).
The model field is **seeded `[]` in `store.init()`** (every store starts empty,
so the seed mirrors the store at boot) and the cb drives every update from
there — so `start` does NOT synchronously prime (a nested dispatch inside the
reconciler's `start`, before `_liveSubs.set`, would re-enter and re-`start` the
same sub → recursion). Cheap: the bench showed discrete low-frequency changes +
bounded snapshots.

**Notify granularity — list-shape changes, not every byte.** `cb()` fires when
the snapshot's *shape* changes (an entry added / removed / completed), not on
every within-entry mutation. The only high-frequency case is **history**'s
per-output-line `append()`: it does NOT fire `cb` (that would double the
dispatch rate of streaming output). The entries are shared refs, so a growing
`entry.output` is visible through `model.history` already, and the viewer's own
per-line append dispatch supplies the repaint — same cadence as today. So
history fires `cb` on `start()` + `end()` only. diag-log / jobs are low-frequency
throughout, so they fire on every mutation.

**Fork F5 — `store-mirror` Sub kind vs. an injected dispatch host the store
calls directly.** Recommend the Sub kind: it reuses the FIX-3 framework, keeps
the store decoupled from dispatch (the cb is injected, the store layer imports
nothing upward), and gets teardown for free.

### 8.2 Per-store plan (easiest → hardest)

All three converge on `store.snapshot()` (the contract method, §8.1) — `all()`
and `list()` are renamed; `diag-log` already has `snapshot()`.

| Store | Model field | Reader name | Notes / risk |
|---|---|---|---|
| **history** | `model.history` | `all()` → `snapshot()` | Easiest — already hub-backed + already threaded via `augmentMsg` (`history.js:153`). The navigator's `getItems()`/`augmentMsg` switch to `model.history`. `cb` fires on `start`/`end` (not per-append, §8.1). **Low.** |
| **diag-log** | `model.diagLog` | `snapshot()` (already) | Bounded ring; `diag_synced` on every push. Overlay reads `model.diagLog`. Gains active notification it never had. **Low.** |
| **jobs** | `model.jobs` | `list()` → `snapshot()` (metadata only — output already flows to the viewer slice) | The Running overlay's reads (`overlay/jobs.js:97,110`) → `model.jobs`. Its internal `scheduleRender()` calls become `cb()`. NOTE: `jobs_activate` already threads `msg.job` (exc. C) for the cursor lookup — that STAYS (a specific lookup, not the list); only the list read moves. **Medium.** |

The terminal (#D14) stays out — irreducible.

### 8.3 Sequencing + done

1. `store-mirror` kind (app/state.js) + `setOnChange` seam on one store.
2. **history** (lowest risk; proves the kind end-to-end).
3. **diag-log**.
4. **jobs** (metadata projection; leave `jobs_activate`'s `msg.job` threading).

**Done:** `frame = f(model)` holds for jobs/diag/history; the only remaining
off-model render read is the terminal island (#D14). The #D5 boundary shrinks to
exactly the irreducible PTY. That closes the v0.6.6 arc.

## 9. Code-only TEA re-review — Findings A & B

After FIX-1/2/3 shipped, a **fresh "forget docs/memory/comments, code-is-truth"
TEA audit** (4 parallel reviewers: decision-layer purity, render = f(model),
single-writer, subs/store-mirror) was run to verify the result wasn't just what
the comments *claimed*. Single-writer and the sub/store-mirror machinery came
back clean; two real leaks surfaced.

### Finding A — `tab_switch` read the live model (FIXED)
The viewer's `tab_switch` reducer arm called `ctx.getTabInfo()` → `getModel()`
(×2) + the route/merged-action registries — a live global read inside a
Component `update`, on a routinely-hit path (`]`/`[`, mouse tab-click, stream/
jobs routing). The comment above it *claimed* "not direct model reads" — false.
Fix: derive `{actionTabs, termTabs, total}` from the already-threaded
`msg.viewerModel` bundle via the pure twin `flatTabInfoFromBundle` (which
existed). `ctx` drops to `{paneId, wrap}`. The decision layer is now **fully**
pure — every reducer/modal/Component-update arm is `(state, msg) → [state, cmds]`.
Guarded by a new purity test (tab_switch honors a bundle idx the live model lacks).

### Finding B — stats read the live hub bus (FIXED via the `metrics-mirror` Sub)
`stats.render` read `hub.history(topic, rowKey)` + `hub.schema(topic)` live — a
fourth off-model render read FIX-1 hadn't covered (it folded jobs/diag/history,
not the hub time-series). This is **not** like the FIX-1 stores: `docker.stats`
is a **continuous** sampler (published per-container per 10 s poll). The naive
move — a `store-mirror` firing per publish — would re-introduce exactly the
per-publish dispatch the hub's #D17 deleted, and re-derive the whole matrix M×
per poll.

**Decision — Option B (throttled snapshot-mirror).** A new `metrics-mirror` Sub
kind (app/state.js) subscribes to the hub (retention) and, on each publish,
schedules ONE trailing sample `ms` later (default 250) that mirrors
`hub.matrix(topic)` → a `metrics_synced` Msg; a burst coalesces to one dispatch
per window. Keyed by topic. `stats.render` reads `model.metrics[topic]`.

**Why B (the TEA grounding).** The mechanism is picked by *source rate × view
need* (see PRINCIPLES §12's mirror sub-table), and TEA's canonical handling of a
**high-frequency source feeding a derived view (a graph)** is to **sample at a
bounded cadence, not fold every event** (the throttle-the-subscription pattern —
the same shape as a frame-rate/time-sampled subscription). The alternatives we
weighed:
- **A — per-publish `store-mirror`**: the discrete-store pattern; *re-introduces*
  the #D17 per-sample dispatch + an O(M²)/poll re-snapshot. Right for discrete
  low-freq stores (jobs/diag/history), wrong for a continuous sampler.
- **C — producer-batched** (docker dispatches the matrix once per poll, reusing
  the existing `dockerResult` boundary): cheapest for docker *specifically*, but
  docker-coupled and doesn't generalize to a boundary-less push source.
- **B — throttled mirror** (chosen): source-agnostic (poll *or* push), bounded
  cost (one dispatch per window regardless of wire rate), and the destination is
  **topic-keyed** (`model.metrics[topic]`) so a future push-metrics source (a
  websocket feed) slots into the same shape with no consumer change. A poll-driven
  producer rides it as a low-rate stream (docker's 10 s loop → ~1 dispatch/poll
  after coalescing). Chosen because more (push-capable) monitoring sources are
  expected; B is the one mechanism that spans them.
- **D — accept stats as a 2nd island**: rejected. Unlike the PTY (a genuinely
  foreign reactive widget), the hub is our own in-process structure and the read
  is eliminable — TEA's view-is-f(model) invariant admits no off-model read for
  data we can mirror.

**Logs stay APPEND, not mirrored.** `tail -f`/command output is append-semantics
(every line, ordered, scrollable) and already model-resident in the viewer slice
buffer via `viewer_append*` — throttle-sampling would drop/reorder lines. The
mirror mechanisms are for SNAPSHOT views only.

**Result.** `frame = f(model)` now holds for every panel + overlay except the
terminal island (#D14).

### Finding C — the two render-path diagnostic writes (FIXED via a deferred lane)
Two `diag.warn` calls remained on render/read paths: the **plugin purity/timing
guard** (`panel/plugin-guard.js`, run per `groupActions` on the tab strip +
actions panel) and the **strict-miss tripwire** (`panel/route.js`
get/setInstanceSlice, on per-frame slice reads). A synchronous `warn()` →
`record()` → `_notify()` → the diag store-mirror's `onChange` → **`applyMsg(
diag_synced)`** — i.e. a *re-entrant dispatch from inside render*, feeding
`model.diagLog` (the very frame being drawn); `record()` also called
`Date.now()` on that path. Originally documented-and-accepted ("they don't
change frame output"); on closer reading the re-entrant `applyMsg` is a genuine
read-path side-effect, so it was fixed rather than left.

**Fix — a deferred lane in `io/diag-log.js`.** The two producers `recordDeferred`
into a process-local queue (no `_notify`, no dispatch, no `Date.now()` — the read
path stays pure). The dispatch finalizer (`dispatch/runtime/finalize`) drains it
once per outermost dispatch via `flushDeferred()`, landing the whole batch through
the normal `diag_synced` flow from a dispatch-side context (one notify per drain
regardless of how many detections coalesced). Dedup stays at the call site, so
the queue can't flood. Opening the diagnostics window (leader `e`) is itself a
dispatch whose finalizer drains first, so a pending warn is visible the moment
the window opens. The effect/boot-path producers (config warnings, `_recordError`)
keep recording synchronously — they're not on a read path. Guarded by
`test-diag-log` (deferred-lane + no-synchronous-write tests) and the updated
`test-plugin-purity` (flush-before-assert). `dispatch/runtime/finalize →
io/diag-log` is a clean dispatch→io down-edge; layer graph stays acyclic.
