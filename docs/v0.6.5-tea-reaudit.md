A fresh skeptical audit run AFTER the four-track pre-release review, on the
explicit instruction *"don't trust memory/docs/comments ‚Äî they may be stale;

(1) dispatch core + root reducer, (2) every Component `update` + leaves,

## TEA bar applied
- **Root reducer** `(model,msg) ‚Üí {model,cmds}`: pure ‚Äî no IO, no `getModel`,
  no input mutation, no `Date.now`/`random`, effects emitted as Cmds.
- **Component.update** `(msg,slice) ‚Üí {next,cmds}`: pure ‚Äî no model arg, no
  `getModel`, no foreign-slice read; root facts arrive via the Msg payload.
- **Leaves**: pure transforms; no `getModel`/IO/upward requires.
- **Render**: `model ‚Üí output`; never dispatches, never mutates model.
- **Single-writer**: exactly one owner per slice.

## Verified CLEAN (re-confirmed in bodies)

  owning Component's `update`; nav chrome routes through

- **Render is a pure `model ‚Üí output` view** ‚Äî `render/` has zero

### Cross-system precedent (B + C) ‚Äî added 2026-06-17

Both live exceptions are well-precedented patterns, not lazytui quirks ‚Äî recorded
here so the "KEEP" verdicts are defensible against how comparable systems work.
First, two senses of "pure" this section keeps separate:
- **I/O purity** ‚Äî does code read the *external world* (terminal size, clock, IO)?
- **Functional-core discipline** ‚Äî is the reducer a pure function of its args, and is
  state written only through `update`? Ambient reads / out-of-band writes break this
  *without* being I/O.

#### C ‚Äî Impure-shell model read (`getModel`) ‚Üí the `getState()` seam
Every TEA/Flux architecture has an impure caller that reads *current state* to feed
the pure core. `getModel()` is lazytui's. It is NOT I/O-impure (a deterministic read
of internal state) ‚Äî it's the mild functional-discipline kind (ambient read vs arg).
- **Redux** ‚Äî `store.getState()`; thunk middleware `(dispatch, getState) => {‚Ä¶}`;
  redux-saga's `select()` effect. The direct analogue.
- **Zustand** ‚Äî `store.getState()` (read outside the reactive path).
- **XState** ‚Äî `actor.getSnapshot()`.
- **Elm / Bubble Tea** ‚Äî the runtime holds the model and threads it into the pure
  `update`/`view`; the read IS the runtime's job (no public getState ‚Äî purity is
  language-/loop-enforced ‚Äî but the read still happens in the runtime).

lazytui contains this in `dispatch/fanout.js` (the runtime) ‚Äî exactly where Redux
(store/middleware), Elm/Bubble Tea (program runtime), and XState (actor) put theirs.
A codebase with *zero* such read would be the anomaly. ‚Üí universal, uncontroversial.

#### B ‚Äî Runtime-written derived viewport geometry (`innerH`) ‚Üí the `viewport.Height` pattern
The natural pattern in component systems is to **store** the layout-derived size on
the component and **update it when the layout/size changes**:
- **Bubble Tea** ‚Äî `bubbles/viewport` has a `Height` field; the parent sets it on
  `tea.WindowSizeMsg` (`m.viewport.Height = msg.Height - chrome`); the viewport's own
  `Update` reads `m.Height` to clamp. Stored on the component, written on resize.
- **React** ‚Äî a scrollable/virtualized list takes `height` as a **prop**; a parent
  measures (`useLayoutEffect` + `getBoundingClientRect`, or AutoSizer/ResizeObserver)
  and passes it down. Materialized as data, handed to the child.
- **Elm** ‚Äî `Browser.Dom.getViewport` ‚Üí `Task` ‚Üí result `Msg` ‚Üí `update` sets it.

So **storing `innerH` on the viewer (B's data shape) is the natural one.** The point
of difference is the *writer*:
- In those systems the **parent owns the child sub-model** (compositional nesting) and
  writes the height directly in its own `update` ‚Äî pure, no message.
- lazytui uses a **FLAT instance registry** (sibling instances keyed by paneId; no
  parent holds the viewer's slice). The natural parent-writes-child move is therefore
  unavailable; the height must be written by either a re-dispatched Msg
  (`viewer_set_viewport`, the former approach) or the runtime finalizer (B). **B is a
  flat-registry adaptation of the natural pattern: natural storage, runtime writer.**

I/O purity is intact in B: the raw terminal-size read (`io/term.dims()` ‚Üí
`term_resized` Msg data) is at the edge, like every runtime (Bubble Tea
SIGWINCH‚Üí`WindowSizeMsg`, Elm resize-event‚ÜíMsg); the pure core never does that read,
and `innerH` itself is pure math. B's only deviation is **single-writer placement** ‚Äî
derived geometry written outside `update`.

Pure alternatives (recorded, not chosen):
- **Write-through-`update`** via a `viewer_set_viewport` Msg on each layout change (the
  faithful Bubble Tea form) ‚Äî pure single-writer, but costs a re-dispatch per layout
  change because the flat registry has no parent to write for free (the resize-as-Msg
  arc removed exactly this).
- **Thread `innerH` via `augmentMsg`** (fold B into C) ‚Äî purest for lazytui's reducer
  rule, but no mainstream system threads geometry per-message; idiosyncratic.
- The *truly* natural form (parent writes child height in `update`) needs
  **compositional nesting** instead of the flat paneId registry ‚Äî a larger
  architecture change lazytui deliberately doesn't have.
