# v0.6.5 — deep TEA / arch / dataflow re-audit (2026-06-16)

A fresh skeptical audit run AFTER the four-track pre-release review, on the
explicit instruction *"don't trust memory/docs/comments — they may be stale;
verify in function bodies; do it harder."* Four parallel agents audited
(1) dispatch core + root reducer, (2) every Component `update` + leaves,
(3) layering + dataflow, (4) single-writer + render purity + file layout.
Every load-bearing claim below was re-verified by hand in the function body
(not the comment) before being recorded here.

## TEA bar applied
- **Root reducer** `(model,msg) → {model,cmds}`: pure — no IO, no `getModel`,
  no input mutation, no `Date.now`/`random`, effects emitted as Cmds.
- **Component.update** `(msg,slice) → {next,cmds}`: pure — no model arg, no
  `getModel`, no foreign-slice read; root facts arrive via the Msg payload.
- **Leaves**: pure transforms; no `getModel`/IO/upward requires.
- **Render**: `model → output`; never dispatches, never mutates model.
- **Single-writer**: exactly one owner per slice.

## Verified CLEAN (re-confirmed in bodies)
- **Single-writer holds for every slice.** `setModel` called from one site
  (`dispatch/dispatch.js:986`); `modes`/`modal.*` written only via the
  reducer's `_withModes`/`_withModal`; instance-slice content only via the
  owning Component's `update`; nav chrome routes through
  `nav-state._navDispatch → dispatchMsg → reducer`. No two-writer field.
- **Render is a pure `model → output` view** — `render/` has zero
  `dispatch`/`setModel`/`setInstanceSlice`. (Asterisk: `paint.js:563,579`
  lazily `ensureSession`/`resizeSession` the PTY — external state, not model.)
- **Leaves pure** — all leaf files: no `getModel`/IO/upward requires.
- **Async ingress via Msgs only** — PTY/command/job output enters the model
  through `dispatchMsg`/`applyMsg`, never by mutating the model.

## FINDINGS (each verified in the body)

### F1 — Component.update is impure in four panels  [HEADLINE]
Contradicts the auto-memory claim "every Component update pure of
getModel/cross-slice (incl. viewer + docker)" and the panels' own comments.
Guarded to be *correct today* (the impure read resolves the focused pane,
which is the one being keyed) → latent, not a live bug; but a real
TEA-discipline breach + multi-instance hazard. Fix pattern is already proven
in `config-status`/`groups`/`stats` (verified clean): thread the needed
facts via the Msg payload (`augmentMsg`/modelBundle), don't reach for globals.

- **(a) viewer** — `viewer.js:1014` (`y` arm) + `:1046` (`$` arm) call
  `select.selectedText()`/`plainLineWidth()`, which via `select.js:47-59`
  read `getModel()` AND resolve another slice (`resolveTarget('viewer')`).
  Contradicts the "viewer reducer is now PURE of getModel()" claim at
  `viewer.js:371`.
- **(b) docker** — `docker.js:509-512` `_getItems()` ignores its `slice`
  param, reads `getModel().config.groups[m.currentGroup]`; called from the
  `i/t/s` key arms. Contradicts "no global reads" at `docker.js:340-344`.
- **(c) files** — key/refresh arms read `getInstanceSlice('layout')` (foreign
  slice) + `getModel()` + `getFilter()` via `_paneById`/`_itemsFor`
  (`files.js:92,122,132,167-190`).
- **(d) history** — `history.js:133-138` fully unmigrated: `getFocus()`,
  `getSel('history')`, `history.all()` global straight in `update`.

STATUS: ☑ (a) ☑ (b) ☑ (c) ☑ (d) — ALL DONE. Pattern: pure variant +
augmentMsg threading (extended to `(msg, model, slice)` for files' per-pane
facts). viewer.update/docker.update/files.update/history.update now reach no
getModel/getInstanceSlice/getFilter/getFocus in the reducer path; their
purity-claiming comments (F5) are now TRUE. Tests that drive update() directly
(docker/files) thread augmentMsg to mirror the shell.

### F2 — Root reducer reads route topology (the real purity caveat)
`runtime.js` (`escape`/`list_select`/`nav_select`) + `layout.js:344,836,851`
call `route.getFocus()`/`resolveTarget`/`resolveViewerPaneId`, which read the
mutable layout-service slice + focus — module-global state OUTSIDE `model`.
This is the documented "blessed chokepoint" (route = topology oracle, emits
Cmds not actions, never mutates model). Defensible, but it means "the reducer
is a pure function of (model,msg)" is ASPIRATIONAL, not literal. Resolution =
honesty (state the caveat accurately) ± thread focus/target via Msg where cheap.

STATUS: ☑ DONE — honesty fix. Added a Contract bullet to `app/runtime.js`
stating the reducer is NOT strictly pure of (model,msg): it reads route
topology (the blessed chokepoint). Did NOT thread focus/target into the root
reducer (large change to a deliberate chokepoint — out of scope).

### F3 — The "irreducible" layering core — RE-VERIFIED, audit was OVERSTATED
Layer order (bottom→top): leaves/model → io → panel → dispatch →
render/overlay/feature → **app (top)**. So `app→dispatch` / `app→panel` are
DOWNWARD (legal). The SCC `{feature,overlay,render,panel,dispatch,app}` is held
ONLY by the UPWARD edges: `dispatch/dispatch.js:39→app/runtime` (reducer),
`dispatch/input.js:51→app/cleanup` (teardown), `panel/api.js:64→dispatch/stream`
(streamCommand re-export), plus dispatch→feature/render reaches.

On hard re-verification, NONE of the audit's proposed fixes is a clean win:

- **keybindings move — DEAD (cosmetic + wrong on direction).** `app/runtime:42
  →dispatch/keybindings` is a DOWNWARD edge (app is top) — NOT a violation.
  Moving keybindings to `panel/` just relabels a downward app→dispatch edge as
  a downward app→panel edge: no SCC change, and it parks a key registry in
  `panel/` where it doesn't belong. (Also it's a mutable registry → not
  `leaves/` per route.js:20-21.) The agent's "free reduction" was an artifact
  of not checking edge direction. DO NOT DO.
- **streamCommand re-export (panel/api:64→dispatch/stream).** Real upward edge,
  but its only consumer is `docker.js` (streams docker logs/inspect in its
  effect handlers). Dropping the re-export → docker imports streamCommand
  direct → just MOVES the panel→dispatch edge to docker. Same edge count, no
  SCC change. Marginal legibility gain only (docker's stream dep becomes
  honest vs hidden behind the api façade).
- **dispatch→feature — entangled.** `feature/jobs.js`→`render/render-queue`
  (render is ABOVE dispatch), so jobs can't move below dispatch.
  `feature/history.js`→`panel/hub` only (could move down), but jobs is stuck.
- **reducer/cleanup relocation — real but a big refactor.** Moving the
  1038-line reducer + cleanup out of `app/` would break dispatch→app, but you'd
  also need to resolve `app/cleanup:17→dispatch/action-runner`. A dedicated
  arch arc, NOT a v0.6.5 close-out.

VERDICT: F3 has no clean cheap win. The SCC is held by genuine control-flow
deps (a Component streaming through dispatch; the entry dispatcher invoking the
reducer; teardown). These are BLESSED (consciously accepted + documented), NOT
"irreducible" — per docs/blessed-exceptions.md the exceptions are explicitly
"not load-bearing" and on the drive-toward-empty list. They're REDUCIBLE AT A
COST (relocate the 1038-line reducer; DI-invert the stream edge, which just
relocates it) that currently buys less legibility than it spends. So they stay
FOR NOW, documented — not because they can't be removed. (Avoid the word
"irreducible": the audit's "all reducible placement choices" overstated the
ease; calling them "irreducible" overstates the permanence. Both wrong.)

STATUS: ☑ RESOLVED as DOCUMENT-ONLY — no code moves (verified not worth it).
The big reducer/cleanup relocation is recorded for a future dedicated arch arc
only. (Re-verification 2026-06-16 with corrected edge-direction analysis.)

### F4 — `dep-walker.js` brace-depth classification bug
It updates brace depth AFTER classifying a line (`dep-walker.js:60` then
`:64-67`), so a one-line `function f(){ require(...) }` counts as top-level —
inflating reported top-level edge counts (SCC conclusion still holds). Fix:
account for same-line braces before the top-level/deferred decision.

STATUS: ☑ DONE — position-aware classification (length-preserving string
blank + per-require brace depth). panel→dispatch top 2→1 (correct).
