## What the audit CONFIRMED clean (do not re-litigate)

Verified by grepping inside the actual function bodies, not by trusting
purity-claiming comments:

- Root reducer `runtime.update(model, msg)` — no effectful imports;
  every branch returns a new model or identity-preserves; all effects
  leave as Cmd descriptors; modal-close arms guard on their mode flag.
- **Every Component `update(msg, slice)` body is pure of `getModel()`
  and cross-slice reads** — including the ~700-line viewer reducer
  (`viewer.js` `update` + `_updateInner`, lines 368–1062: zero
  `getModel()` *calls*, only comments) and `docker.update`/`_handleKey`.
  The `getModel()` calls in those files live only in `installEffects` /
  `render` / `getItems` (legitimate effect + view layers).
- Leaves are pure: no `getModel()` calls, no `fs` / `Date.now` /
  `Math.random` / `process` / `console`.
- Render never dispatches: zero `applyMsg` / `dispatchMsg` / `setModel`
  / `.update(` under `js/render/`.
- Single-writer holds: `state.js` helpers are thin Msg routers; boot
  writes (`set_config` / `set_register`) route through the reducer.

The TEA semantics are trustworthy. v0.6.5 changes module *placement* and
*import direction*, not runtime behavior — the test suite must stay green
across the whole arc. Final baseline **1/95** — the only failure is the
`xz`-not-installed `test-archive` env failure (unrelated). (Mid-arc the count
read 2/94: §3's `halfProjection` parametrization briefly regressed
`test-viewer-pane-bounds`; caught and fixed in `206afd6` — see §3.)

## The findings (the v0.6.5 work)

### 1. The `{app, dispatch, panel}` import cycle — CLOSED 2026-06-16 (Phases 1–3)

**Phase 1 (store extraction) landed.** A require-walker (`js/scripts/dep-walker.js`
— kept as re-measurement tooling) drew the layer graph and proved the cycle's
root: `getModel` is the most-depended-on symbol in the codebase, yet the model
*store* sat at the *top* layer (`app/runtime`). Of 33 top-level importers of
`app/runtime`, **32 took only `getModel`**; exactly one (`dispatch/dispatch.js`)
used the reducer `update`.

The fix: extract the store (`init` + `_modelRef` + `getModel`/`setModel`, plus
the root-model shape) to a new bottom layer **`js/model/store.js`** — it depends
only on `leaves/modes` (a pure leaf), nothing upward. The reducer `update`
*stays* in `app/runtime.js` (it reads `panel/route`, so it can't move down).
All production importers repoint *down* to `model/store`; `app/runtime`
re-exports the three accessors for back-compat so the ~55 test/bench files stay
green untouched (tests aren't in the layering DAG). The deferred `getModel()`
calls that were lazy *only* to dodge the cycle collapsed to top-level refs; the
handful that remain (`effects.installBuiltins`, `tui.js`) are legitimately
function-scoped under the existing lazy-require pattern and now point cleanly
*down* at the store. No production code reaches *up* into `app/runtime` for the
store anymore — only the reducer (`update`) and `_ghostSuffix` stay there.

Measured win (same fixed walker, before → after):

| top-level edge | before | after |
|---|---|---|
| `panel → app` | 16 | **3** |
| `dispatch → app` | 11 | **7** |
| `{app,dispatch,panel}` cut-target edges | 43 | **26** (−40%) |

Perf: parity (relocation is free — top-level `getModel` is a direct ref
regardless of module home; collapsing deferred sites is a small win, −0.24µs/call).
Suite parity at Phase 1 (2/94 — `test-archive` xz, plus `test-viewer-pane-bounds`,
which a later §3 step regressed and `206afd6` fixed; final baseline 1/95).
The layer-level top SCC is not fully dissolved
(as anticipated — "bank progress"): the residual is now driven by **`app/state.js`**
(the `getSel`/`getScroll`/`allPanels` readers, ~45 importers) — the **Phase 2**
target. `app/exec.js` + `app/cleanup.js` are the smaller remaining upward reaches.

#### Phase 2 — `app/state.js` split — DONE 2026-06-16

The residual attractor (`app/state.js`'s `getSel`/`getScroll`/`allPanels`
readers + writers, the second-most-imported surface after the store) was
split by layer. These are panel-domain operations (they resolve
panel→instance via `panel/route` and write via `panel/api.dispatchMsg`), so
they moved **down** into a new **`panel/nav-state.js`**:

- 18 accessors relocated: the readers (`getSel`/`getScroll`/`isMultiSel`/
  `multiSelCount`/`allPanels`), the writers (`setSel`/`setScroll`/multisel/
  group-tab/viewer-content), and the `selectedOrFocused` / `infoLinesFromFocus`
  composites (pulled out of `panel/api.js` — they read the nav chrome).
- **One-way dep:** readers are `api`-free (route + leaves only); writers/
  composites need `api.dispatchMsg`, and `api` needs `nav-state`'s
  `syncPanelScroll` (its per-dispatch finalizer) — an irreducible *intra-panel*
  mutual dependency, broken by lazy-requiring `api` inside `nav-state` (cached).
  Top-level graph is acyclic: `api → nav-state`.
- `app/state.js` slimmed 555→335 lines to the boot layer (`loadConfig`/
  `initState`/`_*Slice`) + the two test-facing group helpers
  (`selectGroup`/`resetGroupContext`, which call *down* into dispatch — no
  upward edge); it re-exports the accessors for back-compat so the suite is
  untouched. The one prod `selectGroup` caller (`input.js`) was inlined to
  `navSelect('groups', …)`.
- All cross-layer importers (`dispatch/{dispatch,actions,input,cmdline,
  action-runner}`, `panel/{commands,navigator/history,navigator/config-status,
  monitor/stats}`, `render/{paint,footer}`, `overlay/{help,copy}`) repointed
  to `nav-state` (downward).
- **Crutch dropped:** `allPanels`'s pre-boot auto-register (`installBuiltins`+
  `registerComponent`) was removed — a dead branch post-boot. Perf-tested
  neutral; in fact `allPanels` went **142µs → 0.018µs/op** (top-level
  `route`/`mpool` imports replaced the old per-call `require()`s). The green
  suite confirms no test relied on the crutch.

Result: **`app/state.js` has zero inbound cross-layer edges**; into-`app`
top-level cross-edges dropped **10 → 3** (only `dispatch→runtime` [the
reducer], `input→cleanup`, `api→exec` remain — the Phase-3/accept tail).
Suite 2/94 (`test-archive` xz + the then-live §3 regression, fixed later in `206afd6`); benches at parity.

#### Phase 3 — re-home io sinks → `io/`; §1 CLOSED 2026-06-16

The boundary walk-through (see "panel↔dispatch" analysis) split the residual
upward edges into two classes:

- **Class A — three misfiled bottom-layer I/O modules.** `app/exec.js`
  (`execAsync` / `child_process.spawn`) and the `fs`-backed `event-log` /
  `diag-log` recorders sat *above* the panel layer in `app/` and `dispatch/`,
  so every panel/render/overlay/state use read as an upward cross-layer edge.
  All three have **zero upward deps** — they're side-effecting bottom-layer
  sinks. (NB: they do `fs`/`process` I/O, so they belong in **`io/`**, NOT
  `leaves/` — the purity wall forbids `fs` in leaves.) Relocated all three to
  `io/`, which keeps its zero-upward-import property. 27 importers repointed
  *down* to `../io/` (no re-export shims — the 4 test sites repoint too, since
  this pass exists to make top-of-file imports honest). `e35c04a`.
- **Class B — the `api ↔ {stream, effects}` knot.** A *genuine* mutual
  dependency, left in place by design: the effect/stream runners must call
  *down* into `panel/{api,route}` to route async output into the right pane,
  and `panel/api` must fire effects / start streams. Breaking it would need DI
  (the `setMergedActionsProvider` pattern) — which only relocates the hidden
  dep to a boot-time setter, trading one legibility for another. Documented as
  the blessed mutual core, not cut.

Edge deltas (dep-walker, before → after Phase 3): `panel→app` top **1 → 0**;
`panel→dispatch` deferred **11 → 5** (the 6 Class-A observability edges gone);
`app→dispatch` deferred **16 → 14**. Suite 2/94 (`test-archive` xz + the then-live §3 regression, fixed later in `206afd6`); benches at
parity (relocation is free).

**§1 closes here.** The layer SCC `{feature, overlay, render, panel, dispatch,
app}` does NOT fully dissolve — as anticipated from the start ("measure and
bank progress"), it's held open by `overlay/render/feature`'s independent
reaches plus the legitimate *downward* `dispatch→panel` direction and the
Class-B knot. The two remaining into-`app` edges (`dispatch→runtime` reducer,
`input→cleanup` teardown) are the **blessed irreducible core** — the
entry-point-invokes-app-orchestrator direction, the analogue of `dispatch`
calling the reducer. We do NOT hide them behind lazy requires. The banked win
across Phases 1–3: the two biggest attractors (the model store, the
`app/state` readers) are now true bottom layers, the misfiled I/O sinks are
home, and the deferred-require ratio that obscured real deps is materially
down — top-of-file imports are honest where it matters most.

#### Perf finding + fix: route memo — 2026-06-16

Investigating "why is v0.6.3 faster than v0.6.5 on `bench-hotpaths`"
(`viewer_append` / `select_extend` / `search_clear` were 2–4× slower)
surfaced a real regression — **not** from the store extraction (that's
parity), but from v0.6.4.

**Cause.** The v0.6.4 resize-as-Msg arc added `panel/api._finalizeDispatch()`
after *every* top-level `dispatchMsg` (so `render()` could become a pure
view). v0.6.3 had no such finalizer — that scroll-clamp + viewer-`innerH`
work ran in `render()`, ~once per 50ms frame. The finalizer calls
`route.resolveTarget('viewer')` + `resolveViewerPaneId()` per Msg, and
both walked the arrange tiers **allocating short-lived arrays/closures**
(`mpool.lastColumnPanels` + the `findPaneLocation` predicate) → GC churn
~70–136µs/op (233 GCs / 300k calls, 0 retained). On a streamed line that
was ~66% of per-Msg cost (174µs of 264µs). (Aside: the v0.6.4 buffer
change *T2d* had already made the append *reducer* faster — with the
finalizer disabled, current beats v0.6.3 — so the regression was purely
finalizer overhead.)

**Fix.** Memoize `resolveTarget` / `resolveViewerPaneId` on
`(intent, focused, lastViewerTab, arrange-ref, _instVer)` — every input
they read. `focused`/`lastViewerTab`/`arrange` are read fresh + cheap each
call; `_instVer` is a monotonic counter bumped only in
`setInstance`/`setService`/`disposeInstance`/`_reset` (the instance-set
the tier-4 scan depends on). Mutating an existing slice
(`setInstanceSlice` — the per-Msg viewer append) does **not** bump it, so
streamed appends hit. Over-bumping only forces a recompute, never a wrong
result.

**Hit/miss semantics** (what lands in each bucket — `demo-route-memo.js`
classifies these empirically):

| action | result | why |
|---|---|---|
| streamed `viewer_append`, `select_extend` | **HIT** | none of the 5 key fields change |
| move cursor in focused pane (`j`/`k`) | **HIT** | `focused` is *which pane*, not the row; `sel` isn't in the key |
| **change focus** (Tab / click) | **MISS** (one, then re-hits) | `focused` differs |
| layout edit (resize-drag / add-column / pool / free-config) | **MISS** | `arrange` ref differs |
| open file tab / close pane / reconfigure | **MISS** | `_instVer` bumps |
| plain terminal-window resize (dims only) | **HIT** | `dims` isn't in the key — `resolveTarget` picks *which* pane, not its size |

A focus change is **one** miss, then everything re-hits while focus stays
put — so Tab-spam at 10×/sec is ~10 misses/sec ≈ 0.7ms, nothing. The
high-frequency paths (append/select/scroll, thousands/sec) are exactly
the ones that hit.

**Result** (`bench-hotpaths`, ops/sec, current pre-memo → post-memo):
`append@10k` 3,983 → 20,510 (5×); `append@50k` 2,427 → 5,490;
`select_extend` 4,656 → 1.62M (~350×); `search_clear` 4,688 → 1.69M
(~360×) — all now well past the v0.6.3 baseline. `bench-tea-overhead`
unchanged; suite parity.

**Worst-case validation** (`bench-route-memo.js` — forces 100% miss by
flipping `ctx.focusedTabId` every call, vs the pre-memo parent):

| 100k ops/case | pre-memo | memo |
|---|---|---|
| `resolveTarget` HIT (steady) | 68.6µs/op | **0.022µs/op** |
| `resolveTarget` MISS (churn) | 70.1µs/op | 70.6µs/op |
| `resolveViewerPaneId` MISS | 142µs/op | 139µs/op |

The pathological all-miss case is within ~1% of no-memo — the per-miss
key-build + memo-object alloc is negligible against the recompute it rides
on. So the memo is strictly dominant: ~3000× on hits, free on misses.

**Considered, not done:** gating the finalizer to skip content-only Msgs.
After the memo the finalizer's residual cost is ~0.06µs/op (measured), so
gating would save almost nothing while adding a Msg-classification surface
with a real correctness-drift risk (an allowlist that misses a new
layout-affecting Msg → stale clamps). Left ungated; revisit only if the
per-pane clamp loop itself grows (then memoize it, don't gate by Msg type).

Benches/demo: `js/test/bench-route-memo.js` (HIT vs forced-MISS cost),
`js/test/demo-route-memo.js` (action → hit/miss classifier).

#### Original analysis (kept for context)


The directory names imply a layering (`app` > `dispatch` > `panel` >
`leaves`/`io`/`parser`), and the *runtime* respects it. But the
top-level (non-deferred) `require()` graph proves all three are
mutually circular:

- `app/runtime → panel/route`; `app/cleanup → dispatch/action-runner`;
  `app/runtime → dispatch/keybindings`
- `panel/api → app/runtime + app/state + dispatch/stream`
- `dispatch/dispatch → app/state + app/runtime`;
  `dispatch/actions → app/state + app/runtime + panel/viewer/tabs`;
  `dispatch/input → app/state + app/cleanup + panel/api`

`{app, dispatch, panel}` is therefore one strongly-connected component.
The symptom + cost: **~62% of `dispatch/`'s, ~70% of `app/`'s, and ~50%
of `panel/`'s `require()` calls are deferred inside function bodies** to
dodge load-order failures. It works (Node tolerates the cycle; the lazy
edges are well-commented), but the file's import block no longer tells
you what the file depends on — the #1 reasoning hazard for an AI editing
these modules.

**Work (planning required before execution — draw the graph first):**
- Produce the actual module dependency graph with cycle edges
  highlighted (tooling: `madge` or a small custom require-walker).
- Identify the minimal set of edges whose removal breaks the cycle.
  Likely candidates: extract the small shared registries/readers that
  `app ↔ dispatch ↔ panel` all reach for into a true bottom layer
  (`leaves/` or a new `shared/`), so the three dirs depend *down* on it
  instead of *sideways* on each other.
- Target: drive the deferred-require ratio down materially and make the
  top-level import block honest. Full acyclicity may not be reachable in
  one pass — measure and bank progress.

### 2. `io/terminal.js` contradicts its own header — DONE 2026-06-16

It claimed "stays a true leaf" but still imported *up* into `app/runtime`
(`getModel().projectDir`), `render/render-queue` (`scheduleOverlay`), and
`feature/jobs` (`register`/`close`). All three are now injected at boot,
mirroring the existing `setExitHandler` pattern:
- spawn cwd → an `ensureSession(id, cmd, cols, rows, cwd)` argument
  (`render/paint.js` passes `model.projectDir`; `restartSession` reuses
  the session's stored cwd);
- `setRenderHook(fn)` → the post-output repaint (wired to
  `scheduleOverlay`);
- `setJobsHooks({register, close})` → the jobs-registry adapter.

`panel/viewer/pty-lifecycle.install()` wires all three at boot (alongside
the exit handler); each unset hook = the effect is skipped, so the module
still runs standalone in tests/scripts. Result: `io/terminal.js`'s only
requires are `node-pty` + `@xterm/headless`, and `io/` has **zero** upward
imports — the header is now true.

### 3. Two "leaves" reach up into `panel` — DONE 2026-06-16

Both lazy `require('../panel/...')` reaches are gone; `leaves/` now imports
only sibling leaves (and node stdlib). Resolved per-file by its blast radius:

- **`leaves/geometry.js → panel/route.resolveViewerPaneId` → parametrized.**
  `halfProjection` had only 3 production callers (`panel/layout`,
  `dispatch/dispatch`, `render/paint`), each in a layer that can call
  `route.resolveViewerPaneId()`. New signature `halfProjection(layoutSlice,
  viewerPaneId)`; callers thread the resolved id in. The leaf is now fully
  pure (no route reach). **Follow-up (`206afd6`):** the initial change threaded
  the 3 direct callers but missed two INTERNAL ones (`getPanelViewportH`,
  `_halfBoundsMap`) that still passed one arg — in half view the right (viewer)
  slot defaulted null, breaking hit-test/overlay-anchor/scroll-clamp and
  crashing `test-viewer-pane-bounds`. Fix threads `viewerPaneId` through the
  full `visibleBoundsFor → _visibleBoundsMap → _halfBoundsMap → halfProjection`
  + `getPanelViewportH` chain (callers pass `route.resolveViewerPaneId()`, no
  leaf fallback — §3 retired it). Non-viewer-only callers left unthreaded by
  design (`render/decor`, the api finalizer per-pane loop skip detail).
- **`leaves/pane-tabs.js → panel/api.getMergedActions` → injected.**
  `getMergedActions` is needed by `flatTabInfo` (~20 callers) so threading
  it through every signature wasn't contained, and it's a panel-registry
  computation the leaf can't reproduce. Instead a provider is injected:
  `setMergedActionsProvider(fn)`, wired once at the bottom of `panel/api.js`
  on load (`require('../leaves/pane-tabs').setMergedActionsProvider(
  getMergedActions)`). That single point covers every consumer — prod and
  tests both transitively require `panel/api`, and `pane-tabs` no longer
  requires back, so the edge is a clean downward `panel → leaf` one. Unset
  → `{}` (graceful). Bonus: it removed the former per-call `require`
  resolution the v0.6.4 R1 comment profiled at ~70µs (bench: modelBundle
  now ~0.58µs/op).

### 4. `dispatch/modes.js` is mis-homed — DONE 2026-06-16

It was a registry (the MODES table + `isModal` / `isOverlayActive` /
`CHAIN_MODES` / `suppressesChromeClicks` / `resetModes` derivations)
imported by `render/paint`, `render/footer`, `dispatch`, `input`,
`overlay/pane-menu`, and `app/runtime` — so `render → dispatch` read as a
layer violation even though it had no dispatch behavior. Moved to
`leaves/modes.js`. It was NOT actually dependency-free: the predicates
defaulted their modes-bag arg via a lazy `_modes()` → `getModel()`, an
upward reach into `app/runtime`. The move dropped that default so the
predicates now take their bag explicitly (every caller already had one;
only `render/footer.js#renderFooter` relied on the default → now passes
`model.modes`; the test/smoke `resetModes()` no-arg calls now pass
`getModel().modes`). Result: `leaves/modes.js` is a genuinely pure leaf
(zero requires), and the modes-driven `render → dispatch` edge is gone.

> Residual (separate, out of scope here): `render/paint.js:281` still
> lazy-requires `dispatch/event-log` to record a render-time error — a
> different, smaller `render → dispatch` edge, not modes-related.
