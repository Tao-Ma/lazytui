## Goal

Lift the layout from "left column + right column" to an arbitrary
ordered list of **columns**. In free-config mode, dragging a pane
beyond the layout's edges OR into the gap between two columns spawns
a new column at that position. Detail's "must live in last column"
invariant becomes the only column-positional rule that survives.

## Non-goals

- **Not row-splits within a column.** A column is still a vertical
  stack of panes. No horizontal sub-splits, no tmux-style trees.
- **No per-column collapse/expand.** Existing per-PANE collapse stays;
  whole-column collapse is a future story.
- **No detail-anywhere relaxation.** Detail still anchors to the
  bottom of the rightmost (last) column. Same v0.6.1 single-detail
  invariant; that retires with v0.7 multi-instance.

## Today's architecture

`arrange = { leftPanels, rightPanels, leftWidth, detailHeightPct, pool }`.
Hardcoded two-column. `leftWidth` is the column-1 width in cells;
column 2 takes the remainder. All ~30 source files that read the
layout know "left" and "right" by name (`pool_show({column: 'left'})`,
`mpool.findDetailPane` walks `rightPanels`, `pointToDropTarget`
checks `inLeft` then `inRight`).

## Target architecture

### Data model

```
arrange = {
  columns: [
    { width: 32, panels: [pane, ...] },   // leftmost
    { width: 30, panels: [pane, ...] },
    { panels: [pane, ...] },              // rightmost; width = remainder
  ],
  detailHeightPct: 60,
  pool: { ... },
}
```

- `columns[]` тАФ ordered left тЖТ right. Empty array invalid (parser refuses).
- `columns[i].width` тАФ fixed cells. The last column has no `width` тАФ
  it consumes whatever's left after the others. (Same convention
  today's `rightPanels` follows: implicit width = COLS - leftWidth.)
- `columns[i].panels[]` тАФ the wide-pane intermediate form from v0.6.1
  (`{id, type, title, hotkey, paneId, tabs, activeTabId, тАж}`). Single
  panel kind per pane carries forward.
- Detail pane: `columns[columns.length - 1].panels[last]`. The
  parser-time invariants restated:
  - exactly one detail tab across the layout
  - detail must be in last column
  - detail must be last pane of that column
  - detail must be sole tab in its pane (multi-tab + detail deferred)
  - actions in last column (today: detail's column = right column;
    actions also right-only)

### YAML schema (v0.6.2)

```yaml
layout:
  columns:
    - { width: 32, panels: [containers, groups, files] }
    - { panels: [actions, stats, { tabs: [detail], height: 60% }] }
```

- `columns:` тАФ list of column mappings. Required.
- Each column: `{ width?: int, panels: [pool-id | {tabs:[...], height?, heightPct?, collapsed?, activeTab?}] }`.
- Last column's `width:` is ignored (warned at parse?) тАФ implicit.
- v0.6.1 `left:` / `right:` block form is rejected with a migration
  pointer (same loud break v0.6.1 did for inline `{type:...}` cells).

### Slice + Msg surface

`layout` slice's `arrange` field follows the new shape. Msgs that
historically took `column: 'left' | 'right'` switch to
`columnIndex: <int>`:

- `pool_show({ columnIndex, index?, id })` тАФ was `column`
- `pool_drag_release` target тАФ `{kind:'insert', columnIndex, index, тАж}`
- `set_arrange({ arrange, dirty })` тАФ unchanged signature; new arrange shape inside

New Msg for the v0.6.2 feature:

- `drop_new_column({ columnIndex, source })` тАФ handled inside
  `pool_show` / `applyDrop` when target.kind === 'new_column'.

### Drag UX (Phase 2)

`pointToDropTarget` and `pointToPoolDropTarget` gain two new
drop-zone classes BEFORE the existing column-content match:

- Left edge: `mx < EDGE_W` тЖТ `{kind:'new_column', position: 0}`
- Column gap: `mx` within `EDGE_W` of any column-boundary `x` тЖТ
  `{kind:'new_column', position: i+1}` (between columns i and i+1)

`EDGE_W` defaults to 2 cells. The existing 3-zone-per-cell (top/mid/
bot) match runs only when neither of the two edge/gap zones hit.
There is NO right-edge zone тАФ position == N always refuses (would
push detail off the last column), so capturing the rightmost cells
as a spawn zone would just dead-shadow the last column's in-column
hit. Cursor near the right edge falls through to the 3-zone hit.

### Live preview (Phase 2)

The renderer paints `slice.freeConfig.drag.previewArrange` when set
тАФ today this previews insert / swap targets within existing columns.
For a `new_column` target, `computeDragPreviewArrange` builds the
arrange with the dragged pane already in a new column:

```js
{
  columns: [
    ...arrange.columns.slice(0, position),
    { width: <stolen-from-neighbors>, panels: [draggedPane] },
    ...arrange.columns.slice(position),
  ]
}
```

Width allocation: each explicit-width neighbor donates
`max(8, floor(neighborW / 3))` cells, shrinking to a floor of 10
cells. The new column collects the sum of those donations (default
24 cells if no explicit-width neighbor exists тАФ i.e. only the
implicit-width last column is adjacent). The `/3` heuristic keeps
the source column livable (loses a third, retains two-thirds)
instead of the symmetric `/2` halving which left both columns
cramped on common terminal widths. The renderer paints columns[]
uniformly тАФ the new column "just appears" at its intended position
with the dragged pane in it.

### Detail invariant under new_column drops (Phase 2 shipped rules)

Phase 2 shipped two refusals to keep the detail-in-last-column
invariant intact:

1. **Detail / actions sources refuse ALL new_column spawns.** The
   reserved-pane group stays in the last column; spawning a new
   column containing one of them would either split the group across
   columns or push detail off "last". Future arc may relax via
   dedicated UX.
2. **Position == N (right edge) refuses for ANY source.** Inserting
   at position N would make the new column the new last column,
   demoting the detail-bearing previous last column off "last" and
   breaking the invariant. `:add-column` mirrors the refusal at the
   cmdline. The drag hit-test does NOT surface a right-edge zone тАФ
   it always refused, so capturing the terminal's last 2 cells as a
   "spawn here" zone just dead-shadowed the rightmost cells of the
   actual last column. Cursor near the right edge falls through to
   the in-column 3-zone hit on the last column's cells instead.

## Phased plan

### Phase 1 тАФ `columns[]` refactor (atomic commit)

Replace `leftPanels` / `rightPanels` / `leftWidth` with
`arrange.columns: [{width, panels}, ...]` everywhere. Suite red until
the last edit; expect ~30 files. Files in dependency order:

1. `parser/schema.js`, `parser/index.js` тАФ emit new shape
2. `leaves/arrange.js` тАФ `rebuildLayoutFromConfig` builds columns[]
3. `leaves/pool.js` тАФ `placedIds` / `findDetailPane` / `hasDetailPane`
   / `panelListItems` walk columns[]
4. `feature/yaml-layout.js` тАФ serializer emits new YAML shape
5. `panel/layout.js` тАФ slice init + Msg arms (`pool_show`, `pool_hide`,
   `set_arrange`, all free-config arms)
6. `leaves/free-config.js` тАФ drag math (`pointToDropTarget`,
   `applyInsert`, `applySwap`, `moveColumn`, `columnTotalH`, тАж)
7. `leaves/free-config-pool-drag.js` тАФ pool drag math
8. `render/layout.js` тАФ column distribution + paint
9. `render/panel-widgets.js` тАФ collapse/close glyph hit-tests
10. `panel/commands.js` тАФ `:hide`/`:show`/`:save-layout`
11. `app/state.js` тАФ `allPanels()`
12. `dispatch/input.js` тАФ mouse routing
13. YAML fixtures: `test/test.yml`, `demo/postgres/tui.yml`,
    `demo/cloudberrydb/tui.yml`
14. Tests тАФ anywhere that constructs an arrange fixture or asserts
    against `leftPanels`/`rightPanels`/`leftWidth`

### Phase 2 тАФ drag-edge spawn + live preview

- Add `new_column` drop-target kind
- Add edge/gap zones to `pointToDropTarget` + `pointToPoolDropTarget`
- `applyInsert` / `applyDrop` reducer arms handle `new_column`
- `computeDragPreviewArrange` extends to handle `new_column`
- Layout reducer creates the column entry with stolen width

### Phase 3 тАФ invariants + UX

- `:add-column [position]` / `:remove-column <n>` cmdline verbs
- Status-bar messages for new_column drops
- Refuse `new_column` drops with detail source (Phase 2's simpler
  rule kept here)
- Defensive: removing a column with `detail` pane refuses

## Migration

v0.6.1 layouts do not parse under v0.6.2:

```yaml
# v0.6.1 тАФ rejected with SchemaError pointing at this doc
layout:
  left:  { width: 30, panels: [groups] }
  right: { panels: [detail] }

# v0.6.2
layout:
  columns:
    - { width: 30, panels: [groups] }
    - { panels: [detail] }
```

Mechanical hand-conversion. The migration guide is
`docs/v0.6.2-migrate.md` (written in Phase 3 finalize).
