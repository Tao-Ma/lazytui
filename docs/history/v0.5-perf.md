# v0.5 — perf measurements (Phase 6 close-out)

The pure-TEA arc made the reducer + every Component's `update` return
new state objects rather than mutate in place. The arc rule was "no
in-place exceptions for hot paths" — measure first, mitigate only if
the measurements justify it.

Two paths were flagged ahead of time for measurement:

1. **`viewer_append`** — streamed action output (`docker logs -f` etc.)
   pushes one line per Msg. Each Msg spreads `[...slice.lines, line]`,
   so the cost per append scales linearly with the existing buffer.
2. **`select_extend`** — fires on every mouse-motion event during a
   detail-panel drag selection (~60-100 Hz from the terminal).

## Benchmark

`js/test/bench-hotpaths.js` exercises both paths through the full
dispatch graph (`dispatchMsg` → `Component.update` → `setSlice`),
matching what a real keystroke / mouse / stream event hits at runtime.

Numbers below from a single warm-V8 run on the dev box (Linux, Node
22). Variance is small (±5%) across reruns.

| Path | Buffer / state | Throughput |
|------|----------------|-----------:|
| `viewer_append` | empty buffer | **187,000 ops/sec** |
| `viewer_append` | 10k pre-existing lines | **21,000 ops/sec** |
| `viewer_append` | 50k pre-existing lines | **5,700 ops/sec** |
| `select_extend` | active selection | **3,200,000 ops/sec** |

## Interpretation

**`viewer_append`.** `docker logs -f` on a chatty service sustains
~500-1,000 lines/sec; bursts climb to ~5,000 lines/sec. Even at a
50k-line buffer (10 minutes of streaming at 100 lines/sec) we're
still ~5x over the sustained-rate target. The spread cost grows
linearly with buffer length, so a *very* long-running session
(100k+ lines, ~17 minutes at 100 lines/sec) would eventually push
into the same order of magnitude as the producer rate. At that
point GC pressure from per-line array allocation becomes the
concern, not throughput — but production users hitting 100k+ lines
in a single session is the long tail, not the workload v0.5.0 has
to optimize for.

**`select_extend`.** ~3.2M ops/sec against a target of 60-100 ops/sec
is effectively free. The slice spread + shallow copy of `select`
costs nothing measurable next to the terminal's own event rate.

## Decision for v0.5.0

**Ship as-is.** Both hot paths have comfortable margin. The pure-TEA
shape is correct for the architecture and the measured cost doesn't
justify the complexity of a mitigation (persistent vector for lines,
or a coalesced/batched append Msg).

## When to revisit

If field reports come in for either:

- Long-running streamed-output sessions (100k+ lines) showing
  user-visible lag or GC pauses during scroll / render.
- Mouse-drag selection feeling laggy on terminals reporting >200Hz
  motion (uncommon but real on some xterm builds).

The realistic mitigations are:

- **`viewer_append`**: either cap `slice.lines` length with a
  ring-buffer trim, or coalesce bursts via a batched-append Msg
  (collect lines in a small array effect-side, flush every N ms).
- **`select_extend`**: not expected to need anything — the math
  shows there's no budget pressure even at the most aggressive
  terminal rates.
