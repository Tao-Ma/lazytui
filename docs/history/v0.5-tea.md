## What v0.5 set out to fix

Pre-v0.5 lazytui had a single global mutable record `S` that:

- doubled as the data home for app state AND as a facade that any module
  could write through,
- was read by ~180 sites and written from ~30 of them — including
  modules that had no business owning that state (`tabs.js`, `actions.js`,
  `layout.js`),
- made it impossible to point at *the* model without walking the codebase.

The goal of v0.5 was a real **Elm Architecture** data model: one root
model the app is a function of (`init() → model`, `update(model, msg) →
[model, cmds]`, `view(model) → string`), owned by the runtime and
threaded down to the renderers; no module reaches for a global to
mutate it.

Mid-effort, an architecture review found that *deleting `S`* wasn't the
real lever — the flat panel API was. Every panel registered as a peer
`panelType` even though panels played heterogeneous roles (cursor
panels driving a viewer; one viewer hosting tabs; pure monitors).
The cross-cutting mess (`setDetail` called from ~13 places, multiple
writers of `model.focus`, etc.) was a symptom of that flat contract,
not of `S` itself. So the direction shifted from "delete `S`" to "give
panels a taxonomy and single-writer per layer falls out." That phase is
[v0.5-layering.md](v0.5-layering.md).

`S` itself was retired at the end of the arc once nothing wrote it
anymore — a chunk-by-chunk migration that ended with the shim and the
export deleted (commit `4cc0bcf`).

## How the migration ran

Four overlapping waves on branch `v0.5-tea` (see `git log
origin/main..v0.5-tea` for the full chain):

1. **Storage relocation.** `viewMode`, then the mode-flags, the chrome
   cluster (`focus`/`ui.*`), the detail cluster, the group tree, framework
   state, and the last plugin caches — each moved off `S` into the root
   model behind an `S` accessor shim that delegated to `getModel()`.
   Tests + production stayed green throughout.
2. **The update spine.** `update(model, msg) → [model, cmds]` introduced;
   detail-scroll, Cmd-only verbs, the list-selection cascade,
   `run_selected`, `list-select`, and groups-tab / quit routed through it.
3. **Threading + de-shimming.** The owned model threaded through render
   / layout / dispatch / mouse / detail / actions / tabs, then the
   internal `S` readers de-shimmed module-by-module.
4. **Modals + stateful panels onto the reducer.** Every modal sub-model
   folded into `update` (prefix, detail-search, filter, menu, confirm,
   args-prompt, copy, register-popup, design, visual-select,
   yank-register); the genuinely-stateful panels migrated to the
   Component API (config-status, files, docker), then the rest of the
   in-tree panels followed (Phase A → B → C in
   [v0.5-layering.md](v0.5-layering.md)).

The S-drop ran last, as five chunks (A–E) once nothing in production or
tests still touched the shim.

## Architecture inventory (the two homes for state)

After the migration, in-tree the *panel* surface is uniform — every
in-tree panel is a Component. The "two homes" axis still applies to
**state**, and naming it helps prevent future drift:

> **Note: the table below describes pre-layout-Component / pre-Phase-4
> homes.** The layout-Component arc moved `viewMode`, `focus`,
> `layoutDirty`, `layout` (arrange), `panelHeights`, `panelBounds`,
> `modal.design`, and `designEnabled` onto the **layout Component's
> slice**; Phase 4a/c moved `ui.{sel, scroll, filters, multiSel}` onto
> each Navigator's `slice.nav[panelType]` and retired `model.ui`
> entirely; the pure-TEA arc made the reducer + every Component's
> update return-new (no in-place writes). For the current shape see
> [v0.5-layout-component.md](v0.5-layout-component.md) §"End-state
> shape" + `docs/PRINCIPLES.md §12`. The two-homes axis below remains
> the right framing.

- **Centralized** — the root model. One object, owned by the runtime,
  written only by the root reducer (`runtime.update`):

| Group | Fields |
|-------|--------|
| view | `viewMode` (now on layout slice) |
| modes | `modes{}` — 13 modal flags (confirm, prompt, design, designTitleEdit, menuOpen, filter, copy, detailSearch, registerPopup, prefix, cmd, terminal, listSelect) |
| chrome | `focus` (now on layout slice), `currentGroup`, `ui{}` (retired — moved to `slice.nav[panel]`) |
| modal sub-models | `modal{ filter, menu, confirm, prompt, copy, registerPopup, cmdline }` (design moved to layout slice) |
| config | `config`, `projectDir`, `configPath` (layout/panelHeights/panelBounds moved to layout slice) |
| framework | `lastRunAction`, `focused`, `prefixNode`, `prefixSeq` (layoutDirty moved to layout slice) |
| subsystem caches | `register` (yank) (designEnabled moved to layout slice) |

- **Decentralized** — Component slices, written only by the Component's
  own `update`. The slice is the Component's encapsulation boundary;
  cross-layer reads go through `getModel()`, cross-layer writes through
  effects (`apply_msg` / `dispatch_msg`).

| Component | Slice shape |
|-----------|-------------|
| `docker` | `{ status, stats, inFlight, started, eventsStarted }` |
| `config-status` | `{ tab, cache, branch, expanded, computing }` |
| `files` | `{ browsers: { [panelType]: { cwd, showHidden, lastError, items, loading, seq } } }` |
| `detail` (the viewer) | `{ lines, scroll, tab, search, select, cursor, contentTabs, ephemeralTerminals }` |
| `groups` | `{ list, expanded:Set, tab }` |
| `actions` / `stats` / `history` / `file-manager` | empty/trivial slice (uniform Component shape) |

- **Module-private state** (neither model nor slice). Effect-side state
  that never lived on `S` and doesn't fit either home; a future
  "collapse to one pole" wouldn't move these:

  - `hub.js` — pub/sub bus (topics, series, subscribers, `windowCache`)
  - `history.js` — action-history ring buffer
  - `copy.js` — held copy-option thunks; `cmdline.js` — held command registry
  - `docker.js` — the `docker events` subprocess (`_eventsProc`, buffer, timers)
  - `stream.js` (current child proc — `io/stream.js` at v0.5 time, since
    moved to `dispatch/stream.js` per v0.6.2 R8), `terminal.js` (PTY
    sessions), `render-queue.js`

### Logic homes

- **Root reducer** (`runtime.update`, ~90 Msg branches): view, chrome
  (`focus_set` / `nav_select` / `escape` / `list_select` / `multisel_*` /
  `panel_reset`), groups cross-layer Msgs (`set_current_group` /
  `reset_group_context` / `set_panel_cursor`), all modals (`filter_*` /
  `menu_*` / `confirm_*` / `prompt_*` / `copy_*` / `register_popup_*` /
  `cmdline_*` / `design_*` / `prefix_*` / `terminal_*`), framework
  (`set_layout` / `focus_event` / `stream_start` / `refresh` / `quit`).
- **Component `update()`** — docker / config-status / files / detail /
  groups / actions / stats / history / file-manager.
- **`handleAction`** (`dispatch.js`) — the effectful arms + Msg→Cmd
  routing (`nav_*`, `focus_*`, `run_selected`, `page_*`, `goto_*`).
- **Effect registry** (`effects.js`) — the unified Cmd/effect vocabulary
  shared by `runtime.update` and Component `update`. Built-ins:
  `setDetail`, `focus`, `render`, `tick`, `apply_msg`, `dispatch_msg`,
  `show_selected_info`, `destroy_pty_session`, plus the root-reducer
  Cmds (`force_full_repaint`, `refresh`, `show_help`,
  `start_design`, `quit`, `do_run`, `run_action`, `kill_proc`,
  `stream_action`, `copy_commit`,
  `emit_osc52`, `cmdline_rebuild/run/clear`, `menu_action`,
  `run_binding`). Component-registered: `cfgStatusCompute`,
  `cfgStatusDiff`, `dockerFetch`, `dockerEventsStart`, `dockerExec`,
  `dockerShell`, `loadDir`, `openFile`, `resetPanelChrome`.

### The "one API" question

The Plugin API survives as the simpler shape for **external / user-
authored** plugins. The choice is no longer "stateful → Component,
stateless → Plugin" (that was the language while both shapes were in
use in-tree). Today it's:

- *External plugin, stateless view over the model:* a Plugin. Less
  boilerplate, no `update`, no slice. (`register(plugin)`.)
- *External plugin, needs a slice / async / replay:* a Component.
  (`registerComponent(component)`.)
- *In-tree panel:* a Component, by convention. The taxonomy in
  [v0.5-layering.md](v0.5-layering.md) (Navigator / Viewer / Monitor)
  classifies them.

Both shapes are TEA. Collapsing to one surface would be a deliberate
redesign, not a cleanup — pick a pole and absorb the cost of the
other side. The current split is the design.
