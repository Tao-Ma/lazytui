# Migrating YAML configs from v0.6.0 → v0.6.1

> **Superseded by v0.6.2.** v0.6.2 replaces `layout.left:` / `layout.right:`
> blocks with `layout.columns: [...]` — see
> [`v0.6.2-migrate.md`](v0.6.2-migrate.md). Error strings that said
> "right column" now say "last column"; the `last pane of the right
> column` invariant is now `last pane of the LAST column`. This file
> remains useful for the cell-shape migration (v0.6.0 inline →
> v0.6.1 pool-ref); apply both guides if jumping from v0.6.0 to
> v0.6.2+.

v0.6.1 generalizes the v0.6 grid so every placement slot is a **pane**
and every piece of content is a **tab**. The YAML cell shape changed
to match — the v0.6.0 inline `{type: ...}` cell form no longer parses.

> **No silent rewrite.** v0.6.0 configs fail to load with a parser
> error that points at this file. Migration is a hand-edit; the steps
> below are mechanical.

For the design context behind the change, see
[`v0.6.1-panes-tabs.md`](v0.6.1-panes-tabs.md).

## TL;DR — three rules

1. Pool entries must be declared under a top-level `panels:` block
   (was: synthesizable from inline cells, now: required).
2. Layout cells reference pool entries — bare pool-id string for a
   single-tab pane, or `{tabs: [pool-id, ...]}` mapping otherwise.
3. Placement overrides (`heightPct`, `collapsed`, `height` on detail)
   stay on the cell, never on the pool entry.

## What changed in the YAML

### v0.6.0 — inline `{type: ...}` cells (no longer accepted)

```yaml
layout:
  left:
    width: 32
    panels:
      - type: containers
        title: Containers
      - type: groups
        title: Groups
  right:
    panels:
      - type: actions
        title: Actions
      - type: stats
        title: Stats
        topic: docker.stats
        select_from: containers
      - type: detail
        title: Detail
        height: 60%
```

### v0.6.1 — declare in the pool, reference from layout

```yaml
panels:
  containers:
    type: containers
    title: Containers
  groups:
    type: groups
    title: Groups
  actions:
    type: actions
    title: Actions
  stats:
    type: stats
    title: Stats
    topic: docker.stats
    select_from: containers
  detail:
    type: detail
    title: Detail

layout:
  left:
    width: 32
    panels:
      - containers
      - groups
  right:
    panels:
      - actions
      - stats
      - { tabs: [detail], height: 60% }
```

The pool entries carry **identity** (`type`, `title`, plugin config
like `topic`, `select_from`, `source`, `decorators`); the layout cells
carry **placement** (`heightPct`, `collapsed`, the detail-only `height`
override).

## The conversion, step by step

### 1. Split each inline cell into a pool entry + a cell reference

For every cell of the v0.6.0 inline form:

```yaml
- type: stats
  title: Stats
  topic: docker.stats
  select_from: containers
```

Lift the panel's identity into the pool, keyed by a stable id:

```yaml
panels:
  stats:
    type: stats
    title: Stats
    topic: docker.stats
    select_from: containers
```

Then reference it in the layout:

```yaml
layout:
  right:
    panels:
      - stats
```

The pool id is yours to name. For one-of-a-kind panels, using the
panel `type` as the id (e.g. `containers: { type: containers }`) keeps
diffs minimal and reads naturally. For panels with non-default
configuration — say a `files` panel that lives at `/var/lib/postgresql/data`
— pick an id that describes the role (`pgdata`, `tracked-files`) or
keep `files` if the YAML only has one.

### 2. Multi-tab panes — `{tabs: [...]}`

A pane with two or more tabs uses the mapping form:

```yaml
layout:
  left:
    panels:
      - { tabs: [docker, logs] }
```

`activeTab` is optional — defaults to `tabs[0]`. Set it explicitly
when you want a non-first tab focused at boot:

```yaml
      - { tabs: [docker, logs], activeTab: logs }
```

Every entry in `tabs` must be a declared pool id. Two tabs of the same
kind in one pane are rejected (active-tab selector would collide).

Switch the active tab at runtime via the `:switch-tab <pool-id>`
cmdline verb — autocomplete restricts to the focused pane's other
tabs. Keyboard / mouse UX for cross-tab cycling is deferred to v0.7
(`docs/v0.6.1-panes-tabs.md` §Decisions #4 explicitly punts on the
keybind choice until use-patterns settle).

### 3. Placement overrides — also use the mapping form

Single-tab panes with overrides also use the mapping form, with
`tabs:` listing the single id:

```yaml
- { tabs: [groups], heightPct: 40 }
- { tabs: [detail], height: 60% }   # detail height belongs on the cell
- { tabs: [files], collapsed: true }
```

`heightPct` is a placement-only field (1–100, percentage of column).
`collapsed: true` starts the pane collapsed. `height: N%` is detail-
only and lifts onto the layout-level `detailHeightPct`. None of these
belong in the pool entry — the parser rejects them there with
`'<field>' is a placement field; put it on the layout cell`.

### 4. Hidden pool entries — still in the pool, not in the layout

A pool entry that no layout cell references is **hidden** — still
configured, surfaced in the free-config `w` overlay so you can summon
it back without editing YAML. Same as v0.6.0, just expressed cleanly:

```yaml
panels:
  notes: { type: viewer, title: Notes }   # not in any layout cell → hidden
```

## Invariants the parser enforces

These existed in v0.6.0 too; v0.6.1 restates them at the **tab** level
since one pane can now host multiple tabs:

- **Exactly one `detail` tab anywhere** — across both columns.
- **Detail must be in the right column** — and in the **last pane**
  of the right column. (Render order assumes detail anchors the
  column bottom.)
- **At most one `actions` tab anywhere.**
- **No `detail` / `actions` tab in the left column.**
- **Per-side pane caps** — left ≤ 6 panes, right ≤ 3 panes.
- **No duplicate kinds inside one pane** — two `files` tabs in
  different panes is fine, two in the same pane is rejected.
- **Detail can't be collapsed** — `{tabs: [detail], collapsed: true}`
  is rejected at parse time.

## Common errors and how to fix them

| Parser error | Likely cause | Fix |
|---|---|---|
| `v0.6 inline cell shape ({type: ...} / {id: ...}) is not supported in v0.6.1` | Inline cell still in YAML | Split into pool entry + cell reference per §1 above. |
| `layout cell mapping requires 'tabs: [pool-id, ...]'` | Mapping cell missing `tabs:` | Either use the bare-string shorthand or add `tabs: [<pool-id>]`. |
| `'tabs' must be a non-empty list of pool ids` | `tabs: []` or non-list | Provide at least one pool-id. |
| `'activeTab' must be one of the entries in 'tabs'` | `activeTab` references an id not in `tabs:` | Match `activeTab` to one of the listed pool ids. |
| `layout cell references unknown panel id '<x>'` | Bare-string or `tabs:` entry references an id missing from the pool | Declare `<x>` under `panels:` (or fix the typo). |
| `panels.<x>: '<field>' is a placement field; put it on the layout cell, not the pool entry` | Placement field (`heightPct` / `collapsed` / `height` / `hotkey`) leaked into pool | Move the field onto the cell that references this pool entry. |
| `layout must have exactly one tab of kind 'detail', found N` | Zero or two `detail` tabs across the layout | Add or remove until exactly one pane has a `detail` tab. |
| `tab of kind 'detail' must be in the last pane of the right column` | Detail tab placed somewhere else | Move the detail-containing pane to the bottom of the right column. |

## Round-trip

The serializer always writes both `panels:` and `layout:` blocks now
(the v0.6.0 "keep the legacy inline form for round-trip" gate is
gone). After running `:save-layout` in free-config mode, the file
lands in v0.6.1 shape regardless of how it was written before — so a
once-edited config stays in v0.6.1 form.
