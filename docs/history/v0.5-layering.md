## Why this doc exists

The v0.5 effort began as "delete `S` / reach pure TEA." After cataloging the
existing architecture (see [v0.5-tea.md](v0.5-tea.md)) the review found
(state *before* Phase A — the problem this doc solved, not the current code):

- **`S` wasn't the footgun.** By that point nothing in production wrote
  `S` directly — only the facade internals. Deleting `S` first wouldn't
  have made the architecture more correct.
- **The model had multiple writers.** `tabs.js`, `actions.js`, `layout.js`
  mutated the model *outside* the reducer — "single writer = `update`" was
  a convention partly followed, not an enforced invariant.
- **Both symptoms had one root cause:** every panel registered as a peer
  `panelType`, but panels weren't the same kind of thing. Heterogeneous
  roles shared one flat contract, so the real structure (who feeds whom,
  who hosts tabs, what chrome each owns) escaped as cross-cutting
  special-cases.

This doc proposes the missing foundation — a minimal panel taxonomy — and
shows that **single-writer and immutability fall out of it naturally**,
instead of needing to be enforced on a flat model. With single-writer
structural, `S` was retired at the end of the arc as a small mechanical
follow-up (chunks A–E), not as the architectural lever.

## The kinds (the foundation)

Three kinds of panel plus one overlay layer. Kept small on purpose — adding a
kind that doesn't absorb existing special-cases is its own speciality.

### Navigator
A cursor over items; selecting a row drives the viewer.

- **Owns:** items (config-derived or in its own slice), cursor + scroll +
  filter, multi-select.
- **Today (Phase C complete):** groups, actions, containers (docker), files /
  file-browser, history, file-manager, config-status. All are Components.
- **Contract:** `getItems`, `render`. Selection lands via the uniform
  `nav_select` Msg (root reducer writes `ui.sel[panel]`); a Navigator that
  wants to cascade further (groups → currentGroup) handles a `dispatch_msg`
  follow-up in its own `update`.

### Viewer
The host layer. One per app today (`detail`). Doesn't navigate; *displays
content + hosts tabs*.

- **Owns:** active tab index + per-tab content, scroll, search, visual-select.
  Tab storage (content tabs, ephemeral terminals) lives here too — it migrates
  out of `tabs.js`'s globals.
- **Inputs:** "show content" (from a Navigator selection) and "add tab" (from
  an action run, a file open, a terminal spawn).
- **Contract:** `setContent`, `addTab`, `closeTab`, `nextTab` / `prevTab`,
  `scroll`, `search` — all Msgs handled by the Viewer's `update`.

### Monitor
Pure render of derived data. No cursor.

- **Today:** stats.
- **Contract:** `render(state, derivedSource)`.

### Overlay (already centralized — listed for completeness)
Transient input modes (menu, confirm, prompt, cmdline, copy, register-popup,
design). Not panels; the v0.5 spine already folded these into the root reducer
(`model.modal.*`). They overlay the panel grid; no change planned.

## The relationships

```
┌─ Navigators (N) ─┐     ┌─ Viewer (1) ──┐     ┌─ Monitors ─┐
│ groups           │  →  │ active tab    │     │ stats      │
│ containers       │  →  │ content       │     └────────────┘
│ files            │  →  │ tabs (info /  │
│ actions          │  ↘  │  output /     │
│ history          │     │  content /    │
└──────────────────┘     │  terminal)    │
                         │ scroll/search │
                         └───────────────┘
                                  ↑
                                  │ "add tab" / "set content"
                                  │ (action runs, file opens, …)
```

- **Navigator → Viewer** is the *only* directional cross-layer channel.
- **The Viewer owns tab management.** No more `tabs.js` globals or
  `m.focus = 'detail'` from outside.
- **Modals are orthogonal** — they overlay everything; not part of the
  Navigator-Viewer grid.

## What this absorbs (mapping current specialities)

| Current mess | Where it lives in the layered model |
|---|---|
| `setDetail` called from ~13 places | One channel: `viewer_set_content` Msg |
| `tabs.js` writes `m.focus`, content/term tab maps | Viewer layer's `addTab` / `focusViewer` Msgs |
| `actions.js` writes `model.lastRunAction` | `action_run` Msg → root reducer |
| `layout.js` writes `model.panelHeights` / `panelBounds` (render-phase) | View output (returned from render), not stored on the model |
| Component vs Plugin "two APIs" confusion | Decided by *what kind of panel it is* — Navigator / Viewer / Monitor (conceptual labels; not stored on panel defs) |
| Half-owned Component chrome (panel slice + central `ui.sel`) | Chrome is defined per kind: Navigator chrome (cursor/scroll/filter) lives where Navigators own it |

**Not absorbed by layering** (separate cleanups, easier afterward): the
two side-effect interpreters were a follow-on; that's since been resolved
— `runtime.update` Cmds and Component `update` effects share one registry
(`effects.runEffects`).

## The correctness cascade (why single-writer + immutability fall out)

```
Panel layering (the foundation)
   ↓ enables
Single-writer  every cross-layer op has a Msg channel — no need to poke
               globals (tabs.js / actions.js writes become layer Msgs;
               layout.js geometry moves off the model into view output)
   ↓ enables
Immutability   each layer's update returns a fresh slice (layer-local spread
               is cheap — Components already do this); root model becomes
               thin assembly of slices + chrome
   ↓ enables
S deletion     with no writers, the shim was just dead surface; removed
               outright at the end of the arc (chunks A–E). Plugin
               ergonomics preserved: external plugins read the model via
               `getModel()` + the `getSel` / `getScroll` / `isMultiSel`
               helpers.
```

Without layering you'd be enforcing correctness rules against an unstructured
model — patching exceptions forever. With it, the structure produces the
correctness properties naturally.

## What landed (Phase A → B → C → step 5)

Each phase shipped as its own commit; the suite stayed green between them.

### Step 5 — Single-writer enforcement (commit `725b257`)

Closed the remaining outside-writers BEFORE going Component:
- `actions.js`'s `model.lastRunAction = …` → `set_last_run_action` Msg.
- `dispatch.js` wedge-guard (`model.modes[flag] = false` on handler throw) →
  `mode_clear` Msg.
- `input.js` mouse focus + wheel scroll → `focus_set` / `viewer_scroll` Msgs.
- `viewer.js`'s `switchToTab` (tab + terminalMode writes) → `viewer_set_tab`
  + `terminal_exit` Msgs.
- `layout.js`'s PTY-exited terminalMode cleanup → `terminal_exit`.

### Phase A — Cross-layer write extraction (commit `32f935b`)

Discovered preparing for Phase B: three viewer_* reducer branches wrote
ACROSS layers (`model.modes.terminalMode` + `model.focus` alongside the
viewer slice); the group-tree cascade did the same. These were the *real*
obstacle to encapsulating detail. Phase A introduced the
**`apply_msg` Cmd** (the cross-layer Msg-dispatch primitive) and split each
cross-layer write into a Cmd carrying the target Msg. After Phase A every
reducer branch only writes its OWN layer's fields.

### Phase B — detail becomes a Component (commit `bb0ed7e`)

The viewer state (was `model.viewer` + ~14 viewer_*/select_*/stream_start
branches in `runtime.update`) moved into the detail Component's slice + its
own `update`. While `S` still existed, the state.js shim re-pointed
`S.detailLines` / `S.contentTabs` / etc. to `getComponentSlice('detail')`
and lazy-registered on first access; the lazy-register half survives in
`state.js` (used by tests that touch detail state without an explicit
register call). Cross-layer writes (modes / focus / pty teardown) emit
`apply_msg` / `destroy_pty_session` effects from `detail.update`. Two
parallel effect registrations added (`apply_msg` + `show_selected_info` +
`destroy_pty_session`) so the Component-side effect channel matches the
root-reducer Cmd channel.

### Phase C — groups becomes a Component (commit `b093387`)

The last in-tree Plugin. groups owns its tree slice (`list` / `expanded` /
`tab`); the app-wide cascade (`currentGroup`, per-group root-chrome reset,
viewer reset) goes out as `apply_msg` / `dispatch_msg` Cmds. `nav_select`
became uniform (writes `ui.sel[panel]` for all panels; for `panel='groups'`
also emits `dispatch_msg { groups_selected, index }` so the Component
runs the cascade). Three new cross-layer Msgs in `runtime.update`:
`set_current_group`, `reset_group_context`, `set_panel_cursor`. The
`dispatch_msg` effect was registered (parallel to `apply_msg`) so a
Component update can dispatch into ANOTHER Component (e.g. groups →
viewer_reset_chrome → detail).

### Blessed outside-writer exceptions

A 2026-05-28 audit (find every direct `model.X = …` write outside an
update branch) found these — and only these — categorically-different
exceptions, each documented inline at the site:

- **`render/layout.js`** writes `layoutSlice.panelHeights` /
  `layoutSlice.panelBounds` during render (Phase 1e re-homed them onto
  the layout Component's slice from the root model). The layout engine's
  *view output*, memoized on the slice so mouse hit-tests + design-mode
  drag math can read without recomputing. The viewer Component does the
  same for `panelBounds.detail.tabs` (the tab-bar hit-test cache).
- **`render/layout.js calcLayout`** dispatches `set_scroll` Msgs from
  the layout pass (one per visible navigator, per resize). The scroll
  resync keeps each panel's selected item in view as heights change.
  It's a Msg dispatch rather than a direct slice write — the owning
  navigator's `update` remains the single writer of `nav.scroll` — but
  the dispatch is initiated from a render-pipeline call site, so it
  belongs in this list as a renderer-as-trigger pattern.
- **`panel/monitor/stats.js _ensureSub`** registers a hub subscription
  on first render of each `(topic, window)` pair. Idempotent (deduped
  by key in a module-private Set). Mixes lifecycle with paint; the
  alternative (subscription bookkeeping in the stats Component's
  update on a config-derived Msg) is cleaner but no current pathway
  triggers a topic change so the cost outweighed the benefit at v0.5.
- **`register.js`** lazy-initializes `model.register` on first use; **`tui.js`**
  writes `model.designEnabled` from the `--design` CLI flag at boot. Both
  write-once init — no second writer to disagree with them.

### Skipped at this layering arc (later revisited)

- **Step 6 (immutability)** — when this doc was written, full immutability
  across ~90 root-reducer branches was identified as buying nothing
  without snapshot-tests / replay. **Reversed by the pure-TEA arc** (Phases
  0-4, tip `2be348a`): the reducer + every Component's `update` + every
  reducer leaf now return new state objects. The pay-off ended up being
  observability (state transitions visible at the return point) and the
  bug-class detection that comes with it — two real stale-ref bugs caught
  in the arc, both at integration sites that pre-conversion silently
  passed. Freeze-test coverage in `js/test/test-immutable-*.js`.
- **Step 7 (`S` lockdown)** — closed differently than originally planned.
  Rather than freezing setters, `S` was removed outright once nothing
  still wrote (or read) it (chunks A–E, tip `4cc0bcf`). The ~180-site
  test migration that lockdown would have cost was paid as the
  straight-line de-shimming work over the same chunks.

## Things to know about the post-Phase-C shape

- **Module-private state stays mutable + effect-side.** The hub, PTYs, the
  `docker events` subprocess, `layout.js`'s diff cache, `cmdline.js`'s
  held registry, `copy.js`'s held thunks — none of these are model state.
  They live in module scope and the single-writer rule doesn't apply to them
  (they're effect-side / view-side caches, not the system's state).
- **Reducer-leaves are pure return-new transforms.** `leaves/design`,
  `leaves/register`, `leaves/search`, `leaves/tabs`, `leaves/nav` all
  take their slice and return a new one (or the same ref on no-op).
  `leaves/menu` is a pure builder. This was the eventual outcome of the
  pure-TEA arc — the "Skipped" §above describes the prior position that
  the arc reversed. The slot still describes them as "reducer-leaves"
  rather than helpers because the dispatch shape (call site is a
  reducer/Component branch) is unchanged.
- **One Viewer.** detail is a singleton; if a future panel needs a second
  viewer (split detail), nothing structural prevents it — register a
  second Component with the same viewer-shaped contract (`addTab` /
  `setContent` / etc.) and route to it.
- **Kinds are conceptual, not stored.** Navigator / Viewer / Monitor name
  the roles in this doc but aren't recorded on the panel defs — the
  taxonomy is doc-as-design, not data. (An earlier draft stored
  `kind:` per def; it was paper-only and was dropped.)
- **One effect channel.** `runtime.update` Cmds and Component `update`
  effects share the same registry — `effects.runEffects` interprets both,
  called from `dispatch.applyMsg` (root path) and `plugins/api.dispatchMsg`
  (Component path). All handlers register via `effects.registerEffect`,
  built-ins set up at boot by `effects.installBuiltins()`.

## Status

**Phase A → B → C + S-drop all landed (2026-05-28).** The architectural
goal — correct, consistent, single-writer per layer — is realized:

- **9/9 panels are Components.** `docker`, `config-status`, `files` (+
  `file-browser`), `actions`, `stats`, `history`, `file-manager`, `detail`,
  `groups`. Zero in-tree Plugins; the Plugin API survives as the documented
  shape for stateless user/external plugins.
- **Single-writer per layer is structural.** Root model fields (chrome /
  modal / config / framework) are written only by `runtime.update` branches.
  Component slices (detail / groups / docker / files / config-status) are
  written only by their own `update`. Cross-layer ops route through
  `apply_msg` (Component → root) and `dispatch_msg` (root → Component, and
  Component → Component) — both registered as Cmds AND as effects so either
  side can emit them.
- **`S` is deleted.** Chunks A–E (commits `cc5f566` → `4cc0bcf`)
  migrated production code, then tests, then removed the shim + export.
  `state.js` slimmed from 483 → 316 LOC; readers use `getModel()` /
  `getComponentSlice()` directly.
- **Audit clean.** A 2026-05-28 sweep for unsanctioned cross-layer writes
  found none; every direct `model.X = …` outside an update branch is in
  the blessed set above (view-output / lazy / boot init).

### What's *not* done (deliberately)

- **Immutability** — see "Skipped (deliberate)" above.
- **Surface unification (option (c))** — `render(panel, w, h, ctx)` with one
  uniform context object covering both Component slice and root reads. A
  redesign of the panel API, not a cleanup. Defer to v0.6 if ever wanted.
