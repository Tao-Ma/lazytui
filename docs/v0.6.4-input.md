# v0.6.4 Theme F — unified key/mouse intent layer + mouse actions

> **Status: SPEC (Phase 0), 2026-06-11.** Theme F is the one
> **user-facing feature** of v0.6.4 — Themes A–E ([v0.6.4.md](v0.6.4.md))
> were refactor-only. It (1) introduces a shared **intent layer** so
> keyboard and mouse both normalize to the same semantic vocabulary
> before reaching the reducer, and (2) adds the common mouse actions that
> the inline-orchestration `handleMouse` never grew:
> **left-click = select+focus, double-click = activate, right-click =
> context menu**, plus middle-click reserved. All gesture→intent
> mappings are designed to become **YAML-overridable** (parallel to the
> existing `keys:` block), wired in Phase 4.

---

## Why

The keyboard side is already **reducer-driven through one chokepoint**:
`handleNormalKey` (a key→verb switch, `dispatch.js:437`) funnels into
`handleAction(verb, arg)` (`actions.js:198`) — the single name→effect
switch shared across bare keys, leader chords, `:`-cmdline, and the
command menu. A key never reaches the store except as a named verb that
some reducer arm owns.

The **mouse side has no such chokepoint.** `handleMouse(kind, x, y)`
(`input.js:350–599`, ~250 LOC) does geometry-resolution **and** effect
dispatch inline: it hit-tests chrome glyphs, walks the modal mouse
registry, handles the wheel, runs detail text-selection, resolves the
tab-strip, and finally does the focus+select cascade — calling
`dispatchMsg` / `render` directly at each branch. There is no
intermediate "this click *means* focus-this-pane-and-select-this-row"
representation; the meaning is implicit in which `if` arm ran.

Two concrete consequences:

1. **Inconsistency.** A maintainer adding a new behavior wires it once on
   the keyboard side (a verb + a reducer arm) and a *second, differently
   shaped* time on the mouse side (a hit-test arm + an inline dispatch).
   The two can — and historically did — drift (the mouse-mode precedence
   list was a hand-pinned duplicate of `CHAIN_MODES` until C1 derived it;
   `input.js:310–320`).
2. **Buttons are dropped at the door.** `input.js:748–749`
   (`if (button !== 0) continue;`) discards every non-left button for
   non-wheel events, so right- and middle-click never reach `handleMouse`
   at all. There is nowhere to *add* a right-click behavior without first
   un-dropping the button — and once un-dropped, the only place to handle
   it is yet another inline arm.

Theme F closes both: a **shared intent vocabulary** that keyboard and
mouse both emit, so `handleMouse` stops orchestrating (it *resolves a
gesture to an intent*, exactly as `handleNormalKey` resolves a key to a
verb), and new buttons become additive gesture→intent rows rather than
new inline arms.

## The intent vocabulary

An **intent** is the semantic middle layer: *what the user wants*,
independent of whether they typed a key or moved a pointer. Five shared
intents cover the navigation/activation core common to both input
sources:

| Intent | Meaning | Payload |
|---|---|---|
| `focus`    | Make a pane the focused pane | `{ paneId }` |
| `select`   | Set / move the cursor within a pane | `{ paneId, idx }` (mouse: absolute idx; keyboard: relative ±1 via existing `nav_up/down`) |
| `activate` | Run / open the selected item | `{ paneId }` |
| `context`  | Open the context menu | `{ paneId, anchor: {x, y} }` |
| `scroll`   | Scroll a pane's content | `{ paneId, dir: ±1, amount }` |

Each intent is **realized by emitting the existing reducer Msg(s)** — the
intent layer adds no new store concepts. `focus` → `focus_set`; `select`
→ the `navSelect` cascade; `activate` → `run_selected` (`handleAction`);
`context` → `menu_open` (threading `buildItems`, `leaves/menu.js`);
`scroll` → the existing wheel / `page_*` paths. The intent layer is a
**naming + routing seam**, not a new state machine.

### What is deliberately NOT a shared intent

Two classes stay outside the five-intent set, by design:

- **Specialized keyboard verbs** — filter (`/`), copy (`y`), cmdline
  (`:`), register popup (`"`), list-select (`v`/space), help (`?`), view
  expand/shrink (`+`/`_`), tab cycle (`[`/`]`). These have no pointer
  analog today; they keep their existing `handleAction` / `applyMsg`
  dispatch unchanged. The intent layer covers the **common core**, not
  every key.
- **Mouse-direct manipulations** — chrome-glyph clicks (`[_]`/`[+]`
  collapse, `[X]` close, `[≡]` tab-list / pane-select triggers),
  tab-strip switch/close, and detail text-selection drag
  (begin/extend/commit). These are **spatial direct manipulation** with
  no keyboard equivalent; they remain mouse-resolved. The reorganization
  (Phase 2) groups them as an explicit *pre-resolution cascade* that runs
  before a click can fall through to the shared `focus`+`select` intent —
  which is exactly the order `handleMouse` already uses, just named.

This honesty matters: Theme F does **not** claim to make every gesture a
keyboard-equivalent intent. It unifies the navigation/activation core and
gives the rest a consistent home.

## key → intent table

The existing key→verb mappings, re-read as intents. **No keyboard
behavior changes** — this table just shows the mapping the intent
scaffold formalizes in Phase 1 (the verbs already route through
`handleAction`).

| Key(s) | Verb today | Intent |
|---|---|---|
| `j` `↓`        | `nav_down`    | `select` (relative +1) |
| `k` `↑`        | `nav_up`      | `select` (relative −1) |
| `h` `←`        | `focus_left`  | `focus` |
| `l` `→`        | `focus_right` | `focus` |
| `Enter`        | `run_selected`| `activate` |
| `x`            | `menu_open`   | `context` (anchor = focused pane) |
| `1`…`9` (hotkey)| `focus_panel` | `focus` |
| wheel / `,` `.`| `page_*` / wheel | `scroll` |

Everything else in `handleNormalKey` stays a specialized verb (above).

## gesture → intent table

The new mouse mapping. The **pre-resolution cascade** (chrome → tab-strip
→ text-select) runs first on a left press; only a press landing on a pane
*body* produces the shared `focus`+`select` intent.

| Gesture | Resolves to |
|---|---|
| left press (pane body)   | `focus` + `select` (the row under the cursor) |
| left press (chrome/tab/body-text) | mouse-direct (collapse/close/trigger/tab-switch/tab-close/selection-begin) — unchanged from today |
| **double left press** (≤250 ms, same cell) | `activate` |
| **right press**          | `context` (anchor at cursor) |
| **middle press**         | *reserved* — emitted as a `middle` gesture, mapped to a **no-op intent** today |
| left motion (button held)| text-select extend (mouse-direct) |
| left release             | text-select commit (mouse-direct) |
| wheel up / down          | `scroll` (±1) |

## Double-click model — ~250 ms, same-cell, in the SGR parser

**Decision (2026-06-11): a ~250 ms same-cell window, tracked in the SGR
parser** (`input.js:735–752`), not in the model/reducer.

Rationale: the parser is the only layer that sees raw press timing, and a
self-contained `{ lastX, lastY, lastTime }` triple there keeps the
double-click derivation off the hot reducer path and out of the model
state. A press emits gesture `double` when `button === 0 (left)` **and**
`x === lastX && y === lastY` **and** `now − lastTime ≤ 250 ms`; otherwise
gesture `press` (and the triple updates). Timing uses `Date.now()` (this
is the live app, not a workflow script — no resume constraint).

- **Window = ~250 ms** (tighter than the 400 ms first floated). Fewer
  accidental double-fires from two deliberate single clicks; the tradeoff
  is a missed double on a very laggy SSH session, judged acceptable and
  YAML-tunable later.
- **Same cell** (`x`,`y` equality, 1-based SGR coords) — not "near", to
  keep the parser dumb and avoid a distance metric.
- A `double` does **not** suppress the preceding single click's
  `focus`+`select` — the first press already focused+selected the row; the
  double then `activate`s it. This matches the GUI convention (single
  selects, double opens) and needs no "swallow the first click" timer.

The parser emits a **gesture**, not an intent — `handleMouse` (Phase 2)
maps `double` → `activate`. So the timing logic and the semantic mapping
stay in separate layers (parser = *what happened*, resolver = *what it
means*), and the window is overridable independently of the mapping.

## Middle-click — un-dropped, reserved no-op

**Decision (2026-06-11): un-drop middle-click at `input.js:749` and route
it through as a `middle` gesture mapped to a no-op intent.** This wires
the plumbing end-to-end now (parser → gesture → resolver → intent) so a
future behavior — primary-selection paste, or any YAML-bound action — is
a **pure additive** gesture→intent row, not a re-plumb. No user-visible
behavior today; the gesture is recognized and discarded by the resolver.

(The alternative, leaving it dropped, would mean a future middle-click
feature has to re-touch the SGR button filter. Reserving it now costs one
no-op branch.)

The button filter at `input.js:748–749` becomes: left (0) → existing
gestures; right (2) → `context`; middle (1) → `middle` (no-op); other →
drop. Wheel (the `0x40` bit) is already handled above the filter.

## YAML hook (designed now, wired Phase 4)

Gesture→intent mappings are **data**, mirroring the existing `keys:`
override block. The default table (above) lives in code; a config may
override it:

```yaml
mouse:                 # parallel to `keys:`
  double-click: activate
  right-click:   context
  middle-click:  paste     # e.g. once a paste intent exists
  double-click-ms: 250     # the window, tunable
```

Phase 4 introduces the parse + default-merge; Phases 1–3 hardcode the
defaults so each phase is independently shippable. The override surface
is intentionally the **gesture→intent** edge (not gesture→Msg) — configs
bind to stable semantic intents, the same way `keys:` binds to verbs, so
internal Msg renames don't break configs.

## Phased plan

Each phase keeps the suite green and is independently shippable. Phases
1–2 are **behavioral no-ops** (pure restructure); the feature lands in 3.

1. ✅ **Intent scaffold + keyboard, no-op — SHIPPED 2026-06-11.**
   `dispatch/intent.js` defines the vocabulary (`focus`/`select`/
   `activate`/`context` constructors + a single `realize`) with module
   refs resolved lazily+memoized (dispatch.js requires it → load-time
   require-back would cycle). `handleNormalKey`'s core nav keys route
   through it: `k/↑`→`selectBy(-1)`, `j/↓`→`selectBy(+1)`, `h/←`/`l/→`
   →`focusDir`, `Enter`→`activate`, numeric hotkey→`focusHotkey`, and the
   `x` key's genuine menu-open→`context()` (its dead-terminal / content-tab
   close preamble stays inline — those are keyboard-specific `x` overloads,
   not context-menu semantics). The keyboard forms **delegate straight back
   to the existing `handleAction` verbs**, so the realization is
   byte-for-byte the prior dispatch — a true no-op. `handleFilterKey`'s
   modal `j/k` is untouched (modal handlers sit above the intent layer).
   The absolute forms (`focusPane`/`selectAt`) + `context`'s anchor are
   built but unused until Phases 2–3. `test-intent.js` (14 checks) pins
   each intent→downstream-call mapping by spying the delegate targets;
   full suite green (80/81 — `test-archive.js` is the pre-existing
   xz-not-installed env failure) + smoke 6/6.
2. ✅ **Mouse through the intent layer — SHIPPED 2026-06-11.**
   `handleMouse`'s focus+select tail now routes through the layer: the
   spatial resolution (which pane, which row) stays inline, then
   `focusPane(paneId, {skipInfo})` + `selectAt(paneId, idx)` realize the
   focus + select (replacing the inline `dispatchMsg(focus_set)` +
   `navSelect`). `focusPane` gained a `skipInfo` field so the cascade's
   double-`show_selected_info` guard is preserved byte-for-byte. The wheel
   routes through a `scrollAt(mx, my, delta)` intent that **delegates to
   `_handleWheel`** — pointer scroll is spatial *and* per-pane
   heterogeneous (detail scrolls content; a list pane moves its own
   cursor, with focused / side / groups variants), so that resolution +
   behavior stays in `_handleWheel`; the intent is the seam, and unifying
   the list-arm into `select` + the detail-arm into a content-scroll Msg
   is an explicit later semantic pass (not a mechanical lift). The
   pre-resolution cascade (chrome glyphs → modal mouse registry →
   tab-strip → text-select) is unchanged — it short-circuits to its
   mouse-direct Msgs before any click can fall through to the
   `focus`+`select` intent, exactly as before. **Behavioral no-op for
   left-click / wheel / drag** (left-only world still). `test-intent.js`
   grew the mouse-intent mappings (18 checks total); full suite 80/81
   (xz env-only) + smoke 6/6; `test-mouse-wheel` / `test-select` /
   `test-half-view-focus` green unchanged.
3. **New buttons + double-click (THE feature).** SGR parser: add the
   ~250 ms same-cell double-tag and the middle/right button rows
   (un-drop `input.js:749`). Resolver: `double` → `activate`, `right` →
   `context` (→ `menu_open` via `buildItems`, **anchored at the cursor** —
   the intent threads `{x,y}`, the menu render clamps + opens there),
   `middle` → no-op. First user-visible change. New tests: double-click
   activates the row under cursor; right-click opens the menu; middle is
   inert; a slow double (>250 ms) stays two singles.
4. **Defaults + YAML hook.** Parse the `mouse:` block; merge over the
   code defaults; make `double-click-ms` configurable. Parser tests for
   override + default-merge; an example in a demo config.
5. **Smoke + docs.** A smoke scenario driving raw SGR bytes →
   gesture → intent → Msg for each new gesture (double / right / middle /
   slow-double). Doc parity: fold the shipped status into this file +
   update [v0.6.4.md](v0.6.4.md)'s Theme F row; refresh
   [DATAFLOW.md](DATAFLOW.md) with the intent seam.

## Risk / no-op guarantees

- **Phases 1–2 are pure restructure.** The realized Msgs are byte-for-byte
  the ones `handleNormalKey` / `handleMouse` emit today; identity tests
  pin this. The risk is a missed branch in the ~250-LOC `handleMouse`
  lift (Phase 2) — mitigated by keeping the pre-resolution cascade
  *literally the same arms*, only relocated, and by the existing mouse
  test coverage (`test-mouse-wheel.js`, `test-panel-border-color.js`,
  smoke `drag.js` / `hit-zones.js`).
- **The modal mouse registry** (`_modeMouseHandlers`, `input.js:304`) and
  its keyboard sibling (`_modeHandlers`, `dispatch.js:708`) sit *above*
  the intent layer — a modal claim short-circuits before intent
  resolution, unchanged. Theme F does not touch modal precedence (C1's
  `CHAIN_MODES`-derived order stays the single source).
- **Right-click reuses `menu_open` + `leaves/menu.js buildItems`** as-is —
  no new menu machinery. The only new input is the cursor anchor: the menu
  **opens at the cursor** (decision 2026-06-11), so the `context` intent
  threads `{x, y}` into `menu_open` and the menu render anchors there
  (clamped to stay on-screen) rather than at a fixed position.
