## Theme A — multi-instance Components

### Goal

Let **two panes of the same kind each own their own slice**, so a
config can place, e.g.:
- two `detail` viewers side-by-side showing different files,
- two `files` browsers parked at different `cwd`s,
- a `docker` panel in two columns watching different container sets.

Today every Component *kind* (`viewer`/detail, `groups`, `files`,
`docker`, `actions`, `stats`) has **exactly one** live instance / slice.
v0.6.1 made *panes* generic containers and *tabs* generic content, but a
pane still hosts one instance per kind. Theme A removes that "one per
kind" ceiling.

It is "the spine" because it is the one new *capability*; every other
Theme-A sub-item is a consumer that currently shortcuts through "the
single instance of this kind."

### Scope decision (what v0.6.4 ships vs defers)

- **IN:** same-kind panes in **distinct slots**, **declared in YAML**
  (`panels: { d1: {type: detail}, d2: {type: detail} }`). This is the
  minimal coherent capability; everything else builds on it.
- **DEFER:** two same-kind *tabs inside one pane* (one pane, one focus,
  two slices — needs a tab→instance map; harder). And a runtime
  "split this pane" gesture (a feature on top of the declared form).

### What is already built (v0.6.3 B-phase groundwork)

More than half the substrate exists:
- **Store is paneId-keyed, not kind-keyed.** `route._instances[id] =
  { id, kind, slice }` where `id` is the paneId (`pane-detail`,
  `pane-groups`), not the kind (`js/panel/route.js:100-147`).
- **Minting is per-pane at PLACE time.** `app/state.js:151-165`
  iterates *every* placed pane and mints one instance per `paneId`.
  Two same-type panes (`pane-d1`, `pane-d2`) both mint today — the
  store already supports coexistence.
- **The parser already lets you declare it.** `panels:` is a Map keyed
  by *your* id with `type:` a field; `paneId` derives from the pool id
  (`buildPlacedPane → newPaneId(activeTabPoolId)`, `parser/index.js:219`).
  So distinct ids → distinct paneIds, even at the same type.
- **paneId → kind resolution works once placed.** Verified:
  `route.instanceKind('pane-d2') === 'detail'` (resolves via the
  instance's stamped `kind`).

### What actually blocks it — the consumers that resolve by "primary"

`route._primaryByKind[kind]` maps a kind → its *first* instance. These
hot paths resolve through it (or by panel-type) and would address the
**wrong instance** with two same-kind panes:

1. **Keyboard routing — `dispatchKeyToFocused` (`panel/api.js:419`)** —
   ✅ **FIXED in Phase 1 (`b4ff8d6`).** It routed the key to
   `getPrimaryByKind(compName)`, NOT the focused paneId — two viewers →
   keystrokes hit the primary. Now resolves the focused paneId
   (`hasInstance(focus) ? focus : getPrimaryByKind`), falling back to
   the kind primary only for docker-style kind-keyed panes.
2. **`resolveTarget('viewer')`** — ✅ **no change needed.** Already
   focus-first (arm 1 = focused viewer-kind tab), so content already
   targets the focused viewer; the N13 concern was only about the
   non-focused fallback arms. (`dispatchMsg` likewise already does
   paneId-first routing — its `getPrimaryByKind` is the correct
   Component-name-form fallback.)
3. **B2 — `getInstanceSlice('detail')` literal lookups** (~13 sites)
   resolve to the primary; must route via `resolveTarget`/paneId.
4. **`paneBounds[type]` writes** are keyed by panel-type — one bounds
   rect per kind → two same-kind panes collide. Key by paneId (the
   deferred P1.4 write-retire; the 14 readers already use
   `boundsFor(paneId)`).
5. **Pool-id → Component-name registry** — a *raw pool id* that differs
   from its type does NOT resolve today: `route.instanceKind('myView')
   === null` (only the paneId `pane-myView` resolves). A standalone
   pool-id-keyed registry makes declared same-kind panes resolve
   independent of the arrange walk.
6. **refresh / hub / action fan-out** — currently per-kind / broadcast;
   must address the specific instance.

Net: two same-kind panes today would *render*, but share keyboard
routing, content target, and bounds with the primary — broken
interaction, not a crash.

### Phased plan

1. ✅ **Focused-instance routing — SHIPPED (`b4ff8d6`).**
   `dispatchKeyToFocused` resolves the focused paneId rather than
   `getPrimaryByKind`. Behavior-identical under singleton (verified:
   suite 79/80 + smoke 5/5, live single-pane key routing intact).
   `resolveTarget`/`dispatchMsg` were already focus-/paneId-first.
   Regression test in `test-instance-registry.js` (two same-kind
   instances; key lands on the focused one — fails pre-fix).
2. ✅ **paneId-key the framework bounds hit-tests — SHIPPED (`f65a48c`).**
   `paneBounds` is dual-written (type + paneId → same object), so two
   same-kind panes collide on the type key. Flipped the framework
   hit-test reads (`input.js` wheel + click loops, `render/decor.js`
   chrome-glyph) to `visibleBoundsFor(p.paneId)`. No-op under singleton;
   fixes the collision. Verified live (click-to-focus, `[_]` collapse,
   wheel). **Discovered entanglement (re-scoped):** free-config's bounds
   reads live inside its resize/height machinery, which is type-keyed
   end-to-end AND carries a singleton `arrange.detailHeightPct` schema
   assumption — making free-config multi-instance-correct needs that
   schema change, so it's a separate later unit, NOT a bounds-read flip.
   The `boundsFor('detail')` literals + the type-keyed paneBounds WRITE
   retire fold into Phase 3.
3. ✅ **'detail' bounds literals (N13) — SHIPPED (`7d60c65`).** The
   `getInstanceSlice('detail')` literals (B2) were already retired by
   T1.4 (only the bootstrap helper remains). Flipped the five
   `boundsFor('detail')`/`paneBounds.detail` literal readers (tab-drag,
   detail text-select, terminal restart + overlay, tab_drag_motion) to
   `resolveTarget('viewer')`. No-op under singleton; focused-viewer-
   correct under multi-viewer. **Write-retire NOT done — re-gated:** see
   the bounds-helpers endgame below.
3b. ✅/⏸ **Bounds-derived helpers — `getPanelViewportH` SHIPPED
   (`7defe67`); write-retire deferred.** `getPanelViewportH` was the
   type-contracted blocker (`instanceKind(visiblePanel) === panelType`
   + `boundsFor(panelType)`, 3 callers) — reworked to take a paneId
   (`visiblePanel === paneId`; `boundsFor(paneId)`); callers pass
   paneId. The ONLY remaining type-key readers are the free-config
   bounds reads — but flipping those alone **breaks ~105 test
   assertions** across 4 type-keyed fixtures AND does NOT make
   free-config multi-instance-correct (its arrange-mutation / height-SET
   machinery + the singleton `arrange.detailHeightPct` schema stay
   type-keyed). So the free-config bounds flip + the `paneBounds[type]`
   **write-retire** move INTO the free-config-multi-instance unit, where
   the fixture churn is justified. Until then the type-keyed write stays
   a harmless dual-write alias.
4. ❌ **Pool-id registry — ASSESSED, NOT NEEDED (skip).** Its premise
   ("`instanceKind(non-default-pool-id)` returns null → id≠type configs
   break") dates to v0.6.0, BEFORE the v0.6.3 `_typeByArrangePaneId`
   arrange-walk consolidation (`3f39031`) that fixed it. Verified: no
   runtime path passes a raw pool id to the resolvers (focus + arrange
   use paneIds; pane-select/panel-list read pool data directly); the
   two-pane test (ids ≠ type) rendered + routed fine. A standalone
   registry would only add a sync burden (dual source vs the
   single-source arrange walk) for no gain. Subsumed by v0.6.3.
5. **⚠ Component per-instance redesign (the REAL bulk).** Empirical
   two-`files`-pane test (2026-06-10): both panes RENDER and keys/content
   ROUTE correctly (Phases 1–3b work), but they SHARE config — FilesB
   (root /etc) showed FilesA's /tmp. Cause: the Navigators are
   internally **panelType-keyed** — `files._panelOf(panelType)` returns
   the FIRST pane of that type (files.js:92) and browser state lives in
   `slice.browsers[panelType]`; `getItems(slice)` can't even tell which
   pane it's for (slice isn't self-identifying). So a second instance's
   slice still computes against the first pane's config. Making this work
   is a **per-Component redesign**: each instance's slice must be
   self-contained (one browser, knows its own paneId/config — the
   v0.6.1 Phase-3 "keys collapse out" note), repeated for files / docker
   (`_slice()='docker'`) / config-status / etc. This is the heart of
   multi-instance and the bulk of the remaining cost — far bigger than
   the original "lift one-per-kind in mint/refresh/hub" framing.
6. **Acceptance demo + smoke scenario** (see below).
   *(Separate unit, gated on schema work: free-config multi-instance —
   the type-keyed resize/height machinery + `arrange.detailHeightPct`
   → per-pane height.)*

Phase 1 is the keystone (a no-op under today's single-pane configs).
2–5 follow in any order that keeps the suite green; each is
independently shippable.

### Backwards compatibility — does existing YAML change? NO.

**Existing configs are unaffected and require no edits.** Verified:
- The demos (`demo/postgres/tui.yml`, `demo/cloudberrydb/tui.yml`)
  declare one pane per type with id === type (`groups`, `files`,
  `detail`, …). Each mints exactly one instance; `_primaryByKind`
  points to it; every kind/primary fallback resolves to it.
- Multi-instance is **purely additive / opt-in** — a second instance
  exists only if you *declare* a second same-type pane. Single-pane
  configs are a strict subset; their behavior is byte-for-byte
  unchanged.
- The Phase-1 routing change (focused paneId instead of primary) is a
  **no-op when there's one instance per kind** — the focused paneId IS
  the primary.

The only NEW requirement falls on the new capability, not old configs:
declaring two same-type panes needs **distinct pool ids** (so at least
one id ≠ type), which is exactly why Phase 4 (pool-id registry) exists —
to make id ≠ type panes resolve robustly. It changes nothing for a
config that doesn't use the feature.

### Acceptance test

- A demo/fixture config with two `detail` panes (and/or two `files`
  panes at different roots) in distinct columns.
- A `js/test/smoke/` scenario: focus pane A, type/scroll → only A's
  slice changes; focus pane B → only B's; open a file in A → content
  lands in A's viewer, not B's; bounds/hit-zones are per-pane.
- Real-TTY `/verify`: drive both panes, confirm independent state.

### Open design decisions (settle before Phase 1)

- **Focus disambiguation.** Focus is already a paneId (post-B3), so
  `h`/`l`/click into a second viewer focuses *that* paneId and routing
  should follow. Confirm there are no cross-instance "primary"
  fallbacks left after Phase 1.
- **Entry point.** YAML declaration only (this spec), or also a runtime
  split gesture? (Deferred per the scope decision — confirm.)
- **`_primaryByKind`'s future.** Keep it as a boot/test fallback, or
  retire it entirely once focused-instance routing lands? (Lean: keep
  as the bootstrap/degenerate fallback; stop using it on hot paths.)

## Themes B–E (scoped; specs to follow)

See `[[v064-backlog]]` (auto-memory) for the consolidated detail.

- **B — `render/layout.js` restructure.** ✅ **Rename →
  `render/geometry.js` SHIPPED (`58fa514`)** — kills the dir-twin with
  `panel/layout.js`. ⏸ Remaining (optional, higher-risk): the SPLIT
  (footer → `render/footer.js` + geometry extract + chrome-context
  dedup across renderNormal/Half/Full). The `js/wm/` carve is an
  alternative shape — pick split-OR-carve if pursuing further; the
  rename alone already removes the navigation hazard. "geometry"
  slightly undersells the file (also holds paint + footer) — the split
  would make the name precise.
- **C — TEA purity retires.** DECISION 2026-06-10: root reducer IS held
  to full purity (thread slice values via Msg; routing reads stay).
  - ✅ **escape (1/4, `0c0e96c`)** — reads `msg.hadMultiSel` (handler
    computes via `multiSelCount`).
  - ✅ **_cycleViewerTab (2/4, `1e71718`)** — handler threads the viewer
    tab bundle `{curTab, total, tabKeys[]}`; arm keeps routing
    (resolveTarget) + the pure cycle math. Preserves the
    compute-in-reducer testability.
  - ✅ **menu_open (3/4, `185ab8f`)** — handler builds items from the
    layout slice + threads `msg.items`; arm reads them.
  - 🔒 **jobs_activate (4/4) — BLESSED, NOT threadable (`185ab8f`).**
    Its viewer-slice reads (route a routed/pty/info job to its tab)
    depend on the POST-cascade `currentGroup` the arm computes mid-flight
    — the handler can't precompute them. The one intrinsic root-reducer
    cross-slice read; documented at the arm + in PRINCIPLES.
  Routing reads (getFocus/componentForPanel/paneTypeOf/resolveTarget)
  stay in the arms throughout — the blessed chokepoint.
  - ❌ **B1 (`slice.lines` retire):** deferred — it's a finalizer-derived
    perf cache + the ref-equality transition-detect basis. Not a win.
  - ❌ **P5.2-4 (effect lifts):** deferred — purity-wall, "comparable in
    scope to P3"; the remainders are documented blessed exceptions.
- **D — chrome-widgets / mode hygiene.** ✅ `_suppressesChromeClicks`
  folded into the MODES `suppressChrome` column (`426c5f9`) — the last
  non-derived mode predicate, now single-sourced. ❌ The other half (a
  registry replacing the 4 inline overlay hit-tests) assessed as a
  likely non-win: the file-layout review judged those overlays
  legitimately bespoke (pane-anchored geometry, not reinvention). Left
  alone. **Theme D effectively done.**
- **E — independent.** ✅ **R1 (`51fc02d`)** — but NOT the specced
  "plugin-projection cache": profiling showed the projection is cheap
  (`getMergedActions` 0.25µs); the ~71µs cost on modelBundle /
  flatTabInfo / resolveTabKey (per render / per ]/[ keystroke) was
  `_mergedFor`'s **per-call `require('../panel/api')`** (relative-path
  fs resolution every call). Memoized the api ref → **~130× faster**
  (71µs → 0.5µs). The getGroupActions-cache attempt was reverted (wrong
  target). ❌ **config-status flatten skipped** — not used in either
  demo, feature-shaped restructure, low payoff. **Theme E done.**
- **🔎 FOLLOW-UP (new finding, beyond v0.6.4 scope):** the same per-call
  `require()` pattern is pervasive in `render/geometry.js` — 7×
  `require('../panel/route')` + others as INLINE requires on the
  per-frame render path. At ~70µs/require that's a real per-frame cost.
  A "memoize hot-path inline requires" sweep is the clear next perf win
  (same fix as R1; needs per-site hot/cycle verification).
