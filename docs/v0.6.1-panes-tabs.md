# v0.6.1 — panes as containers, tabs as content

> **Status: draft spec.** Not yet started. Successor to
> [v0.6-free-config.md](v0.6-free-config.md). User-facing reference
> updates will land in [LAYOUT.md](LAYOUT.md) when the arc closes.
> Live discipline rules remain [PRINCIPLES.md §12](PRINCIPLES.md).

## Goal

Generalize the v0.6 grid so that the unit of placement is a **pane**
(a rectangular slot in the layout), and the unit of content is a
**tab** (one of N renderable kinds — viewer, groups, files, history,
config-status, docker, actions, stats, …). A pane holds 1+ tabs; the
tab strip auto-hides at n=1 and renders only inside the panel's top
border (no extra row). Tabs can be moved between panes via drag,
keybind, or `:move-tab` cmdline. The singleton-detail assumption that
threads through v0.5+v0.6 retires: every pane can host any tab kind,
the slice store keys by tab id instead of Component name, and
`getFocus()` returns a tab id rather than a panel-type string.

## Non-goals

- **No workflow execution model.** v0.6.1 ships exactly one piece
  of forward-compat scaffolding: a `resolveTarget(intent, ctx) →
  tabId` chokepoint that replaces every `wrap('detail', …)` site.
  No role metadata, no channels, no fan-out, no named ports, no
  wiring blocks, no cycle detection. v0.7 will define inter-component
  access via real APIs each Component implements; producer/consumer
  identity emerges from those signatures rather than from advisory
  flags decorated onto specs today.
- **No YAML layout-file migration.** v0.6 configs do not parse under
  the v0.6.1 schema. The break is loud (parser error at load) and
  documented in CHANGELOG. No silent rewrite, no compat branch.
- **No multi-instance of the same panel kind in the same pane.**
  Two `files` tabs in different panes is fine; two `files` tabs in
  one pane is rejected at insert-time (collision in the per-pane
  active-tab selector with no UX win).
- **No per-tab hotkeys.** Pane hotkeys stay positional (1..6 / 7..9
  / `o` / `0`); tab switching uses `]`/`[`, click, or `[≡]` overlay.
- **No new global pool browser.** `w` stays the global panel-list
  overlay (placed + hidden across the whole layout). `[≡]` stays
  per-pane (this pane's tab list).
- **No retire of `[≡]` at n=1.** The auto-hide is for the strip
  itself, not the management trigger — but the trigger ALSO hides
  at n=1 since the overlay would offer no choices to switch to.
  Adding a tab is reachable via `w` or `:move-tab`.

## Today's architecture

v0.6 placements are flat: `arrange.leftPanels: panel[]` /
`arrange.rightPanels: panel[]` where each cell is a single panel,
referenced by id into `arrange.pool`. The slice store is keyed by
Component name (`panel/api.js:43` — `components = {}`); panel→Component
ownership is keyed by panel-type string (`leaves/route.js:26` —
`_panelOwner[panelType]`). `getFocus()` returns a panel-type string
(`leaves/route.js:85`); ~30 sites compare `getFocus() === 'detail'`.
Tabs exist only inside the detail panel — `slice.tab` is an int into
a flat list (Info | actionTabs | termTabs | contentTabs), the `[≡]`
overlay is detail-anchored (`overlay/tab-list.js:254` early-returns
for non-detail), and every navigator-side write to "the viewer" goes
through `wrap('detail', …)` or the singleton `setDetail(text)` helper
(`app/state.js:320`). Detail is special-cased throughout: the
half-view "last non-detail panel on the left" rule, the un-closable
invariant, the chrome injection gates. Two panes hosting the same
kind isn't expressible: `panelBounds[p.type]` would collide,
`getComponentSlice('files')` returns one shared slice, parser
invariants check `leftPanels.some(p => p.type === 'detail')`.

## Target architecture

### Slice model (id-keyed)

Two-layer registry:

- **Spec table** (`components[name]`, `panel/api.js:43`) — unchanged.
  One Component spec per name; `registerComponent` still rejects
  duplicate registration. Specs are stateless code.
- **Instance table** — new. `route._instances[tabId] = {tabId, spec,
  kind, slice}`. `route.instantiate(tabId, kind)` mints a slice via
  `spec.init()`; `route.dispose(tabId)` tears it down. `kind` is the
  panel-type string (`'detail' | 'groups' | 'files' | …`); `spec` is
  the resolved Component (looked up via the kind→spec map that
  `registerComponent`'s `panelTypes` already builds).

The `tabId` IS the instance key. Source: the existing pool entry id
(parser-minted at `parser/index.js#buildPlacedPanel`, threaded
through `state.js:113`). When the same pool entry is mounted twice
(future: ephemeral tabs, `:open` spawns), `synthIdFor` mints
`"<poolId>#2"`, `"<poolId>#3"`, … (mechanism already used for inline
pool synth at `parser/index.js:121`).

Layout remains the **chrome singleton**: `layout` is a fixed root
slot, not an instance. `getChromeSlice()` (or the existing
`getComponentSlice('layout')`) keeps working. Every other slice
lookup moves to `getInstanceSlice(tabId)`.

The wrap envelope at `panel/api.js:339` grows one optional field:

```
wrap(kind, msg)            // chrome singleton (kind='layout') or
                           //  resolves to focused instance of kind
wrap(kind, inst, msg)      // explicit instance address
```

Resolution rules:
- `kind === 'layout'` → chrome slice. `inst` ignored.
- `inst` present → `route._instances[inst]`. Asserts
  `instance.kind === kind` (catches stale ids).
- `inst` absent → focused instance of `kind` via `getFocus()`.
  Falls back to "the single instance of `kind` if exactly one
  exists" (the test/fixture path).

Broadcast fan-out (`dispatchMsg` for `refresh`/`hub`/`action` —
`panel/api.js:351`) iterates **instances**, not specs. Every live
slice reacts. The naive port that still iterates `components`
silently regresses — explicit invariant.

### Pane model

```
arrange.leftPanels:  Pane[]   // array name preserved to limit churn;
arrange.rightPanels: Pane[]   //  element shape changes

Pane = {
  id:          string,       // 'pane-l0', 'pane-r1' — slot identity
  column:      'left' | 'right',
  hotkey:      string,        // '1'..'6' / '7'..'9' / 'o' / '0'
  tabs:        Tab[],
  activeTabId: string | null, // null only when tabs.length === 0
  heightPct?:  number,        // lifted from per-panel
  collapsed:   boolean,
  // chrome cache (renderer-written, blessed exception):
  bounds?:     Rect,
  tabBounds?:  TabBound[],
}
```

Pane id is the SLOT identity — survives tab churn. Hotkey, height,
collapse all lift up from the old panel to the pane. `bounds` /
`tabBounds` are renderer-side cache; today's `panelBounds[p.type]`
becomes `pane.bounds` keyed by pane id, breaking the type-collision
risk.

Panes never auto-remove. Empty pane (tabs.length === 0) keeps its
slot, hotkey, height; body shows "Drop a panel here  (press w to
pick)" centered. Explicit pane removal is a separate gesture (drag
to pool, or `:close-pane`). Closing the last tab of a non-detail
pane is allowed; closing the last detail tab anywhere is refused
(detail-singleton invariant lifted from panel-level to tab-level).

### Tab model

```
Tab = {
  id:     string,    // pool id or '<poolId>#n'; INSTANCE key
  poolId: string,    // ref into arrange.pool (identity stays in pool)
}
```

Nothing else lives on the tab at the schema level. Per-tab body
state (viewer lines/scroll/cursor/search/select/contentTabs, files
cwd/items/showHidden, config-status branch/cache/expanded, docker
nav, groups list/expanded/tab — every today-slice-field) lives on
the **instance slice** under `route._instances[tab.id].slice`.

Pool stays per-panel — `arrange.pool[poolId] = {type, title, config}`
unchanged from v0.6. Pool entries are IDENTITY (what kind of panel,
configured how); tabs are PLACEMENTS (which pane currently hosts an
instance).

Single mutation Msg surface (all `wrap('layout', …)`):

```
pane_open_tab        { paneId, poolId, atIndex?    }
pane_close_tab       { tabId                       }
pane_move_tab        { tabId, toIndex              }  // within pane
pane_move_tab_to     { tabId, toPaneId, toIndex    }  // across panes
pane_set_active_tab  { paneId, tabId               }
```

`pool_show` / `pool_hide` become thin sugar:
`pool_show(poolId, paneId?) → pane_open_tab`,
`pool_hide(tabId) → pane_close_tab`. Three drag state machines
(in-grid pane drag, pool drag, tab-bar drag) collapse into
`pane_move_tab` / `pane_move_tab_to`. Validator
`validateTabMove(arrange, tabId, target)` is shared by all surfaces
— enforces column caps (6/3 panes), detail-singleton,
detail/actions-left-column refusal.

### Tab strip + chrome

Strip renders in the top border row, NOT a separate sub-row. Layout
when `tabs.length >= 2`:

```
╭─(h)─[Info]─Logs─Diff[x]─Status──────[X]─[_]─[≡]╮
│                                                │
│  <active tab's body — renderer chosen by tab>  │
```

`buildTabStrip(pane, tabs, activeIdx) → {titleMarkup, tabBounds}`
lives in `render/panel-widgets.js` (new helper, lifts from
`viewer.js#detailTitle`). Every panel renderer composes its title
through this helper. `tabBounds` writes to `pane.tabBounds` (the
v0.5-blessed render-side write, rekeyed from `panelBounds.detail.tabs`
to per-pane).

Auto-hide rules:
- **n=0**: empty-pane placeholder body; strip is absent; `[≡]`
  hidden (no tab list to show); pane keeps its top border + `[X]
  [_]` so close/collapse still work.
- **n=1**: strip absent (pane shows the single tab's title as plain
  border text — today's "Detail" literal path generalised); `[≡]`
  hidden; `[X][_]` present.
- **n≥2**: full strip; `[≡]` present.

`injectTopRowChrome` (`render/panel-widgets.js:85`) drops its
`p.type === 'detail'` early-return. `injectTabTrigger`
(`overlay/tab-list.js:254`) drops its parallel detail-only
early-return AND lifts into `render/panel-widgets.js` next to
`injectTopRowChrome` — its detail-anchored geometry
(`overlay/tab-list.js:53`, `TRIGGER_X_OFFSET=5`) becomes per-pane.

Chrome budget: a non-detail pane with all three glyphs needs ≥17
inner cells. `injectTopRowChrome` learns a `wantTrigger` branch +
revised min-w; precedence when starved (drop in order): `[≡]` →
inactive tab labels → `[X]` → `[_]`.

`tabListMode` (today a global flag at `runtime.js:76`) grows a
companion `tabListOwnerPaneId` field on the layout slice. Only one
overlay open at a time; opening B atomically closes A. Overlay
geometry (`overlay/tab-list.js:96`) reads owner's bounds, not
hardcoded detail.

`_flatTabs()` (`overlay/tab-list.js:65`) reads `pane.tabs` from the
arrange instead of calling detail-specific `getTabInfo()`. The
kind-tags catalog (`'action'`,`'term'`,`'content'`,…) becomes a
per-tab-kind capability struct exposed by the spec.

### Navigator routing

Every site that today writes the singleton detail panel routes
through one helper:

```
resolveTarget(intent, ctx) → tabId | null
```

Located in `leaves/route.js`. `intent` is a closed enum:
- `'viewer'`           — replace body / show info
- `'viewer_tab_add'`   — add a content tab
- `'terminal'`         — spawn ephemeral terminal tab

`ctx = { focusedTabId }` (no `sourceTabId` thread — that lands
when the v0.7 producer API ships). Resolution, purely kind-based:
focused viewer-kind tab if any; else `lastViewerTab` (new layout-
slice field, written in `focus_set` whenever focus lands on a
viewer-kind tab); else first viewer-kind tab in right-column
order; else any viewer-kind tab; else null (no-op). "viewer-kind"
means `instance.kind === 'detail'` (kind name TBD if rename to
`'viewer'` lands in this arc).

Call sites rewritten (replace hardcoded `'detail'` with
`resolveTarget('viewer', ctx)`):

| Site | File:line |
|------|-----------|
| `setDetail()` body | `app/state.js:320` |
| `run_selected → viewer_show_info` | `dispatch/actions.js:244` |
| `show_selected_info` effect | `dispatch/effects.js:136` |
| `groups` cascade `viewer_reset_chrome` ×2 | `panel/navigator/groups.js:178, 208` |
| `historyReplay → setDetail` | `panel/navigator/history.js:147` |
| `cfgStatusDiff → setDetail` | `panel/navigator/config-status.js:421` |
| `openFile → addContentTab` | `panel/navigator/files.js:495` (via `feature/open-file`) |
| `dockerExec → streamCommand`, `dockerShell → addEphemeralTab` | `panel/navigator/docker.js:306` |
| `viewer_add/update/remove/reorder_content_tab` ×5 | `panel/viewer/tabs.js:131` |

Wraps that target the focused detail directly (search/select/scroll
/key-on-viewer) stay as `wrap(focusedTabId, msg)` — those aren't
routed, they're owned by their pane.

### Inter-component routing chokepoint

The ONLY v0.6.1 scaffolding for the future workflow feature is the
`resolveTarget` chokepoint described above. Everything previously
floated as "advisory metadata" is dropped:

- No `role: producer | consumer | sink` field on Component specs.
- No `produces:` / `consumes:` shape declarations.
- No channel objects, no wiring config in YAML, no pubsub registry,
  no cycle detection.

Why: an advisory flag that participates only in a tie-breaker is
weak typing. When v0.7 introduces real inter-component access, each
Component will implement a concrete API (methods, hooks, or typed
channels — TBD) and producer/consumer identity will emerge from
those signatures. A flag added now would either be unused noise or
would harden into a contract that disagrees with the eventual API.

What v0.6.1 leaves as the v0.7 seam is exactly one function —
`resolveTarget(intent, ctx) → tabId` in `leaves/route.js`. v0.7
swaps its return from `tabId` to `tabId[]` for fan-out; call sites
iterate. No other plumbing.

`sourceTabId` is NOT threaded into `getInfo` at this stage — same
reasoning. When v0.7 defines the producer API, signature change
lands then.

## Decisions (resolved)

All seven design questions are locked. Captured here for traceability
during implementation.

1. **Empty pane policy → STAY.** Pane keeps its slot, hotkey, and
   height after the last tab closes. Body shows the Drop Zone
   placeholder. Explicit `:close-pane` (or drag-to-pool) is the
   removal gesture. (Rationale: auto-remove silently renumbers
   hotkeys; staying is predictable.)

2. **config-status internal tabs → FLATTEN.** Three peer pool kinds
   (`config-status:tree`, `config-status:tracked-tree`,
   `config-status:tracked-flat`) sharing one backing diff cache.
   Outer `]`/`[` cycles them when mounted in the same pane. v0.6.0
   muscle memory (`]`/`[` inside the panel) breaks intentionally —
   the new layout language makes the comparison case (all three
   side-by-side) expressible for the first time.

3. **Tab drag mid-zone → ADD-AS-TAB.** Dragging a tab into another
   pane's mid-zone joins that pane as a new tab and becomes active.
   No REPLACE-ACTIVE-TAB semantic (would have ambiguous "where does
   the displaced tab go" answer).

4. **Cross-pane keybinds → DEFERRED.** v0.6.1 ships tab-move via
   drag (mouse) and `:move-tab` (cmdline) only. No `{`/`}`,
   `g m`, `g M`. Keyboard bindings revisit after the feature ships
   and use-patterns are observed.

5. **Stats `select_from` → first-matching-kind.** Stays the
   v0.6 behavior under the generalized model: stats binds to the
   first navigator instance of the named kind it finds in arrange
   order. Explicit tab-id binding waits for v0.7.

6. **Role taxonomy → DROPPED ENTIRELY.** No `role: producer |
   consumer | sink` field. v0.7 will define real Component APIs
   for inter-component access; producer/consumer identity emerges
   from those signatures rather than from advisory metadata
   decorated today. (Rationale: a flag that participates only in
   a tie-breaker is weak typing and risks disagreeing with the
   eventual real API.)

7. **Schema break → docs only, no migration tool.** Ship
   `docs/v0.6.1-migrate.md` with hand-conversion examples; parser
   error message gains a hint pointing at it. v0.6.0 layouts
   reject at load; users hand-rewrite (or paste their config into
   an AI that has the migration doc in context).

## Risks

1. **Type-as-identity assumption is pervasive.** `slice.focus`,
   `panelHeights`, `panelBounds`, `rendererFor(p.type)`,
   `allPanels().find(p.type===X)`, `selectedIdx`, `clampSelected`,
   `_reassignHotkeys`'s hardcoded `actions→'0'` / `detail→'o'`,
   `halfLeftPanel`, the validateTarget detail/actions left-column
   rules — all assume one panel of each type. The Phase 7 mass
   flip is the structural inflection; a missed site quietly routes
   focus to the wrong pane or shares `panelBounds` between two
   same-kind tabs.

2. **Broadcast fan-out regression.** Phase 3 changes
   `refresh`/`hub`/`action` from iterating specs to iterating
   instances. A naive port that iterates specs keeps the slice on
   the spec instead of per-instance — silent regression where
   `docker` polls update one slice and the second docker tab shows
   stale state. Explicit instance-loop assertion in test-component.

3. **`tabListMode` two-overlay race.** Today's single global flag
   assumes one overlay at a time, anchored to detail. The owner
   field (`tabListOwnerPaneId`) must atomically close A when
   opening B; the residue-blank logic in
   `overlay/tab-list.js#_lastPanelH/_lastTop` is module-private
   and per-instance state — if not also keyed, two opens in
   sequence corrupt each other.

4. **`config-status` flatten regresses muscle memory.** Users who
   leaned on `]`/`[` inside config-status get a different gesture
   (the outer pane's tab cycle). Mitigation: flatten happens with
   CHANGELOG callout; the three sub-modes become discoverable as
   three separate `config-status` panel kinds.

5. **Schema break with no migration.** Users' v0.6.0 YAMLs reject
   at load with a parser error pointing at the layout block. If
   the error is unhelpful ("expected mapping with `tabs` key"),
   users hit a wall on first upgrade. Risk-mitigation: parser
   error message gets a "v0.6.0 layout detected — see docs/v0.6.1
   for new shape" hint at the offending line. One line of code,
   one risk averted.
