# Replay — recording and reconstructing a session (v0.6.6)

> **Status:** implemented. This is the in-repo reference for the replay feature
> built on top of the v0.6.6 purity arc. The design brief that preceded it is
> [docs/v0.6.6-replay-readiness.md](v0.6.6-replay-readiness.md); the foreign-component
> side-channel it uses is [docs/foreign-components.md](foreign-components.md) §Replay.

## What it is

Replay records the ordered **message stream** that drives a session and re-applies
it through the **pure reducers** — without re-running effects against live external
state — to reconstruct an identical model and frame. The embedded terminal (the one
off-model island, `#D14`) is reconstructed too, from its recorded output byte stream.
Long sessions are **resumable**: periodic checkpoints let replay seek instead of
folding from the start.

It is both a debugging/repro tool (attach a session log to a bug report;
`--record-load` it) and the empirical proof of the purity arc's central claim:

- `model = fold(reducer, MsgLog)` — verified by `js/test/test-replay.js`
- `terminalGrid = fold(write, ByteLog)` — verified by the same test

## Architecture — one write-ahead log, one global sequence, one file

A single monotonic `seq` stamps every recorded entry, so the interleaved stream *is*
the global order (a `set_arrange` that places a terminal precedes that terminal's
first output). The log is one append-only **JSONL** file (`io/session-log.js`),
default disabled, enabled by `enable(true)` or the `LAZYTUI_REPLAY_LOG` env var (which
also attaches a live-append stream). Three entry kinds:

| kind | shape | meaning |
|---|---|---|
| `msg` | `{seq,t,kind:"msg",lane,msg}` / `{…,lane:"key",key,keySeq}` | a Msg entering the loop (the model WAL) |
| `term` | `{seq,t,kind:"term",id,ev,…}` (`ev`: spawn/resize/out/exit; `out` carries `d`) | a terminal change (off-model side-channel) |
| `checkpoint` | `{seq,t,kind:"checkpoint",state,grids}` | a resumable snapshot of `{model,slices}` + terminal grids |

The recorder hooks the **three** loop entry points in `dispatch/runtime/loop.js`
(`applyMsg` → root, `dispatchMsg` → comp, `dispatchKeyToFocused` → key). An audit
confirmed every state write funnels through these (`setModel` only in `applyMsg`;
`setInstanceSlice` only in the loop), so recording there is complete. The terminal
feeds `term` entries through an injected recorder hook (`io/terminal.setSessionRecorder`,
wired at boot) so `io/terminal` stays a leaf.

## Reconstruction — the two rules

`dispatch/runtime/replay.js` brackets a fold with `setReplaying(true/false)`. Under
replay:

1. **Skip ALL effects.** `effects.runEffects` no-ops entirely. No effect writes
   model/slice state directly — each either re-dispatches a Msg (itself recorded) or
   does IO whose *result* returns as a recorded Msg — so re-running any effect would
   double-apply a Msg already in the log.
2. **The finalizer does instance-mint only.** `finalize.finalizeDispatch` keeps the
   per-pane instance reconcile (deterministic slice creation, regenerated from the
   recorded layout Msgs) and skips the scroll-clamp (its `set_scroll` is in the log),
   the subscription reconcile (live IO), and the PTY ensure/resize (live IO).

Recording is disabled during a fold, so replayed Msgs are not re-recorded.

## The terminal — a port + a text grid snapshot

The emulator is isolated behind a **screen port**, `io/term-screen.js` — the single
module that imports `@xterm/headless`. `io/terminal.js`, `render/paint.js`, and the
finalizer reach the screen only through it (`createScreen`/`write`/`resize`/`size`/
`readViewport`/`scroll*`/`scrollInfo`/`mouseMode`/`serialize`/`restore`). Swapping the
emulator means reimplementing this one adapter.

The grid **snapshot** is plain text. lazytui renders the embedded terminal as
monochrome text — `render/paint.js` reads each row via `translateToString` (characters
only, no attributes) — so capturing the buffer's text rows + viewport position is
faithful to what is actually painted, with **no SGR serializer and no extra dependency**.
The recorded byte stream remains the exact source for continued output.

Portability note: the *byte stream* (the `term`/`out` diff log) is emulator-agnostic —
re-feeding it reconstructs the grid on any emulator. The grid snapshot is an
emulator-specific optimization that lives entirely inside the port.

## Checkpoints

`replay.checkpointNow()` records a checkpoint: `{model, slices}` (Set-encoded to plain
JSON — see below) plus each live terminal's text grid (`terminal.snapshotAllSessions()`).
`replay.replayTo(log, targetSeq, opts)` seeks to the latest checkpoint at or before the
target, restores it, and folds the remaining entries forward. Correctness comes from the
WAL; checkpoints are the speed/resume optimization *and* a built-in assertion — folding
the WAL from the start to a checkpoint's seq must reproduce that checkpoint's stored state
(`js/test/test-replay-checkpoint.js`).

**Set-aware codec.** The replayable state is almost plain JSON, but nav slices hold
`Set`s (`multiSel` on every navigator; `expanded` on groups) — the only non-plain type
in state. `sessionLog.encodeJson/decodeJson` map `Set ↔ {__set__:[…]}` so checkpoints
serialize; in-process checkpoints clone with `structuredClone`. The Msg stream itself
carries no Sets.

**Auto-cadence.** While recording, the finalizer writes a checkpoint once enough has
accrued since the last one — `replay.maybeCheckpoint` consults `sessionLog.checkpointDue`
once per outermost dispatch (a gated no-op otherwise, never under replay). The metric is
**bytes-primary** (default 256 KB of recorded WAL): the dominant fold cost on seek is
re-feeding recorded *terminal output*, which scales with bytes, not entry count — so a
checkpoint every N entries would under-checkpoint a heavy `tail -f` (N huge chunks = MB
to re-feed) and over-checkpoint pure navigation. A high **entry-count ceiling** (5000)
rides along as a safety cap (compaction + pathological Msg spam). So a long recording
stays fast to seek without manual checkpoints. record-save (flag + verb) and
`LAZYTUI_REPLAY_LOG` turn the cadence on; record-stop clears it. **Mint-on-restore** (`restoreState` recreates the per-pane instance set from
the restored arrange) lets these checkpoints reconstruct even from a bare boot, so
`--record-load`/`:record-load` seek to the nearest checkpoint rather than folding from
the start.

## Triggers — record / load / stop

Recording and reconstruction each have a flag (boot-time) and a cmdline verb
(mid-session). The file is optional everywhere (default: `LAZYTUI_REPLAY_LOG`
env, else `lazytui-session.jsonl`; `:record-load` defaults to the file the session
is currently recording to).

| | Record | Replay |
|---|---|---|
| **flag** | `--record-save <file>` — stream from boot (the full `set_config`-onward WAL) | `--record-load <file>` — launch the TUI straight into **interactive replay** (the scrubber pane); no config needed |
| **cmdline** | `:record-save [file]` — checkpoint the full state, then stream follow-ups (self-contained from mid-session) | `:record-load [file]` — enter interactive replay (the scrubber pane — see below) |
| **cmdline** | `:record-stop` — stop recording | |
| **flag** | | `--record-print <file> [--seq N]` — headless: reconstruct + print the frame, then exit (scripts / CI / a bug-report frame) |

The runtime side is `dispatch/runtime/record-control.js` (save/load/stop), reached
by the cmdline verbs through the effect host (`recordSave`/`recordLoad`/`recordStop`).
Both replay entry points reach the same controller: `--record-load` boots the TUI's
runtime scaffolding (via `app/replay-cli._installRuntime` — host wiring + effect handlers
+ the built-in Component set from `app/components.js` + the reconcilers) then enters
interactive replay; `:record-load` enters it from a running session (snapshotting the live
session aside, restored on `q`/`esc`). `--record-print` is the only headless path — it
folds the WAL via `app/replay-cli.runReplay` and prints the frame (TTY: raw positioned;
pipe: decoded text grid). Tests: `js/test/test-replay-cli.js` reconstructs cross-process in
a fresh `node tui.js --record-print` subprocess; `js/test/test-replay-control.js` covers
save/stop + the self-contained file; `js/test/test-replay-control-pane.js` the scrubber.

## Interactive replay-control pane

`:record-load` enters **interactive replay**: it snapshots the live session aside, loads
the WAL, reconstructs the recorded session, and opens a float **control pane** listing the
checkpoints ↔ timestamps. The panels underneath show the reconstructed frame at the current
position; the pane shows play state / speed / mode / position. Keys: `j/k` move the checkpoint
cursor (seek), `space` play/pause, `b` reverse play, `+/-` speed (slow ⅛× … 16× fast), `[`/`]` step one
entry, `g/G` first/last, `m` toggle playback mode, `i` cycle the idle-gap cap, `p` cycle the
pane view, `q`/`esc` exit (restore the live session).

The pane has **three view states** (`p` cycles full → mini → hidden) so it needn't cover the
reconstructed view: **full** is the checkpoint list + status + legend (for checkpoint
navigation); **mini** is a compact bottom bar — play state, seq, timestamp, and a progress bar
only — so playback stays watchable with the content visible; **hidden** draws nothing (only `p`
to bring it back and `q`/`esc` to exit still act). Playback controls work the same in full and
mini — they differ only in what the pane draws. (A true reserved-row split isn't used: the
reconstructed frame renders at its *recorded* size, so shrinking the viewport would clip it;
the bar overdraws the footer rows instead, never the content area.)

The controller (`dispatch/runtime/replay-control.js`) is **module-held — outside the
reconstructed model** — because each reconstruction (`replayTo`/`advance`/`reverseTo`)
replaces the whole model; the controller's state, active flag, and play timer would be
clobbered if they lived in it. Input early-routes to it (`dispatch/control/dispatch.handleKey`)
and render reads it via an injected seam (`render/paint.setReplaySource`), so the reconstructed
model stays pure and there's no render→dispatch import. It's a *meta-tool over* the replayed
system — the same spirit as the foreign-component exception.

### Playback engine (robust-playback arc)

The play loop separates the three concerns a media player keeps orthogonal — a jittery first
cut conflated them (drift, an O(n²) re-fold sawtooth, and freeze-on-gap/leap-on-burst):

- **Presentation clock** — playback position is a virtual clock *anchored to a monotonic real
  source* (`performance.now()`): each frame the position is recomputed from elapsed real time
  (`anchorClock + dir*scale*(now-anchor)`), so it is drift-free and self-correcting (a slow or
  late frame never accumulates speed error). Speed lives entirely in the clock — a single
  multiplier spanning **⅛× slow-motion to 16× fast-forward** (`+/-`), so sub-1× *slow mode*
  (watch a fast burst frame-by-frame, or pair with even mode for a slow logic walk-through)
  needs no special path. The scheduler wakes at a steady ~30 fps cadence independent of the
  speed, and **skips frames that don't change the index** (no re-fold/re-render churn during
  idle dwell or slow-mo dwell). The clock/timer are injectable for
  deterministic tests (`replayControl._setClock`).
- **Timeline** (`leaves/replay/timeline.js`, a pure leaf) maps the clock to a WAL index in two
  modes (toggle with `m`): **realtime** runs the clock over an *effective-time* axis —
  cumulative inter-entry deltas with each idle gap **capped** (cycle the cap with `i`:
  250 ms…2 s…∞) so dead air never freezes playback (cf. asciinema's `idle_time_limit`); **even**
  runs the clock over entry-index at a fixed entries/sec, for reviewing the logic regardless of
  recorded pacing. The effective axis is built from `max(0, Δt)` so it stays monotonic even
  across a backward wall-clock (`Date.now`) glitch.
- **Forward** — `replay.advance(log, from, to)` applies only the new entries (terminal bytes
  append to the persistent emulator; proven equivalent to a checkpoint restore + fold), so play
  is O(n) over the run, not the re-fold-from-checkpoint sawtooth.
- **Reverse** — reducers aren't invertible, so reverse can't step incrementally. Instead a
  bounded per-interval **model base ladder** (`REVERSE_BASES` cheap `{model, slices}` snapshots,
  no terminal grids) shortcuts the model fold, and the terminal is anchored on the interval's
  WAL **checkpoint grids** (drained at record time) + fed forward — both in one ordered pass
  (`replay.reverseTo`), giving flat per-frame cost. A snapshot **ring** was considered and
  rejected: it would have to snapshot full terminal scrollback per slot (multi-MB) to buy a
  bound the 256 KB checkpoint cadence already provides.

## Spawned terminals + tmux

`type:spawn` normally opens a real `tmux new-window` under `$TMUX` (a better home for
a long interactive session than an in-process tab). But a tmux window runs *outside*
the app — its output never enters the WAL and is tied to that host — so it can't
replay. **While recording, spawns are forced onto the embedded PTY**
(`action-runner._spawnUsesTmux` = under `$TMUX` AND not recording), whose output *is*
captured as the byte-stream side-channel. So a recorded session stays fully replayable
and **position-independent**: it reconstructs the spawn's output on any host, with or
without tmux. (During the fold itself tmux can't trigger anyway — effects are skipped,
and `$TMUX` is read only in the spawn effect.)

- `test-replay.js` — `model = fold(reducer, MsgLog)` (model + slices + frame) and `terminalGrid = fold(write, ByteLog)` + grid-snapshot round-trip.
- `test-replay-checkpoint.js` — Set codec round-trip; full fold == checkpoint seek; a checkpoint stores the fold-to-its-seq state.
- `test-replay-cli.js` — the WAL file round-trips and reconstructs cross-process.
- `test-replay-control.js` — record-save (checkpoint-seeded self-contained file, repeated-save skip) / record-stop, and the file reconstructs from a bare registry.
- `test-replay-cadence.js` — auto-cadence writes periodic checkpoints while recording; cadence 0 disables it.
- `test-replay-tmux.js` — `type:spawn` uses tmux only under `$TMUX` and not recording (forced embedded while recording, so spawns are captured).
- `test-replay-control-pane.js` — the scrubber: enter builds the checkpoint list + reconstructs the end; seek reconstructs the exact checkpoint state (backward + forward); step + play/pause; the overlay renders; exit restores the live session. Plus the clock-driven playback engine via an injected clock: forward advance, sub-entry no-op skip (no render), ratio re-anchor, end-stop, reverse via the ladder, and mode/idle-cap position preservation.
- `test-replay-timeline.js` — the pure timeline leaf: idle-gap capping, monotonic effective-time under a backward-clock glitch, `idxForClock`/`clockForIdx` round-trips (realtime + even).
- `test-replay-advance.js` — incremental forward fold: `advance(0→k)` ≡ `replayEntries`/`replayTo` full-fold for the model, every slice, and the terminal grid.
- `test-replay-reverse.js` — the reverse base ladder: `reverseTo` from a checkpoint-aligned and an intermediate base ≡ `replayTo` at the same index (model + slices + grid).

## Accepted caveats and follow-ups
- **History timestamps** render in the replay machine's local time zone
  (`panel/navigator/history.js` `fmtTime`). The timestamp itself is reconstructed exactly;
  only the human-formatted string varies cross-machine, and history is an overlay rarely in
  the compared frame. Local time is the correct UX, so it is kept; same-machine replay is
  byte-identical.
- **`cfgStatusDiff`** runs its git diff off-tick and ends in a recorded `viewer_set_content`
  Msg, so replay reconstructs its content from the log (already replay-safe).
- **Full-screen terminal apps** (absolute cursor addressing) restored from a *grid* snapshot
  and then fed further output may diverge in edge cases; the byte-stream WAL re-feed (from
  spawn) is the exact fallback.
