# v0.6.6 — Replay-readiness assessment (design brief for the next release)

**Status:** assessment only — nothing here is a bug in v0.6.6; the app is correct
today. This is the design brief for landing a **replay** feature in a future
release, produced by a fresh *code-is-truth* (ignore comments/docs/memory) 5-angle
adversarial review on 2026-06-25.

## The bar

Replay = record the ordered **Msg stream** and re-apply it through the **pure
reducers**, **without re-running effects** against live external state, and
reconstruct an identical model + frame — *except* the accepted terminal island
(#D14). This is stricter than `frame = f(model)`: it also demands `model =
fold(reducer, MsgLog)`, with every source of non-determinism **captured into a Msg
payload** so the recorded log is self-contained.

## What is replay-ready (verified against code)

The hard part is done — the TEA core satisfies the precondition for replay:

- **Reducers are pure & deterministic** of `(state, msg)` — the root reducer
  (`dispatch/update/reducer.js`), all 9 modal machines (`dispatch/update/modal/*`),
  and every Component `update(msg, slice)`. No wall-clock / random / `getModel()` /
  live external / module-mutable reads inside any reducer arm. `augmentMsg` is the
  impure shell: it reads the model and **stamps facts onto the Msg** (never mutates
  it), so the reducer never reads live state. (305 immutability assertions
  corroborate.)
- **Single-writer holds strictly.** `setModel` is called only by `applyMsg`
  (`dispatch/runtime/loop.js`); slices are written only by a Component's own
  `update` return; **FIX-2's `innerH` retirement is real** — `augmentMsg` stamps
  `msg.innerH` and the viewer's own reducer commits it, no finalizer write; no
  in-place model/slice mutation anywhere.
- **Everything that reaches the model does so through a Msg.** The off-model
  discrete stores (jobs/diag/history) and the continuous hub series are mirrored
  into `model.{jobs,diagLog,history,metrics}` via `*_synced` Msgs (whole-snapshot,
  self-contained payloads). Render reads the model, not the stores.
- **Module-local state is overwhelmingly pure caches** (memos keyed on immutable
  refs: `_layoutMemo`, route `_rtMemo`/`_rvpMemo`, search `_matchMemo`, selector
  slots, plugin-guard `_memo`) or effect-side plumbing. The render **diff caches**
  (`paint.js` `_frame`, overlay `_lastPanelH`) affect only the *emitted ANSI byte
  delta*, not the logical frame (`painter.composeRows` is pure of the model);
  `paint.forceRepaint()` resets them.

## The one accepted island — #D14 terminal / xterm (NOT replayable, by design)

`io/terminal.js` (node-pty + `@xterm/headless`) is the explicitly non-TEA region:

- `sessions` is an off-model store of `{ pty, xterm, … }`.
- `p.onData(data => xterm.write(data, () => _renderHook()))` — the PTY's stdout
  callback writes the xterm screen buffer **and triggers a repaint directly via the
  injected `_renderHook`, bypassing the Msg loop entirely**.
- Render reads the buffer **live**: `paint.js` `getSession(id)` / `sessionScrollInfo(id)`,
  `footer.js` `getSession(id).exited`, and dims via `io/term.cols()/rows()`.

The model holds the PTY **lifecycle** (which tab is a terminal, the session id —
replayable); the screen **contents** are off-model and produced by a child process,
so **replay cannot reproduce terminal output**. This is the single sanctioned
exception to `frame = f(model)` and to replay.

**Replay-harness handling:** do NOT spawn/reconcile PTYs during replay; accept that
terminal-pane regions replay blank; feed `io/term` dims from `model.dims` /
recorded `term_resized` rather than the live TTY.

## The gap — recorder altitude (the gating decision)

The only recorder today is `io/event-log.js`, and it records **inputs**, not Msgs:
`key` / `mouse` / `refresh` / `publish` / `action` (its header: *"inputs to the
system, not outputs … NOT recorded: internal state mutations"*). Its documented
replay plan re-injects those inputs into `dispatch.handleKey` / `handleMouse` —
i.e. **re-runs the entire live pipeline**. That re-runs:

- the **wall clock** — `clock_tick` re-stamps `Date.now()` (`app/state.js`), and the
  store timestamps re-stamp inside their stores (`feature/jobs.js:50,86`,
  `feature/history.js:72,115`, `io/diag-log.js:52`, `config-status.js:142,492`);
- the **metrics-mirror 250 ms trailing throttle** (`app/state.js` `metrics-mirror`) —
  *which* publishes coalesce into a sample becomes wall-clock-dependent, so the
  stats-graph payload diverges;
- the **finalizer's inline IO** — every dispatch runs `reconcileSubscriptions` + PTY
  `ensureSession`, which would spawn `docker events`, arm timers, add the resize
  listener, and spawn PTYs *during replay*. There is no `replayMode` seam.

### The insight that resolves most of it

**Record the Msg stream (at `applyMsg`/`dispatchMsg` entry), not raw inputs.** The
`*_synced` / `metrics_synced` / `clock_tick` Msgs carry whole snapshots **with their
timestamps baked into the payload**. Re-applying them reconstructs the model
exactly — so the in-store `Date.now()` stamps and the throttle timing become
**non-issues** (their outputs are captured, not re-derived). Under a Msg-stream
recorder, canonical TEA replay = *re-apply Msgs + skip effects* (every async effect
already folds its result back as a dispatched, recordable Msg — verified:
`dockerFetch→dockerResult`, `cfgStatusCompute→cfgStatusResult`, `streamCommand→
viewer_append*`, `jobs_route→jobs_routed`).

## What landing replay requires (3 items — next release, not a v0.6.6 patch)

1. **A Msg-stream recorder.** Record the exact Msg objects entering
   `applyMsg`/`dispatchMsg` (including effect-result Msgs and Sub-injected
   `clock_tick`/`*_synced`/`metrics_synced`/`dockerPoll`/`term_resized`). The current
   input-level `event-log` is the wrong altitude to drive "re-apply Msgs."
2. **A replay-mode IO suppression gate.** In `dispatch/runtime/finalize.js`,
   no-op the **subscription reconcile** and the **PTY ensure/resize** under replay.
   The seams already exist — `setSubscriptionReconciler` / `setInstanceReconciler`
   are injected — so a "suppress-IO" predicate plugs in cleanly.
3. **Skip effects on replay.** State reaches the model only via Msgs (single-writer,
   verified), so re-applying the recorded Msgs reconstructs state without running
   effects. Effects that did IO had their results return as their own logged Msgs.

## Narrow residuals (mostly moot under Msg-stream replay)

- **`cfgStatusDiff` is single-phase** (`panel/navigator/config-status.js`): it runs
  `git show/diff` *inside* the effect and writes the viewer there, unlike the
  two-phase `cfgStatusCompute → cfgStatusResult` Msg pattern. Worth making two-phase
  for consistency regardless of replay; it does end in a `viewer_set_content` Msg, so
  a Msg-stream recorder captures the content.
- **Timezone-local history formatting** (`panel/navigator/history.js` `new
  Date(ms).getHours()…`) — the *timestamp* is captured, but the rendered string
  depends on the replay machine's TZ. Format from a fixed zone (or carry the string)
  if byte-identical history rows matter.
- **Double-click classification** (`dispatch/control/input.js` `_classifyPress` uses
  `Date.now()` + prior-click state) — upstream of the Msg stream; only a *raw-input*
  replayer would re-run it. Under a Msg-stream recorder the resolved gesture is
  already in the dispatched Msg.

## Verdict

Nothing here is a bug in v0.6.6. The purity arc landed the hard part — pure
reducers, strict single-writer, non-determinism captured into Msgs, exactly one
off-model island (#D14 terminal). Replay is **not landable as-is**, but the
remaining work is a well-scoped next-release feature: a Msg-stream recorder + a
replay-mode IO gate + skip-effects-on-replay. This document is its starting point.
