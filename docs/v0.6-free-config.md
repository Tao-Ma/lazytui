# v0.6 — free-config mode + the panel pool

> **Status: shipped (2026-06-01).** Tip `4bdd45b` on `v0.6`. This doc
> captures the architectural decisions; the user-facing reference is
> [LAYOUT.md](LAYOUT.md), the per-feature changelog is the v0.6 block
> in [../CHANGELOG.md](../CHANGELOG.md), and the live discipline rules
> are [PRINCIPLES.md §12](PRINCIPLES.md).

## What v0.6 set out to do

v0.5 closed with a clean TEA architecture (per-Component slices, single-
writer per layer, the global `S` retired). What it DIDN'T close was
the layout editor's UX:

- The v0.5 "design mode" was a CLI-flag-gated toggle (`--design`) that
  let you drag-reorder panels, but offered no way to bring panels in
  or out of the grid — every panel declared in YAML was always placed,
  and every placed panel was always rendered.
- The fixed grid meant configs had a hard ceiling on how many panel
  definitions fit usefully. Users wanting "five action panels but
  only showing two at a time" had no way to express that.
- Mouse interactions were limited to drag-reorder; clicks for hide /
  switch / expand didn't exist.

v0.6's arc was: lift the layout editor from "a developer mode" to a
first-class user feature, add a pool concept that lets configs declare
more panels than the grid shows at once, and give every panel real
mouse chrome.

## The model — pool vs grid

YAML grew a top-level `panels:` block — the **pool**:

```yaml
panels:                 # the pool — all declared panel definitions
  docker: { type: docker }
  logs:   { type: tail, file: /var/log/syslog }
  notes:  { type: viewer, title: Notes }   # hidden by default
  actions: { type: actions }
  detail:  { type: detail }

layout:                 # the grid — id-refs into the pool
  left:  { panels: [docker, logs] }
  right:
    panels:
      - actions
      - { id: detail, height: 60% }
```

Two block separation, on purpose:
- `panels:` is the catalog. Each entry declares a panel by id + type +
  per-panel config. Entries can be PLACED in the grid or HIDDEN (still
  configured, not currently rendered, reachable via `:show <id>` or
  the panel-list overlay).
- `layout:` is the grid arrangement. References pool entries by id.
  Cells can carry layout overrides (`height: 60%`) without touching
  the pool definition.

**Legacy compat**: configs with inline panel definitions
(`{ type: docker }` cells) continue to parse and round-trip unchanged.
The pool is synthesized implicitly at load time when the user has no
top-level `panels:` block. `:save-layout` only writes the new
`panels:` block when the legacy inline form can't express the state
(any hidden entries, or a user-declared pool already exists).

## Where the state lives

The layout slice's `arrange` field carries:
- `arrange.pool` — id → entry map. The full catalog.
- `arrange.leftPanels` / `arrange.rightPanels` — the placed cells. Each
  carries `{ id, type, title, hotkey, column, ...config }`.
- Derivations (`placedIds`, `hiddenIds`, `panelListItems`, etc.) live
  in `js/leaves/pool.js` — pure transforms over `arrange`, dependency-
  free.

The pool / grid invariant: every placed panel's id resolves in the
pool. `pool.orphanPlacements(arrange)` is the explicit check; the
parser enforces at load, the reducer enforces on mutations.

## Free-config mode

The v0.5 "design mode" renames to **`freeConfigMode`** (mechanical
rename, behavior preserved). Three entry points:

- `:free-config` cmdline verb (or `:design` as the v0.5 alias).
- `--design` CLI flag — auto-enters after the first paint.
- Menu entry (the `:` cmdline picks it up).

All paths funnel through one `design_enter` Msg to the layout
Component. The reducer arm guards `viewMode !== 'normal'` (refuses
entry from half/full — see "view-mode guards" below).

### Freeze gate

While `freeConfigMode` is on, `panel/api.js#dispatchMsg` drops:
- Broadcast Msgs (`refresh`, `hub`, `action`) — Components stop
  polling.
- Wrapped Msgs targeting non-`layout` Components — Components keep
  rendering their last snapshot.

Layout-wrapped Msgs (`pool_*`, `design_*`, `view_*`, `focus_set`,
`set_arrange`) pass through, since free-config IS the mode that
drives those. Narrow exception: `viewer_reorder_content_tab` passes
too, because the tab-reorder drag gesture is a free-config-shape
change (visible tab order) implemented on detail's slice rather than
layout's. Same justification class as `pool_hide`/`pool_show`.

The freeze gate is what lets free-config feel like a stable canvas
under drag / resize / pool mutations — the user isn't seeing
docker-events repaint flicker while they're trying to drop a panel.

### Save / restore

- `:save-layout` writes the current arrangement to YAML. Format
  choice: legacy inline if it can express the state; new `panels:` +
  `layout:` blocks otherwise.
- `:restore-layout` reverts to the YAML on disk + clears the
  free-config undo/redo stacks.
- `slice.dirty` tracks pending changes; the footer surfaces an
  "unsaved (:save-layout)" hint in non-modal mode.

## View-mode guards

Free-config's drag/resize gestures need a fully-visible grid; half
and full view modes hide cells. The combo "free-config + non-normal
view" is structurally broken (pool overlay paints over half-view
content, drag targets reference invisible cells, etc.).

Two hard blocks:
- `design_enter` from half/full refuses entry.
  `slice.design.notice` carries the reason
  (`free-config requires normal view ([ to return)`); no `mode_set`
  Cmd emitted.
- `view_expand` / `view_shrink` (the user-input `[`/`]` keys) refuse
  while `freeConfigMode` is on. Notice:
  `exit free-config (q) to change view mode`.

Programmatic view changes (`view_set` from cmdline / pty-lifecycle,
`view_drop_full_to_normal` from terminal exit) stay unguarded — they
fire from system events, not user navigation.

**Notice lifecycle**: continuous-motion Msgs (drag-in-flight) preserve
the notice; a Msg that would re-assert the same notice preserves
slice identity (no churn on repeated blocked attempts); everything
else implicitly clears the notice. See `panel/layout.js#update` for
the precise rule.

## UI surface — chrome + drag preview + tab list

Three big additions on top of the v0.5 panel grid:

### Chrome glyphs

Each placed panel's top-border row hosts up to four small interactive
icons. See [LAYOUT.md § Panel chrome glyphs](LAYOUT.md#panel-chrome-glyphs)
for the full reference. Architecturally:

- `injectTopRowChrome` (`render/panel-widgets.js`) rewrites the row
  before `paintColumns` joins outputs — glyphs ride into the same
  syscall as the row's content, eliminating the flicker class where
  v0.5's paint-on-top approach let `─` show through on every panel
  repaint.
- `injectTabTrigger` (`overlay/tab-list.js`) is the parallel for
  detail's `[≡]`. Inserted after `(o)` (was a replacement in v0.6
  early arc; reverted to "after" so both stay visible).
- Hit-tests are pure derivations over `panelBounds` (the view-output
  cache layout.js writes during paint). All glyphs suppress during
  any drag in flight — drag affordances own the screen.

### Drag preview

The original v0.6 pool-drag affordance painted a seam-bar overlay or
cell-frame to indicate where the drop would land. The drag-preview
arc replaces both with **live layout reshuffling**:

- Both pool drag and in-grid drag use a unified 3-zone hit-test per
  cell: top third = insert before, middle third = swap (in-grid) or
  replace (pool), bottom third = insert after.
- The motion handler computes the would-be-after-release arrange when
  the cursor crosses into a new zone, stashes it on
  `slice.design.drag.previewArrange`.
- `render/layout.js#render` swaps `slice.arrange` for the preview
  during the paint pass, restoring both arrange AND panelBounds
  afterward so the next motion event's hit-test reads stable
  original-layout bounds (avoids feedback flicker).

Detail-at-end clamp is annotated in the footer when fired
(`→ insert at right:N (clamped — detail stays at end)`). Self-swap
shows `(no-op — release to cancel)` in dim — was a bright yellow
fake-action before.

See [LAYOUT.md § Resizing — drag (mouse)](LAYOUT.md#resizing--drag-mouse)
for the user-facing reference; the drag-preview arc lived inside the
existing pool-drag Msg flow (`pool_drag_start` / `pool_drag_motion` /
`pool_drag_release`) — no new modes, just enriched state.

### Tab-list overlay

Detail's `[≡]` opens a centered tab switcher listing every tab in the
detail panel (Info, action tabs, terminal tabs, content tabs). State
lives on `slice.tabList` (`{open, cursor, scroll}`) inside detail's
slice. New mode flag `tabListMode` — chain + overlay, NOT modal (the
overlay doesn't own the footer; design + cmdline footers still show).

## `:open` + scheme registry

Tangential to free-config but a v0.6 surface: `:open <path>` opens
any file as a content tab via a pluggable scheme registry
(`feature/open-target.js`). Two schemes ship:

- **host** — relative or absolute paths.
- **docker** — `docker://<container>/<path>` reads via `docker exec`.

Each scheme declares `match` / `complete` / `open` hooks. Future
schemes (ssh, s3) plug into the same contract. See
[CMDMODE.md § `:open` schemes](CMDMODE.md#open-schemes) for the user-
facing reference.

## What didn't make v0.6

Three items got discussed and deferred:

- **Pane-select dropdown** on every non-detail panel (a per-cell
  `[≡]` analog to detail's tab-list trigger). Specced (`[≡]` glyph,
  shows all pool entries, pick = replace/swap), deferred to a future
  arc.
- **Multiple detail panels** (rethinking the singleton-detail
  architecture). Discussed as a v0.7-level question; sketched in the
  half-view bug investigation. Not a v0.6 blocker.
- **`leaves/design.js` mouse-gesture split** — file grew past its
  prior peak in this arc (the swap reducer + preview compute + view
  guards all added back to it after the pool-drag carve-out). Clean
  seam is `leaves/design-mouse.js`; deferred to the next refactor
  pass.
