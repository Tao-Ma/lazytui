## Why multi-track

The v0.6.2 closure left a backlog of small structural items
([[v062-shipped]] §Deferred): `slice.lines` field still exists
despite being derived (T2d-final); search matches don't auto-
recompute on `lines` change (A4); `panel-widgets.js` mixes chrome
glyphs with `buildTabStrip` (N10); `feature/register.js` is a
thin bridge that could fold (N12). Each is too small to justify
its own release. They share testing surface with Track A
(viewer + render layers) and the pane-select dropdown (Track D)
naturally pairs with Track A's P4 decor refactor — landing them
together avoids cross-arc churn on the same files.

The render engine arc itself (Track A) closes a structural bug
class — what v0.5 did for `update`, this does for `view`.

## Track A — render engine: rects + painter + frame cache

### Why

The v0.6.2 render bug had a textbook shape:

> "ALL panels in the leftmost column collapsed → right column shifts
> to x=0 for rows past the collapsed bars."

`distributeColumnHeights` parks rounding leftover on the last
**non-collapsed** panel; when *every* panel is collapsed, the branch
falls through and the column's per-panel heights sum to `n × 1`
instead of `availH`. `paintColumns` then row-concatenates columns —
for rows past the column's actual output, `splits[ci][i] || ''`
contributes zero horizontal width, and the next column's content
shifts left into the freed space.

The fix in `6d9ad31` pads each column's rendered output to `availH`
in `renderNormal`. It works. But the diagnosis is a deeper structural
issue: the render engine has four stacked **implicit invariants**:

| Layer | Function | Implicit invariant |
|---|---|---|
| L0 | `distributeColumnHeights` | per-panel heights sum to `availH` |
| L1 | each panel `render(p, w, h, slice)` | returns exactly `h` lines × `w` cells |
| L2 | `renderNormal` `\n`-joins per column | column output = `availH` lines × `w` cells |
| L3 | `paintColumns` row-concats columns | row *i* = `cols(term)` cells total |

Every emergent feature has nibbled at one of these:

| Feature | Invariant nibbled |
|---|---|
| Collapsed panels (v0.6) | L0 — "park leftover on last visible" excludes all-collapsed |
| `heightPct` anchored (v0.6.2) | L0 — over-subscribe scaling |
| `_safeRender` on throw (T28) | L1 — returns 1 line instead of `h` |
| Chrome glyphs `[_]/[X]/[≡]` | L1 — regex post-mutation of the top-border row |
| Drag-preview `arrange` swap | render() mutates slice + restores in `try/finally` |
| `panelBounds` slice writes | render → slice writes (blessed exception) |
| `_panelHeights` module-local | side channel between `calcLayout` and `renderNormal` |
| `viewer.innerH` direct write | render → Component slice write |
| `syncPanelScroll` from `calcLayout` | view → reducer Msg dispatch |
| `terminal_exit` setImmediate dispatch | view → reducer Msg dispatch (via timer) |
| Diff caches | 6 independent module-local invalidation flags |
| `renderHalf`/`renderFull` | 3 near-parallel render paths; small `availH` formula drift |

The terminal overlay already shows the structurally correct model:
it reads `panelBounds.detail`, paints with absolute cursor
positioning (`\x1b[y;xH`), and diffs against `session.prevFrame`. No
string-concat, no row-alignment-by-construction. The bug class
doesn't exist there.

### Goal

Generalise the terminal-overlay pattern to the whole render engine.
Replace the per-column string pipeline with:

1. A **`Layout` value** computed once per frame, carrying rects
   (`{paneId, x, y, w, h}`) — no side channels.
2. A **`Rect` contract** for panel renderers: return exactly `h`
   lines, each padded to `w` cells.
3. A **rect-placing painter** that writes each rect at its `(x,y)`
   with absolute cursor positioning — no row concat across columns.
4. **Chrome glyphs as decor**, layered onto rects by the painter
   instead of regex post-mutation.
5. **Render-side effects lifted out** into Msg arms triggered by
   the IO layer (resize, pty-exit).
6. A **single frame cache** for diff repaint, replacing the six
   module-local flags.

Net effect: the four implicit invariants collapse into one explicit
type (`Rect`), each violation becomes a debug-mode assert, and the
column-shift bug *class* cannot manifest.

### Non-goals

- **No visual change.** Same panels, same chrome, same colors,
  same view modes. This is a structural refactor; users notice
  nothing. (`test-render-idempotent.js` snapshots survive.)
- **No new render features.** Multi-viewer, second detail pane,
  per-column collapse — all deferred to v0.7+.
- **No PTY rework.** `renderTerminalOverlay` already does the
  right thing; it becomes a managed-rect declaration in P3 but
  its internals stay.
- **No Component API change.** Render signature stays
  `render(panel, w, h, slice)`. Only its return-shape contract
  tightens (P2).

### Target architecture

#### Layout value (P1 output)

```js
Layout = {
  rects:    Rect[],            // panel placements, ordered draw-back-to-front
  availH:   number,
  viewMode: 'normal' | 'half' | 'full',
  cols:     number,
  rows:     number,
}

Rect = {
  paneId:   string,            // or 'detail', 'footer', 'overlay:menu', …
  type:     string,            // panel type for renderer dispatch
  x: number, y: number, w: number, h: number,
  collapsed?: boolean,
  managed?: 'pty' | null,      // P3: painter delegates to PTY session
}
```

`Layout` is the **only** return of `calcLayout(model)`. No
`_panelHeights` module-local, no `layoutSlice.panelBounds` write,
no `layoutSlice.panelHeights` write. Hit-tests (mouse, drag math)
read the most recent `Layout` via a thin `getCurrentLayout()`
accessor that the renderer publishes once per frame.

#### RenderTree (P2 output)

```js
RenderTree = {
  layout:   Layout,
  paints:   Paint[],          // one Paint per rect
  decor:    Decor[],          // chrome glyphs, layered after paints
  overlays: Overlay[],        // modal overlays (menu / cmdline / …)
  cursor:   { visible: boolean, x?: number, y?: number },
  footer:   { line: string },
}

Paint = {
  rect:  Rect,
  lines: string[],            // EXACTLY rect.h entries; each visibleLen === rect.w
}

Decor = { x: number, y: number, glyph: string, color: string }

Overlay = {
  kind: 'menu' | 'cmdline' | 'copy' | 'prompt' | 'confirm'
      | 'register-popup' | 'which-key' | 'tab-list' | 'jobs'
      | 'panel-list',
  paint: Paint,               // overlays produce their own rect-paint
}
```

`view(model) → RenderTree` is a **pure function**. `view()` calls
`calcLayout`, walks each rect to its renderer, composes decor for
chrome / tab triggers, runs the active overlay's paint, and returns
the tree. No I/O, no slice writes, no Msg dispatches.

#### Painter (P3 + P6)

```js
paint(prevFrame, tree) → { ansi, nextFrame }

Frame = {
  cells:        string[][],     // grid of rendered cells [y][x]
  overlayKinds: Set<string>,    // set of overlays painted this frame
}
```

The painter:

1. Composes a `cells[y][x]` grid by stamping each `Paint.lines`
   into its rect's `(x,y)` region. Decor cells overwrite their
   underlying paint cells. Overlay cells overwrite both.
2. Diffs cell-by-cell against `prevFrame.cells`; emits absolute-
   cursor-positioned writes only for changed cells.
3. Forces a full repaint when terminal size changes OR
   `overlayKinds` shrinks vs. the prior frame (overlay-close
   detection — same logic as today's `_prevOverlayFlags`).
4. Delegates `managed: 'pty'` rects to their session's existing
   `prevFrame` diff (terminal overlay stays as-is internally).

`prevFrame` threads through paint; no module-local flags.

#### P2 — Rect contract (~3 commits)

**Goal.** Every panel renderer's return is contractually
`Paint.lines` — exactly `h` entries, each `visibleLen() === w`.
Debug-mode assertion catches violations at the boundary.

**Mechanics.**
- `_safeRender(panel, rect)` parses the panel render's string
  output into `lines: string[]`, splitting on `\n`. If `lines.length
  !== rect.h` OR any `visibleLen(lines[i]) !== rect.w`, debug mode
  throws; release mode pads-and-truncates to recover (mirrors
  today's column-pad safety net).
- `_renderCollapsed(p, w)` returns `[oneLine]` (length-1 string[]).
- The error-marker on throw now returns
  `Array(h).fill(' '.repeat(w))` with the error message stamped on
  row 0 — preserves the panel's vertical slot regardless of
  height.

**Risk.** Existing panel renderers may emit slightly-too-short
lines (relying on terminal default-fill) or trailing `\n`. Debug
assert exposes them; fixes are local.

**Verification.**
- Run every existing test in debug mode (assertions on) — anything
  failing the rect contract gets one fix-up commit per renderer.
- New: `test-rect-contract.js` — for each registered panel type,
  call `render(panel, w=40, h=8, slice)` with a few fixture slices
  and assert the contract.

#### P4 — Chrome as decor (~3 commits)

**Goal.** `injectTopRowChrome` and `injectTabTrigger` retire.
Chrome glyphs (`[_]/[+]`, `[X]`, `[≡]`) become `Decor[]` entries
the painter layers on top of rect cells.

**Mechanics.**
- `composeRects` walks placed panels, computes each panel's
  decor positions from its rect bounds (existing
  `_collapseGlyphX0` / `_closeGlyphX0` math moves into a
  `decorForPanel(rect, freeConfig, focused)` helper), pushes into
  `RenderTree.decor[]`.
- Painter stamps decor cells on top of the underlying paint cells
  before diffing. Atomic — no regex anchor on `─*╮[/]$`.
- Tab-bar bounds (`panelBounds.detail.tabs`) move into the
  `Layout` value as part of detail's rect: `Rect.tabBounds?:
  [{label, x, y, w}]`. The viewer Component returns `tabBounds`
  alongside its paint lines; `composeRects` lifts them onto the
  rect.

**Risk.** Pure regex retirement — straightforward.

**Verification.**
- Existing tests using mouse-tab-click stay green (hit-test reads
  `Layout.rects[detail].tabBounds`).
- `test-panel-border-color.js` (border-color preservation) — the
  decor-overwrites-cells path must preserve the rect's color
  attribute on cells *adjacent to* the decor glyph.

#### P5 — Render-side effects lifted (shipped P5.1 only; rest deferred)

**Status.** Single commit shipped (`b56cf82`). The other three
items hit the same purity wall P1.4 surfaced — they need view-
derived data available to **leaves** (`leaves/pane-tabs.js` reads
`slice.innerH`; navigator `update` arms need scroll-clamp inputs).
Leaves can't reach into the layout module without breaking test
isolation, and the alternatives (parametrize every leaf
signature, dispatch a fan-out Msg per layout-affecting Msg) are
each comparable in scope to P3 itself.

**Shipped:**
- **P5.1** (`b56cf82`) — `terminal_exit` setImmediate retires.
  The Msg now dispatches event-driven from
  `pty-lifecycle.handleExit` (event-source = PTY `onExit`, not
  render poll). Side benefits: no 1-frame visual lag,
  renderTerminalOverlay loses its only dispatch.

**Documented blessed exceptions** (added to PRINCIPLES §11 list):
- `syncPanelScroll` from `calcLayout` — scroll-clamp dispatch
  per visible navigator per frame. The Msg arm is pure +
  identity-preserving (no ping-pong on same-scroll input);
  retiring would require fan-out across every layout-affecting
  Msg or per-leaf signature change.
- `viewer.innerH` slice write at render exit. View-derived
  projection; same "view-output" class as `panelBounds`. Read
  by viewer + pane-tabs leaf scroll-bottom math. Retiring
  cleanly requires the leaf to receive `innerH` as a param OR
  the layout slice to expose it (also requires update-arm
  recompute on every layout-affecting Msg).
- Drag-preview `arrange`-swap in `render()` (try/finally).
  Mutates `slice.arrange` AND `slice.panelBounds` for the
  duration of a render pass so calcLayout / composeRects see
  preview state; restores in finally so the next mouse hit-test
  reads canonical bounds. Two-state hack but well-contained.
  Retiring requires calcLayout/composeRects to accept an
  `arrangeOverride` param threaded through every reader.

(Original P5 plan kept below for the design reasoning trail.)


**Goal.** `view(model)` becomes a true pure function. No Msg
dispatch from the render path.

**Mechanics.**
- **`syncPanelScroll` → `frame_resized` Msg.** The IO term-resize
  handler (`io/term.js`) gains a SIGWINCH listener that dispatches
  `{type: 'frame_resized', cols, rows}`. Each navigator's `update`
  arm clamps its scroll on receipt. The current "calcLayout
  dispatches set_scroll per visible navigator per frame" pattern
  retires.
- **`viewer.innerH` slice write → derived value.** `Layout` carries
  the detail rect; viewer reducer reads it through a thin layout
  accessor when it needs `innerH` for scroll clamping. The direct
  `setInstanceSlice` write at `render` line ~705 deletes.
- **`terminal_exit` setImmediate dispatch → effect.**
  `renderTerminalOverlay` detects exit + returns a flag in the
  managed-rect's metadata; the painter records the exit + emits
  a `terminal_exit` Msg through the normal effect channel. The
  setImmediate timer trick deletes.
- **Drag preview `arrange`-swap in render() → `view` consumes
  `previewArrange` directly.** No slice mutation + restore;
  `view(model)` reads `layoutSlice.freeConfig.drag.previewArrange`
  and computes `Layout` from it without touching the live slice.

**Risk.** Resize-driven scroll-clamp behavior must match the
current "clamp on every render" pattern in practice (race: a
keystroke that changes scroll between resize and the next render).
Mitigation: `frame_resized` is dispatched *synchronously* on
resize (before next render), so the timing matches.

**Verification.**
- `test-immutable-runtime.js` and the freeze suite — render path
  must not mutate any slice. Add a debug-mode `Object.freeze` on
  every slice during render in test mode.
- Resize stress: rapid SIGWINCH → assert no scroll-clamp regression.

#### P6 — Single frame cache (~3 commits)

**Goal.** Replace `_prevRows`, `_prevCols`, `_forceFullRepaint`,
`_prevOverlayFlags`, `_forceOverlayFull`, `_lastOverlayId`, and
per-session `prevFrame` aggregation with a single threaded
`Frame` struct.

**Mechanics.**
- The painter owns one `Frame { cells, overlayKinds, cols }`
  global to the module. `paint(prevFrame, tree) → nextFrame` is
  pure-ish (side effects = stdout write only).
- `forceFullRepaint()` becomes `invalidateFrame()` — zeros
  `prevFrame.cells`. `invalidateRows(start, end)` becomes
  `invalidateFrameRows(start, end)`.
- PTY session `prevFrame` aggregates into the global `Frame` via
  the `managed: 'pty'` delegate — one cache, one invalidation
  path.

**Risk.** PTY diff currently lives on the session object so
multiple sessions don't trample each other. P6 must keep that
property — perhaps `Frame.ptyCells: Map<sessionId, string[]>` as
a sub-region of the global cache.

**Verification.**
- `test-render-idempotent.js` — full-repaint behavior on
  overlay-close still works.
- Manual PTY toggling — switching between terminal tabs in
  detail does not show stale pixels.

### Migration discipline (Track A)

- **Suite stays green between phases.** Each phase is one or more
  commits; the suite (currently 74/74) runs after each commit.
  Phase boundaries are explicit commits with a CHANGELOG line.
- **No mass renames per phase.** Behavior changes only; renames
  follow as separate NIT commits if needed.
- **Doc parity each phase.** PRINCIPLES §11 "Render is idempotent"
  + v0.5-layering.md §5 "view-derived data" both need updates
  per phase (the blessed-exception list shrinks each time).
- **Adversarial pre-tag pass per [[pre-release-review]].** Same
  4-track review (arch / file layout / code / doc parity) before
  tagging v0.6.3 — same discipline that caught the v0.6.2 column-
  shift bug.

| Render dispatches Msgs mid-frame | View is pure (P5) |
| Multi-cache invalidation desync (6 flags) | One `Frame` struct (P6) |
| `renderNormal/Half/Full` divergence (small `availH` formula drift) | One `composeRects` (P3) |
| Drag-preview swap + restore | `view` reads `previewArrange` directly (P5) |

### Out of scope for Track A (deferred to v0.7 or later)

- **Multi-instance viewer / second detail pane.** P3's
  `composeRects` could in principle place two detail rects, but
  the singleton-detail invariant (parser-enforced) stays for
  v0.6.3. Multi-viewer is v0.7.
- **Custom paint primitives** (image-cells, true-color sprites,
  etc.). `Paint.lines: string[]` is the only paint primitive.
- **Animation / transitions.** Same instantaneous repaint
  semantics as today.
- **Renderer for non-terminal targets.** The painter writes ANSI;
  a Web/Electron target would need its own painter against the
  same `RenderTree`. Out of scope.

### Open questions (Track A)

1. **`Decor` ordering.** When two decor entries overlap (e.g., a
   focused-panel chrome glyph next to an overlay's drop shadow),
   what's the layering rule? Proposal: stable sort by
   `(overlay-rank, declaration-order)`.
2. **`managed: 'pty'` cursor positioning.** The PTY's own cursor
   currently positions inline from `renderTerminalOverlay`. After
   P5 it should be expressed in `RenderTree.cursor`. Does the
   painter ask the session for the cursor coord, or does the
   session push it into the `Paint`'s metadata? Proposal: session
   exposes `getCursorOffset() → {dx, dy}` and the painter
   composes against the rect.
3. **Layout publication shape.** Renderer publishes
   `_currentLayout` as a module-local — is this the same kind of
   side-channel we're trying to retire? Counter-argument: mouse
   handlers need *some* way to read the most-recent geometry, and
   threading it through every input handler would be ceremony.
   Proposal: keep it, document it as the ONE remaining view-derived
   read channel (replacing the four-channel current state).
4. **Debug-mode assertion cost.** P2's rect-contract assert runs
   `visibleLen()` on every line of every rect on every frame.
   Probably fine at 100Hz; measure before committing the assert
   to debug-mode-always. Could gate on `LAZYTUI_RENDER_CHECK=1`.

## Track B — viewer correctness

Two small viewer-internal fixes carried over from v0.6.2's deferred
list. Both share the viewer's reducer + finalizer surface; landing
together avoids two adversarial-review passes on the same file.

### B1 — Retire `slice.lines` field (deferred from v0.6.3)

**Status.** Deferred. The original spec framed `slice.lines` as a
"redundant" derived mirror that every reader could compute on
demand. On closer inspection — the field IS redundant *as a
mirror* (T2d already established that — finalizer derives it from
`viewerLines(slice)` once per Msg), but it serves a load-bearing
second role: it is the **materialized active-tab snapshot** that
pure leaves and tests read without needing model + plugin lookups.

Reader-site count is closer to 10 than 6:
- `viewer.js` — `_beginSelect`, `_scrollView`, `_moveCursor`,
  `_withDerivedFields` (originalSlice.lines), `viewer_set_content`,
  `select_extend` (six reducer-side, matches the spec count).
- `panel/viewer/select.js:66` — `_detail()?.lines?.[i]` (the
  pure ANSI-aware width reader cannot reach the model).
- `leaves/search.js:79, :165` — `computeMatches(slice.lines || [],
  term)` + `scrollToActive` `slice.lines.length`. Pure leaf,
  cannot import `getModel` / `_infoFromFocus`.

Removing the field forces these readers to either accept `lines`
as a threaded parameter (every call site updates) OR import model
context (corrupts leaf purity — same wall P1.4 and P5.2-4 hit).
On the test surface, 8+ test files write `slice.lines = X` for
fixture seeding AND assert `slice.lines.length` for derived-state
checks — a "mechanical" rewrite would balloon past the original
2-commit budget.

The v0.6.2 T2d arc already extracted the perf win (no per-append
mirror spread; +83% at 50k); the remaining structural gain from
deleting the field is small relative to the parameterization
cost. Revisit in v0.7+ if a stronger reason emerges (e.g.,
multi-viewer makes the snapshot ambiguous across panes).

**B2 ships independently** — it only needs the
`prev.lines !== next.lines` ref-equality transition detect, which
still works with the finalizer-maintained field.

### B2 — Search auto-recompute on `lines` change (shipped)

**Status.** Single commit. The finalizer (`_withDerivedFields`)
gained a post-derivation block: when `originalSlice.lines !==
lines` (ref equality is enough for buffer-backed tabs, which
return the source array directly) AND `updated.search.active`
AND a committed term is present, the finalizer calls
`ms.recomputeFor(updated, term)` inline. No separate Msg
dispatch — calling the pure leaf directly keeps it to one
finalizer pass.

The spec proposed emitting a `{type: 'msg', msg:
viewer_search_recompute_for}` effect; inlining is simpler and
avoids the second update + finalizer cycle the effect would
trigger. The `viewer_search_recompute_for` reducer arm stays
available for the test-facing facade.

Gating notes:
- `updated.search.active` (post-reducer state) is checked, not
  `originalSlice.search.active` — `viewer_search_clear_committed`
  flips active to false and we respect that.
- Ref-equality on `lines` is correct for action / content /
  transcript tabs (viewerLines returns the source array directly);
  the Info-via-getInfo path allocates a fresh array per call
  (over-recomputes on every Msg when active search + Info focus),
  but Info content is small so the cost is negligible.

**Verification.** `test-viewer-search.js [10]` (3 cases): match
re-derives on append, non-matching append still re-counts to
prior, inactive search is not touched.

## Sequencing

Track A is the spine; B-E land in slots that minimize re-work.

```
Track A:  P1 ── P2 ── P3 ── P4 ── P5 ── P6
                          │
Track B:                  │      B1 ── B2
                          │
Track C:                  C1 ─ C2
                          │
Track D:                  │   D1 ── D2 ── D3
                          │
Track E:                                     E1     E2
```

Ordering rationale:
- **P1 → P3 first.** Layout-as-value + Rect-contract + painter
  is the foundation; nothing else lands until the suite is green
  on the new path.
- **P4 (decor) gates Track C1 and Track D.** Chrome glyph
  consolidation (C1) and pane-select dropdown (D) both ride on
  the unified `Decor` model.
- **Track B is independent of Track A** — could land first or
  parallel. Default: after P3 so the recompute-on-lines-change
  detection (B2) uses the post-P3 finalizer shape.
- **Track C2 (register fold) is fully independent** — land
  opportunistically; default after C1 for batched-PR ergonomics.
- **P5 → P6 last on Track A** — the effects lift (P5) clears
  the path for the single frame cache (P6).
- **Track E (opportunistic) lands at the end** — E1 is trivial
  and waits for any tag-prep slack; E2 (file split) lands LAST
  to avoid mid-arc merge churn on a hot file.

## What v0.6.3 explicitly defers to v0.7+

These were considered but stay out:

- **Multi-instance Components.** Two same-type tabs sharing one
  slice is the v0.7 spine. Pane-select dropdown (Track D) does
  NOT depend on multi-instance — it swaps WHICH single instance
  occupies the slot, not how many.
- **`panelBounds[p.type]` → paneId-keyed reader flip.** Track
  A P1 introduces `boundsFor(paneId)`; the 14 existing readers
  flip to it as part of P1, but the full type-keyed write
  retires only when multi-instance lands.
- **Pool-id → Component-name registry.** `instanceKind(non-
  default-pool-id)` returning null is a v0.7 fixup.
- **Tab cycle keybind / mouse UX.** Cycle key for tab nav
  ([[v07-candidates]] §6 #4) waits for multi-instance.
- **config-status flatten.** 3 internal tabs → 3 peer pool kinds
  ([[v061-shipped]] §Deferred independent).
- **N13 chrome/input bypass `route.resolveTarget('viewer')`.**
  This is a v0.7 multi-viewer prep item; with singleton-detail
  it's a no-op rename, so defer until the rename is load-
  bearing.
