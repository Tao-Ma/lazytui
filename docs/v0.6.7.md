# v0.6.7 — architecture kernel & data-flow upgrade

A research-driven release: study the Elm Architecture / MVU lineage and mature
TUI tools, *learn rather than copy*, and upgrade lazytui's kernel and data flow
where the borrowed idea genuinely improves ours. Each candidate was grounded in
our actual code before adoption — several were reshaped or dropped once measured
against what we already have.

The work is organized into four thrusts, shipped in phases so each step stays
verifiable (suite + smoke + acyclic layer graph + bench parity).

## Thrusts

- **A — render kernel.** Cell-granular render diff (A2), lazy view-construction
  memoization keyed on slice identity (A1), keyed diffing for dynamic lists (A3).
- **B — replay → debugger.** A debugging surface over the v0.6.6 WAL + checkpoint
  engine: scrubber, skip-and-recompute from the nearest checkpoint, per-Msg model
  diff, pause-vs-lock, a `--dev` WAL console (B6).
- **C — effect / dispatch kernel.** A named inbound dispatch-middleware seam (C7),
  keyed/exclusive effect cancellation (C5), reducer-ergonomics review (C8).
- **E — input / UX as data.** A pure focus *stack* / context system (E10),
  declarative keybindings + auto-generated help/footer (E9), modal-returns-a-result
  (E14).

## Phasing

| Phase | Items | Status |
|-------|-------|--------|
| **1 — foundations + first render win** | C8 · C7 · A2 | **DONE** |
| **2 — render kernel (rest)** | A3 window-only decoration (reshaped) · A1 **deferred** | **DONE** |
| **3 — navigation + input-as-data** | nav-history (jumplist) · E14 modal continuation · E9 footer hints-as-data | **DONE** |
| **4 — capstone** | C5 keyed effect cancellation · B6 replay debugger | **DONE** |

**Phase 4+ candidates (feature-gated, not committed):**
- **E9 dispatch-from-data** — make the normal-key dispatch read the binding
  registry (entries gain a `verb`/`msg`; a `dispatchChain` resolver walks the
  active contexts). Deliberately **not** done in Phase 3: as a pure refactor it
  is risky churn on the core input path for guarded/dynamic arms that already
  work and are single-source. It earns its risk only behind a **user-facing
  feature that needs dispatch to read the registry** — most naturally
  *user-configurable normal-mode keybindings* (a remap must flow through the
  registry to take effect). The Phase-3 `leaves/input/bindings.js` is shaped so
  this is an **additive** step (add the field + a lookup + the resolver), not a
  rewrite; even then it stays partial (the guarded arms likely remain in the
  switch). The powerline/extensible-footer arc does **not** need it (that
  consumes `footerFor`, not dispatch).

Phase 3 was **reshaped** by research + grounding. The planned "E10 full context
stack" was **dropped**: cross-system study (emacs, macOS, browsers, Textual,
lazygit) found everyone computes *one derived dispatch chain* and keeps focus +
modals **orthogonal** — which lazytui already does (flat `layout.slice.focus` +
`CHAIN_MODES`, both replay-snapshottable). A stack buys nothing for flat
(one-at-a-time) modals + single focus, and a stack is LIFO — it can't model "go
forward." The instinct behind the stack was really **navigation history**, which
wants a browser/vim-jumplist (list + cursor), not a stack. So Phase 3 became the
jumplist (the headline) plus the two substrate items (E14, E9), each scoped by
grounding it in the code.

## Phase 1 (shipped)

### C8 — reducer-write ergonomics: `withModalMode`

A draft-proxy immutable-update library was evaluated for the reducer layer and
**rejected**, grounded in our code: the *deepest* nested writes are the *hottest*
paths (the viewer's `actionTabBuffers[group][tab].lines` insert runs per streamed
line), where a per-update proxy allocation is a bench-parity liability; the *cold*
modal writes — the only place such a library would be safe — are already softened
by `model-ops` `withModes`/`withModal`. It would also have been the first runtime
dependency added purely for internal ergonomics (the existing three each provide an
external capability), and would need re-validating against the deep-freeze/purity
test suite.

Substitute: a dependency-free `model-ops.withModalMode(model, modeFlagPatch,
modalPatch)` that encodes the modal coupling identified in the reducer-cohesion work
(#D12) — a modal transition always flips a `model.modes` flag *and* seeds/clears its
`model.modal` sub-state. The six modal sub-reducers whose enter/exit arms match that
shape (confirm, cmdline, prompt, menu, copy, filter) now read as one intent; the
three that also touch a third field/subtree (jobs/diag-log carry `now`,
register-popup touches `model.register`) stay explicit.

### C7 — inbound dispatch-middleware seam

The dispatch loop's only inbound tap was three hardcoded session-log record calls
(one per lane: root / comp / key). C7 factors them into one ordered link list
(`leaves`-free, in `dispatch/runtime/middleware.js`):

```
link = (entry, next) => result        // entry = {lane:'root'|'comp', msg} | {lane:'key', key, seq}
```

Links run in registration order (first = outermost), terminating in the lane's
dispatch. The chain is composed once per stable lane-terminal (cached by identity),
so steady-state cost is two links wrapping a direct call — bench-neutral.

Two built-in links:
- **WAL record** (innermost) — records the post-transform entry; self-gates on the
  session-log enable flag, so it is the same near-zero-cost tap when recording is off
  and inert during replay folds.
- **crash reporter** (outermost) — the root-reducer path had no `try/catch`, so a
  thrown reducer crashed the process uninspected. It now stamps `{entry, error, WAL
  tail}` into the event log and re-throws (observe, don't change control flow).

**Contract:** a link observes / transforms / wraps only — it MUST NOT dispatch (emit
a Cmd instead, so the cross-layer recursion cap stays consistent) and MUST NOT do
non-replay-safe I/O (the record link is the one sanctioned "I/O-ish" link, and it
self-gates). The post-dispatch finalizer and the instance/subscription reconcilers
stay distinct *outbound* stages — different cardinality (once per outermost dispatch,
not per Msg), phase (post-commit, re-enters the loop), and replay treatment.

Extension point (not built in Phase 1): an always-on dev purity-guard link and the
Phase-4 `--dev` WAL console plug into the same list.

### A2 — cell-granular render diff

The painter repainted a whole row whenever it changed, re-emitting an entire line
for a one-cell change (a moving cursor, a spinner frame, one digit in a footer).
A2 adds `leaves/render/cell-grid.js`:

- `rowToCells(ansi)` — parses a row into per-visible-column cells `{glyph, width,
  active-SGR}`, wide-char aware (a double-width glyph claims a continuation column).
- `diffRowToAnsi(prevMarkup, curMarkup, rowIdx)` — emits only the changed cells, with
  the standard buffer-diff economies: a cursor MoveTo only when the next change isn't
  adjacent, SGR re-asserted only when the active style changes (carried across a run),
  and two-sided wide-char invalidation. The `composeRows` fixed-width invariant means
  no per-row clear-to-EOL is needed.

It is wired into `paintFrame` and is **on by default**; `LAZYTUI_CELL_DIFF=0` is an
opt-out escape hatch back to the row-level path (kept for this release; removable once
proven). The diff is a pure function of the two row strings, so the emitted byte
stream is a function of the model — replay-safe. The replay change-highlighter
(`cell-diff.js`) keeps its own row path.

**Correctness** is pinned by `test-cell-grid.js`: each emitted patch is applied to a
simulated terminal row that starts as the prev-rendered row, and the result is
asserted byte-for-byte equal to the cur row — across interior / leading / trailing /
full-row / style-only / wide-char / multi-run cases, plus MoveTo economy.

**Measured trade** (`bench-cell-grid.js`): −85% bytes on localized updates
(clock/spinner), neutral (±1%) on full-row-style changes (a selection bar flipping
reverse-video; typing). Relative CPU is 5–15× higher (it parses both rows) but
absolutely negligible at TUI render rates — the paint path runs at most ~60fps and is
50ms-debounced, so even a worst-case full-screen scroll diff (~150µs) is well under 1%
CPU. The render emit is not a `bench-hotpaths` reducer path. The byte reduction helps
every remote/SSH session (less flicker, less terminal write volume).

### Phase 1 follow-up — the kana/hangul width fix (found by the A2 eyeball check)

Verifying default-on A2 against a real terminal surfaced a latent renderer bug.
`leaves/text/ansi.js` `charWidth` — the single width source of truth (feeds
`visibleLen`, viewer truncation/selection/search, chrome draw, AND A2's cell-diff) —
used a *hand-picked* East-Asian-width range table that **omitted hiragana, katakana,
and hangul** (it returned 1; the terminal renders 2). Two failure modes, one root cause:

- **Row-level path (pre-existing):** `composeRows` padded by the undercount, so a
  kana/hangul row was physically *wider* than its pane — the right border and
  everything after it shifted/overflowed.
- **A2 cell-diff (surfaced as a default-on regression):** `diffRowToAnsi` emits
  *absolute* per-cell `MoveTo` columns computed from `charWidth`; the undercount drifted
  them off the terminal's real cursor (a CJK row's `▐` border landed 3 columns early),
  so cell-diff did **not** reproduce the row path for kana/hangul.

A new A/B harness pinned it: drive real viewer content/scroll frames through the real
pipeline under cell-diff default vs `=0`, replay each paint byte-stream into a real
`@xterm/headless` terminal, and assert per-step screen equality. It diverged on every
kana row before the fix and is identical after.

**Fix — use a library, not a curated table.** Unicode width is a standard,
spec-evolving problem; hand-maintaining a range table is the wrong tool (it shipped this
bug). `charWidth` now delegates to the **`eastasianwidth`** dependency (UAX #11), via its
per-codepoint `eastAsianWidth()` primitive with our locale policy **Wide/Fullwidth → 2,
everything else (incl. Ambiguous) → 1**. Ambiguous→1 keeps box-drawing/arrows/enclosed
digits — all of the chrome — at width 1; emoji classify as 1, matching the embedded
terminal's cursor advance (the cell-diff oracle). An ASCII fast path (`cp < 0x1100`)
keeps the hot width-summing loops allocation-free. This is the first dependency added for
internal correctness rather than an external capability, and it reflects a deliberate
**dependency-policy refinement**: add a dep for a *standard* problem (char/lang/i18n/
Unicode), not for internal ergonomics (the C8 Immer rejection stands — that was
convenience, not a standard problem).

`eastasianwidth` is CommonJS with zero transitive deps (the modern, actively-maintained
`get-east-asian-width` is ESM-only, which a synchronous hot-path `charWidth` can't use).
Three archaic ranges where the lib's Unicode vintage and the terminal disagree (Yijing
hexagrams, Hangul Jamo Ext-A, Kana Supplement) are documented and pinned in the test —
unhittable in real content.

**Correctness pinned by** `test-char-width.js`: a baked oracle table (the actual
`@xterm/headless` cursor advance per codepoint) asserts `charWidth` matches the terminal
across every script + the width-1 traps (box-drawing, ambiguous, emoji, halfwidth), plus
the three accepted divergences; `test-cell-grid.js` gains kana/hangul diff cases.

**Re-verification:** suite 115 green (default **and** `LAZYTUI_CELL_DIFF=0`); smoke 12
green; dep-walker acyclic both modes; dead-exports 0; `bench-hotpaths` parity (A/B vs the
pre-fix code — within noise, ASCII fast path costs nothing); `bench-cell-grid` savings
intact (−85% / −69%); the new screen-equality A/B harness green on kana/hangul.

## Phase 2 (shipped) — render-construction CPU

Phase 1's A2 optimized the *wire*. Phase 2 targets *view-construction CPU*. It was
**measurement-gated**: there was no bench for construction cost, so the first step was to
build one (`bench-render-construction.js`) and let it decide what ships. Run it on a fast fs
(a slow network mount can add tens of µs per relative `require()`, swamping the signal — run
from a local copy with `node_modules` symlinked).

### What the bench found (the gate)

| case | cost | decision |
|------|------|----------|
| viewer decorate, **no** search/selection | already O(1) passthrough (~7µs) | already fine — A3 must not touch it |
| viewer decorate, **active search**, whole buffer | 10k → 52ms, **50k → 270ms/frame (4fps)** | **the problem** |
| same, decorate visible window only | flat **~190µs** regardless of buffer size | **the fix (A3)** |
| full `render(model)`, no search | **~420µs/frame** (300-line ≈ 50k-line — window-sliced already) | A1 ceiling |

The render loop is 50ms-debounced (≤20fps). A full no-search frame at ~420µs is <1% of that
budget and *flat across buffer size* (the viewer already slices to the viewport before
rendering rows). The only construction cost that blew the budget was the viewer decorating
its **entire** buffer when a search or selection was active.

### A3 — window-only decoration (the reshape)

The spec called A3 "keyed list/scrollback diffing." Grounded in the code that proved
**redundant** — `composeRows`+`paintFrame`+A2 already diff at row/cell granularity on the
wire. The real waste was that `search.decorateLines` / `select.decorateLines` mapped over
the **whole** lines array (up to 50k) before `renderPanel` sliced the visible ~38 rows. So
A3 became: **slice the visible window first, decorate only those rows.**

- `viewer.js render()` slices `derived.slice(scroll, scroll+innerH)` and decorates that
  window; `renderPanel` gains a `windowed` flag (the caller pre-sliced — use `scrollOffset`
  for the scrollbar thumb only, don't re-slice).
- `search.decorateLines(window, slice, {offset, full})` / `select.decorateLines(window,
  {offset})` take the window + its absolute start. `matchesFor` still runs over the **full**
  buffer (the match count + active-match index must be global) but is WeakMap-memoized, so a
  pure scroll recomputes no matches. The match→line grouping is filtered to the window, so it
  is O(window) too (kept the global match index for the active-match check).
- Output is **byte-identical** to the old whole-buffer-decorate-then-slice: each line is
  decorated from its own content + its **absolute** index, so slice-then-decorate equals
  decorate-then-slice. The no-search/no-selection path is untouched (already O(1)).

**Measured result:** the active-search 50k render dropped **270ms → 0.93ms/frame (4fps →
1073fps), ~290×**; flat across buffer size. No change to the common no-search case.

### A1 — deferred (the gate said no)

A per-panel render memo (skip rebuilding an unchanged panel) was designed but **not built**.
The bench showed a full no-search frame is ~420µs — <1% of the 50ms budget, flat across
buffer size, and dominated by a handful of panes' cheap construction. Under the 20fps
debounce, skipping unchanged panels would save well under 1% CPU, against a real stale-frame
risk (a panel's output depends on more than its slice — `groups` reads the docker service
slice, `files` the live draft filter, the viewer the focused navigator). Not worth it at
realistic pane counts; revisit only for pathological configs (50+ panes). The measure-first
discipline is the point: the optimization that looked symmetric with A2 wasn't warranted.

### Stage 3 (matchesFor-on-append) — deferred with note

A cold `matchesFor` scan over a 50k buffer is ~15ms; an append makes a new array ref, so a
search active **during** heavy streaming re-scans per render. But the 50ms debounce caps that
at ≤20×/sec, and it only bites the narrow "active search + heavy stream + huge buffer" case.
Documented; revisit if reported. An incremental-append match cache is the fix if needed.

### Phase 2 verification

`bench-render-construction.js` records the trade above. Correctness: `test-window-decorate.js`
pins windowed-decorate == whole-buffer-decorate-sliced across scroll offsets, active-match
edges, and char/line selections straddling the window; the scratchpad screen-equality A/B
harness confirms **byte-identical** rendered screens (new vs the pre-change build) across a
34-step script with active search (incl. wide-char matches), selection, and scrolling. Suite
116 green (default **and** `LAZYTUI_CELL_DIFF=0`); smoke 12; dep-walker acyclic both modes;
dead-exports 0; `bench-hotpaths` + `bench-cell-grid` unaffected (render-path-only change).

## Phase 3 (shipped) — navigation + input-as-data

The headline is a **navigation history** (jumplist back/forward); two substrate
items ride along — **E14** (a uniform serializable modal continuation) and **E9**
(the footer's key hints as structured data). Each was scoped by grounding it in
the code; two were reshaped or trimmed once measured against what already exists.

### The replay constraint (governs nav-history and E14)

`replay.snapshotState()` clones the whole model; `replay.replayEntries()` re-applies
the recorded Msg log with **effects suppressed**. So any state that must survive a
checkpoint + fold has to be **born on the model and written only by the reducer**
(the Msg path), and any continuation stored in the model must be **serializable
data, never a closure** (a checkpoint is JSON; a closure would silently drop).

### Nav-history — `model.nav.history[] + cursor` (the headline)

Browser/vim-jumplist navigation: `model.nav = {history[], cursor, cap}`. Back =
`cursor--`, forward = `cursor++`; a new push after a back **truncates** the forward
branch; the ring is capped. A "location" is captured by **stable identity**, never a
fragile index: the group **name**, the focused pane (the fragile paneId kept only as
a hint behind a stable **type** fallback), and either the viewer's active tab
(**`targetKey`**) or the focused navigator's cursor item (**`idOf`**). The pure ring
math lives in `leaves/wm/nav-history.js`; the reducer arms
`nav_record`/`nav_back`/`nav_forward`/`nav_prune` delegate to it.

**Push seam** — the established "a Cmd that reads then emits a recorded Msg" pattern
(`jobs_route`), which keeps the push replay-safe. The three transition arms
(`set_current_group`, layout `focus_set`, pane-tabs `tab_switch`) emit a
`nav_capture` Cmd on a *real* change — suppressed by `msg.noCapture` during a restore
so retracing doesn't push new history. The `nav_capture` effect reads the post-commit
coordinates by stable identity and `applyMsg`s a fully-stamped `nav_record`; the pure
arm does the ring math. Because the stamped `nav_record` is recorded, the WAL fold
reconstructs `model.nav` without re-running the effect — pinned by `test-replay.js`,
whose recorded session now drives nav transitions.

**Restore** (`nav_restore` effect) resolves each stable coordinate to a *live*
address — `set_current_group(name)`; `focus_set(paneId if it still resolves, else the
type, letting layout fall back)`; `tab_switch(targetKey → current idx)`;
`set_cursor(idOf → current index, nearest if gone)` — and fires the existing
primitives stamped `noCapture`. Per-coordinate best-effort: a pane/tab/item that
vanished lands on the nearest; a record whose **group** is gone is the spine missing,
so it is pruned and the travel continues in-direction.

**Bindings:** leader chords `<leader>o` (back) / `<leader>i` (forward) — the vim
jumplist mnemonic (`Ctrl-o` = Older, `Ctrl-i` = newer/In) minus the Ctrl (project
rule — no `Ctrl-*` binds), auto-documented via the existing leader-chord help walk.

**Tests** (`test-nav-history.js`): the pure ring (push/dedupe/truncate/cap/step/
prune); one stable-identity record per transition (and **zero** on scroll / bare
cursor moves); back/forward restore focus; the cursor resolves by id and lands on the
**same item after the list reorders**; a removed-group record is pruned + skipped.

> **Coarse by design.** A location reaches group + pane + tab + focused-item. There
> is no host coordinate (the only "remote" is the config-static docker container;
> switching "hosts" = switching groups, already captured) and the record is a tagged
> `{v, kind}` envelope, so a directory (files cwd) or host coordinate is a
> purely-additive future key — decided if/when those features land, no migration.

### E14 — uniform modal continuation

The five modals stage a serializable Cmd **descriptor** on one field,
`model.modal.continuation`, and emit it (patched with the user's result) on a
successful dismissal — the Textual `dismiss(value)` shape. confirm/prompt are
opener-staged (the caller passes the Cmd; submit injects the parsed args);
copy/cmdline/menu stage a fixed base (`copy_commit`/`cmdline_run`/`menu_action`)
patched with the chosen idx+label / sel+args+display / verb+arg at the terminal arm.
Every enter sets it; every exit (success or cancel) clears it.

Grounding **simplified** the design: the planned `{on, emit, arg, via}` envelope was
dropped — `via`/`on` are constant for modals and the result fields differ per modal,
so `continuation` simply *is* the Cmd descriptor (as confirm/prompt's old `.cmd`
already was). The #D12 carve-out had already removed closures from the model, so this
is a uniformity + enforcement step, not a replay-safety fix: a new
`model-ops.findModalClosure` scan keeps the "no closure under `model.modal`" contract
*pinned* (a guard helper exercised by `test-modal-continuation.js`) rather than merely
conventional — every modal transition the test drives asserts the scan finds none. The
effect consumers are unchanged (they read the Cmd payload, not the model).
`test-replay.js` gains an isolated mini-fold proving a staged continuation
round-trips a checkpoint identically (a closure would not).

### E9 — footer key hints as data

The footer's key HINTS — the focus-kind segments (`↑↓ select | … | x menu | q quit`)
and the modal-mode tails (`Esc cancel | Enter copy`) — were hand-typed strings
scattered through `render/footer.js`. E9 makes them one declarative source, the pure
leaf `leaves/input/bindings.js`: hints per context, with pure `when(ctx)` guards for
the conditional detail variants (tabs when >1; `x close` vs `x menu` on a dead
ephemeral terminal; `Enter restart`/`activate`; `/ search`). `footer.js` projects
them via `footerFor()`/`footerSegs()` and interleaves only the genuinely-**live**
status it must compute (search match count, filter text, leader pending, terminal
label) — that status is not a key hint and stays inline.

Scoped by grounding ("Focused+"):
- **Byte-identical** output — `test-bindings.js` is a characterization test
  reproducing every prior footer string and all four detail variants; the smoke suite
  renders real multi-frame footers.
- `help.js` is left **as-is** — it is a separate *detailed* reference (prose like
  `x  Open menu popup`, not the terse `x menu`), already single-source via
  `def.keyHints` + the dynamic leader-chord walk; routing it through the terse
  registry would downgrade it.
- Per-panel `keyHints` stay on the panel **defs** — already a single source the
  footer and help both read (no drift), and they would collide with the
  `actions`/`groups` focus-kind contexts. (The docker `i`/`t`/`s` hints are not a
  dead-advertisement bug — `docker.update` dispatches them.)
- No dispatch migration / `dispatchChain` / `_SHADOWED` retirement: those touch the
  core input path for dynamic logic that already works and is single-source.

The win: the footer's hint content is now structured data — the clean "keys segment"
seam for a future extensible/powerline status line — and curating a hint is a
one-line data edit in one leaf.

## Phase 4 (shipped) — capstone: effect cancellation + replay debugger

Grounding showed a lot of each was already covered, which scoped both:

### C5 — keyed/exclusive effect cancellation

The effect protocol gains an optional `key` on a descriptor. In `runEffects`, a
keyed effect is **exclusive by key**: it aborts any in-flight effect with the same
key, then runs with a fresh `AbortSignal` injected on a *per-call host*
(`Object.create(host)`, so the ~40 synchronous handlers that read neither `signal`
nor `releaseKey` are untouched). The handler owns release. A module-local
`_inflight` Map holds the controllers — impure shell, never model state; the replay
early-return precedes the loop, so a fold never touches it (zero model-fold impact).
`execAsync` gains `{signal}` (passed to `spawn`) so an abort SIGTERMs the
subprocess, with the never-reject contract intact.

This **complements** — never duplicates — the existing teardown (stream
slot-preemption + Sub reconciliation, which own their own lifecycles). It is adopted
in the async **compute** effects, which had no such machinery:
- `dockerFetch` — true subprocess kill (the `inFlight` latch prevents a same-key
  supersede and the service owner is skipped by the pane-orphan abort, so its only
  trigger today is a clean kill at quit);
- `cfgStatusCompute`/`Diff` — blocking `spawnSync` (no killable handle), so this is a
  post-hoc **stale-result drop** (guard `signal.aborted` before dispatch), with
  exclusive-by-key superseding a prior same-pane diff.

Teardown: the pane-orphan dispose loop (`app/state.js`) aborts a removed pane's keyed
compute by id; `cleanup.js` `_clearInflight()` SIGTERMs any in-flight fetch at quit.
`test-effect-cancel.js` (25 asserts) covers supersede/cancel/sync-no-op/non-keyed/
replay-safety + the `execAsync {signal}` kill.

### B6 — replay → debugger (Full)

v0.6.6 had already shipped the replay *engine* (`advance`/`reverseTo`/`replayTo` =
seek-to-nearest-checkpoint fold), the interactive *scrubber* (seek/play/pause/speed),
the *change-highlighter* (visual cell/row tint), checkpoints, the WAL + CLI, and 14
tests. B6 adds the *debugger* layer on top — all controller state stays on the
module-held `S` (never the model), all new pure logic in `leaves/replay/`:

- **Per-Msg model diff** — a new pure leaf `leaves/replay/model-diff.js`
  (`diffState(a,b)` → bounded, Set-aware `{path,kind,before,after}` list). The
  controller computes it **on demand** (fold idx-1 vs idx, restore the displayed
  frame), cached on `S.diff`, gated by `S.diffPanel` (key `D`, zero fold cost when
  off) — NOT stored in the WAL. Rendered as a Changes panel in the scrubber.
- **Skip-to-next-change** (`n`/`N`) — advance to the next Msg whose model diff is
  non-empty, a single rolling fold bounded by `SCAN_CAP=512` (skips no-op Msgs +
  term/checkpoint entries).
- **WAL schema-versioning** — `SCHEMA_VERSION` stamped via one `_header()` helper at
  all three write sites; `load`/`loadMeta` read it with a no-hard-fail compat policy
  (unversioned → silent best-effort; older → warn; newer → loud warn, still attempt).

## Pre-release review

Two adversarial review passes (parallel arch / file-layout / code / doc-parity
agents, then a fresh code-first round told to ignore the docs), each finding
verified firsthand before fixing.

### Round 1

- **nav-history forward-prune overshoot (bug):** travelling *forward* across a
  removed-group ("404") record skipped the immediate live target. After the prune
  the next record shifts into the cursor slot, so the cursor already points at the
  target — the old code re-stepped and overshot. Forward now re-resolves in place;
  back is unchanged (its target is `cursor-1`, unaffected by the shift).
- **flat-modal continuation guard:** `confirm_enter` (the one modal-opener that can
  fire asynchronously — stream preempt / `action.confirm`) is dropped when a modal
  chain is already active, so it can't stomp another modal's staged
  `model.modal.continuation`. (Round 2 extended this to `prompt_enter`/`copy_enter`.)
- **docker `inFlight` latch:** an aborted `dockerFetch` now clears the latch (bare
  `dockerResult`) so a future non-quit `cancelEffect('docker:fetch')` can't stall
  polling.
- Doc-parity sweep + a bench rename.

### Round 2 — the cell-diff zero-width bug (the headline)

`charWidth` returned **1** for zero-width codepoints — combining marks, ZWJ,
variation selectors — where a terminal advances **0**. Because the cell-diff emits
*absolute* per-cell `MoveTo` columns from `charWidth`, the columns drifted right by
one per zero-width char, corrupting the screen on **NFD (decomposed) text** (e.g.
macOS filenames) and **any ZWJ/VS16 emoji** (`❤️`, family emoji). The row-level
path (`LAZYTUI_CELL_DIFF=0`) was immune, so default-on cell-diff was a regression.
Verified end-to-end against `@xterm/headless`. Same *class* as the Phase-1 kana bug
(a width source disagreeing with the terminal), not previously closed.

The fix makes `charWidth` the explicit **width truth function**, partitioned by axis
with each axis resolved by a standard library (no hand-maintained range table):
`wcwidth` (POSIX) for the zero-width axis (→ 0), `eastasianwidth` (UAX #11) for the
wide axis (→ 2), else 1. The two cell-grid builders (`cell-grid.js`, `cell-diff.js`)
were folding zero-width into their own column — fixed to fold into the preceding
glyph, mirroring the terminal.

`test-char-width.js` became an **exhaustive differential oracle**: it sweeps the
codepoint space and asserts `charWidth(cp)` equals the embedded terminal's *own*
`wcwidth` for every codepoint, with the residual divergences (archaic/astral,
unhittable in TUI content — newer-Unicode libs vs `@xterm`'s V6) enumerated and
pinned, so any new divergence (a lib bump, an emulator change) fails the test
instead of shipping as silent corruption. This is the conformance gate that would
have caught both this bug and the kana bug.

Round 2 also extended the flat-modal guard to `prompt_enter`/`copy_enter` (the other
continuation-staging openers) and made `runEffects` free a keyed effect's
`AbortController` if its handler throws synchronously (so a sync throw can't wedge
the `_inflight` key).
